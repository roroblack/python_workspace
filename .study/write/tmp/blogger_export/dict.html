CH 01왜 또 자료형이 필요한가 — list 다음에 dict#

학습

리스트는 “순서가 있는 모음”이다. 첫 번째, 두 번째, … 라는 위치가 식별자다. 그런데 부트캠프 실습이 “직원 관리”에 닿는 순간 이 모델이 어색해진다. 사번 200, 201, 202가 있을 때 우리가 다루는 것은 “0번째 직원”이 아니라 “200번 직원”이기 때문이다. 위치가 아니라 이름(키)이 식별자가 되는 자료형이 필요하다. 그것이 매핑(mapping)이고, 파이썬에서는 dict다.

의문 → 가설

가설 리스트로도 되지 않나? employees[0]처럼 인덱스로 찾으면 끝 아닌가?

가능은 하다. 그러나 “사번 200을 찾아라”라는 도메인 요구를 코드로 옮기는 순간, 인덱스 모델은 매번 “사번이 200인 항목의 인덱스”를 찾기 위해 리스트를 훑어야 한다. 평균 O(n)이고, 사번이 바뀌면 코드도 바뀐다. dict는 “사번을 그대로 키로 쓴다”라는 모델이라 평균 O(1)에 끝난다.

결론(중간)

KEY POINT — 가설 통과 위치로 찾을 것인가, 이름으로 찾을 것인가. 위치면 list, 이름이면 dict. 이 한 줄이 자료형 선택의 1차 기준이다.
다음 챕터로 가는 다리 — list 와는 다른 새 자료형이 왜 필요한지 알았으니, 다음 의문은 — 그런데 dict 의 키로는 무엇이든 다 쓸 수 있나? CH02 가 키의 자격을 본다.

CH 02만들기와 키의 자격#

학습

실습 파일 python_fileio/test_dict/dict_sample.py의 test_dict1/2는 다섯 가지 생성 패턴을 보여준다. 리터럴 {}, 생성자 dict(), 키-값 시퀀스로의 생성, zip으로의 짝짓기, 그리고 dict.fromkeys()까지. 모두 같은 자료형 dict를 만든다.

An object is hashable if it has a hash value which never changes during its lifetime (it needs a __hash__() method), and can be compared to other objects (it needs an __eq__() method). … All of Python's immutable built-in objects are hashable; mutable containers (such as lists or dictionaries) are not. — docs.python.org/3/glossary § hashable 
d1 = dict(name="aman", age=33, city="seoul")
d2 = {"name": "Alice", "age": 30, "city": "New York"}
d3 = dict([("a", 1), ("b", 2), ("c", 3)])
d4 = dict(zip(["x", "y", "z"], [10, 20, 30]))
d5 = dict.fromkeys(["a", "b", "c"], 0)
print("dict():     ", d1, type(d1).__name__)
print("literal:    ", d2)
print("from list:  ", d3)
print("from zip:   ", d4)
print("fromkeys:   ", d5)
mixed = {1: "int key", "a": "str key", (1, 2): "tuple key", 3.14: "float key"}
print("mixed keys: ", mixed)
# {[1, 2]: "list key"}  →  TypeError: unhashable type: 'list'

의문 → 가설

가설 키에는 어디까지 넣을 수 있는가? 정수, 문자열, 튜플은 되는데 왜 리스트는 안 되는가?

파이썬 공식 용어집은 “해시 가능(hashable)”을 “일생 동안 해시 값이 바뀌지 않고 다른 객체와의 동등성 비교가 가능한 객체”로 정의한다. dict는 키의 해시 값으로 슬롯을 정하고 그 자리에 값을 넣는다. 키 자신이 시간이 지나도 같은 해시를 줘야 다음에 다시 그 슬롯을 찾을 수 있다. 그래서 가변 객체는 키가 될 수 없다. 리스트, 딕셔너리, 집합은 모두 이 조건을 어긴다.

함정 “튜플은 키로 OK”라는 말은 절반만 맞다. 튜플 안에 가변 요소가 섞이면 그 튜플도 해시 불가다. 
hash((1, 2, 3))       # OK
hash((1, [2, 3]))     # TypeError: unhashable type: 'list'
즉 “재귀적으로 모든 요소가 해시 가능할 때”만 튜플이 키 자격을 얻는다.

테스트

다섯 가지 생성 패턴이 모두 dict를 만들고, 정수·문자열·튜플·실수는 키로 받아들이지만 리스트를 키로 쓰는 순간 TypeError: unhashable type: 'list'가 즉시 발생한다.

결론(중간)

KEY POINT — 가설 통과 키 = 해시 가능한 객체. 정수·문자열·튜플(요소까지 모두 해시 가능할 때)은 가능, 리스트·딕셔너리·집합은 불가. 그 외에 “생성 방법”은 다섯 가지든 열 가지든 결과는 같은 dict다.
다음 챕터로 가는 다리 — 키의 자격을 알았다면, 그 키로 값을 꺼내는 방법이 두 가지라는 사실이 다음 의문이 된다 — 엄격한 d[k] 와 안전한 d.get(). 둘은 언제 어느 쪽을 써야 하는가?

CH 03조회: 엄격한 d[k] 와 안전한 d.get()#

학습

키로 값을 꺼내는 두 가지 길은 의도가 다르다. d[k]는 “이 키가 반드시 있어야 한다”는 단정이고, d.get(k, default)는 “있으면 좋고 없어도 흐름은 이어진다”는 양보다.

d = {"a": 1, "b": 2, "c": 3}
print(f"d['a'] = {d['a']}")
print(f"d.get('z') = {d.get('z')}")
print(f"d.get('z', -1) = {d.get('z', -1)}")
# d["z"]  →  KeyError: 'z'
print(f"'a' in d: {'a' in d}")
print(f"3 in d.values(): {3 in d.values()}")
print(f"len(d): {len(d)}")

의문 → 가설

가설 그럼 항상 get()이 안전한 거 아닌가? d[k]를 굳이 쓸 이유가 있나?

의도를 코드로 드러내기 위해서다. “이 시점에 사번 200이 없다면 그건 버그”라면 d["200"]이 적절하다. 예외가 곧 “설계가 어긋났다”는 신호가 된다. 반대로 “없을 수도 있고 그땐 기본값으로 진행”이면 get()이 맞다. 둘 다 도구이고, 골라 쓰는 기준은 “없다”의 의미가 무엇이냐다.

주의 d.get(k)는 키가 없을 때도 None을 돌려준다. 따라서 “키가 없는 경우”와 “값이 진짜 None인 경우”가 구분되지 않는다. 둘을 구분해야 한다면 k in d로 먼저 체크하거나, 보초값(sentinel)을 두자.

테스트

결론(중간)

KEY POINT — 가설 통과 d[k] = “있어야 한다”, d.get(k, default) = “없어도 된다”. 둘은 안전성의 차이가 아니라 의도의 차이다.
다음 챕터로 가는 다리 — 조회를 익혔으면 다음은 변경이다. dict 는 한 자리(d[k] = v)에서 갱신·삭제까지 끝낼 수 있는데, 그 비결은 무엇인가?

CH 04변경: 한 자리에서 끝내는 갱신·삭제#

학습

dict_sample.py의 dict_test_func()은 일상 연산을 한자리에 모아 둔다. 리스트와 가장 큰 차이는, “인덱스 0이 비면 1이 0이 된다”는 시프트가 없다는 점이다. 키와 값이 한 쌍으로 사라지고 끝난다.

의도코드비고
갱신/추가d[k] = v키 존재 여부에 따라 자동 분기
일괄 갱신d.update(other)겹치는 키는 other가 이긴다
키 기반 삭제del d[k] / d.pop(k)pop은 값을 돌려준다
마지막 항목 제거d.popitem()3.7+ 부터 LIFO 순서, 빈 dict 면 KeyError
없으면 만들기d.setdefault(k, v)이미 있으면 변경 없음
전체 비우기d.clear()같은 객체를 비운다 (재할당 ≠ d = {})
 
d = {"a": 1, "b": 2, "c": 3}
d["a"] = 100; d["d"] = 4
print("after set/add:", d)
d.update({"b": 200, "e": 5})
print("after update :", d)
v = d.pop("c"); print(f"pop('c')     : {v} ->", d)

d.setdefault("f", 0); print("setdefault   :", d)
del d["b"]; print("del d['b']   :", d)
d.clear(); print("clear()      :", d)

의문 → 가설

가설 d.clear() 와 d = {} 가 같은 결과 아닌가?

결과처럼 보이지만 다르다. d.clear()는 같은 객체를 비운다. 따라서 다른 변수가 d를 참조하고 있었다면 그 변수도 빈 딕셔너리를 본다. d = {}는 새 객체를 만들어 이름 d에 다시 묶는 것뿐이고, 기존 객체는 다른 참조가 있다면 그대로 살아 있다.

결론(중간)

KEY POINT — 가설 통과 변경 연산은 모두 “키가 식별자다”라는 한 가지 규칙에서 파생된다. 인덱스가 흔들리지 않으니 시프트 비용도 없다.
다음 챕터로 가는 다리 — 값을 바꿨더니 keys()/values() 의 결과가 따라 바뀐다 — 그렇다면 뷰는 스냅샷이 아니라는 뜻인가? CH05 에서 확인한다.

CH 05뷰(view)는 “스냅샷”이 아니다#

학습

d.keys(), d.values(), d.items()는 모두 뷰 객체를 돌려준다. 새 리스트가 아니다. 뷰는 창문처럼 원본 dict를 들여다본다. 그래서 원본을 바꾸면 뷰의 출력도 따라 바뀐다.

d = {"a": 1, "b": 2, "c": 3}
print("keys  :", d.keys())
print("values:", d.values())
print("items :", d.items())
keys = d.keys()              # 뷰를 변수에 저장
d["d"] = 4                   # 원본에 새 키 추가
print("after add d['d']=4 -> keys :", keys)  # 자동 반영

의문 → 가설

가설 뷰가 살아 있다는 건 알겠다. 그럼 “지금 이 순간의 키 목록”을 박제하려면?

list(d.keys())로 명시 복사한다. 또한 키 뷰와 아이템 뷰는 set-like(집합 유사)라 &(교집합), |(합집합), -(차집합)이 곧장 통한다. 두 dict의 공통 키는 a.keys() & b.keys() 한 줄로 끝난다.

테스트

결론(중간)

KEY POINT — 가설 통과 뷰는 참조다. “기록”이 필요하면 list()로 그 시점을 끊어 잡아라.
다음 챕터로 가는 다리 — 뷰가 살아있는 창이라는 사실을 확인했으니, 그 창이 보여주는 키의 순서 는 어떻게 결정되는가? 삽입 순서인가, 정렬인가?

CH 06순서, 그러나 정렬은 아니다#

학습

3.7부터 dict는 삽입 순서를 언어 사양으로 보장한다. 순회는 “넣은 순서대로” 진행된다. 그러나 그것은 “정렬된 순서”가 아니다. 사전식·숫자 정렬이 필요하면 sorted()를 명시해야 한다.

d = {}
for k in ["c", "a", "b"]:
    d[k] = ord(k)
print("insertion order :", list(d.items()))
# sorted(d.items())  →  [('a', 97), ('b', 98), ('c', 99)]

의문 → 가설

가설 그럼 키 순서가 다른 두 dict는 다른 객체로 취급되나?

동등성(==) 검사는 순서와 무관하다. 같은 키-값 쌍을 가지면 같다. 단 “같다(==)”와 “같은 객체다(is)”는 다른 질문이다. 두 dict가 키-값까지 동일하더라도 메모리상 다른 객체일 수 있다.

a = {"x": 1, "y": 2}
b = {"y": 2, "x": 1}
print("a == b ?", a == b)   # True  — 내용이 같다
# a is b  →  False (다른 객체)

테스트

dict_runner.py §05 · Python 3.14.2 · PowerShell 
============================================================
# 05_order
============================================================
insertion order : [('c', 99), ('a', 97), ('b', 98)]
a == b ? True

결론(중간)

KEY POINT — 가설 통과 “순서가 보장된다 ≠ 정렬되어 있다.” 정렬은 언제나 sorted()로 명시하라.
다음 챕터로 가는 다리 — 순서가 보존된다는 사실은 알았다. 그렇다면 dict 를 copy() 로 복사할 때, 그 안에 들어 있는 list/dict 까지 같이 복사되는가?

CH 07얕은 복사의 함정#

학습

실습 코드의 copy() 절은 “원본을 보호하려면 사본을 만들면 된다”는 직관에 작은 함정이 숨어 있음을 보여준다. dict.copy()는 최상위 매핑만 복사한다. 값으로 들어 있는 리스트나 딕셔너리는 같은 객체를 그대로 공유한다.

import copy
src = {"name": "홍길동", "ids": [1, 2, 3]}
shallow = src.copy()
deep = copy.deepcopy(src)      # deepcopy 시점: src["ids"] = [1, 2, 3]
shallow["ids"].append(4)       # src["ids"] 도 [1,2,3,4] 가 된다
deep["ids"].append(99)         # deep 은 독립 사본 → [1,2,3,99]
print("src    :", src)
print("shallow:", shallow)
print("deep   :", deep)
print("id same? src vs shallow:", src["ids"] is shallow["ids"])
print("id same? src vs deep   :", src["ids"] is deep["ids"])

의문 → 가설

가설 그럼 “안전한 사본”이라는 게 그냥 copy()로는 안 되는 거 아닌가?

맞다. 한 층 위는 분리되지만 그 아래는 공유된다. “값까지 진짜로 분리”가 필요하면 copy.deepcopy()가 답이다. 단 깊은 복사는 비용이 크기 때문에, 변경할 일이 없는 깊은 부분까지 복사하는 건 낭비다. 보통은 변경할 부분만 deepcopy 하거나, 새로 만들어 끼워 넣는다.

다음 챕터로 가는 다리 — 얕은 복사의 함정이 dict 안에서 명확해졌다. 그러면 두 dict 를 합칠 때는 어떻게 되는가? Python 3.9 의 | 연산자와 컴프리헨션이 답한다.

CH 08합치기 연산자와 딕셔너리 컴프리헨션#

학습

3.9부터 dict는 |(합치기)와 |=(in-place 합치기)를 지원한다. 의미는 {**a, **b}나 a.update(b)와 같지만 한 줄이 짧고 의도가 분명하다.

a = {"x": 1, "y": 2}
b = {"y": 20, "z": 30}
print("a | b :", a | b)    # {'x': 1, 'y': 20, 'z': 30}  ← b 가 이긴다
a |= {"w": 9}
print("a |= {} -> a :", a) # {'x': 1, 'y': 2, 'w': 9}

딕셔너리 컴프리헨션은 같은 모양으로 변환·뒤집기·필터링을 한 줄에 담는다.

nums = [1, 2, 3, 4, 5]
squares = {n: n*n for n in nums}
print("squares:", squares)
swapped = {v: k for k, v in squares.items()}
print("swapped:", swapped)
even = {k: v for k, v in squares.items() if v % 2 == 0}
print("filter :", even)

의문 → 가설

가설 a | b 와 {**a, **b} 는 결국 같은 결과인데, 왜 굳이 새 연산자를 추가했을까?

가독성과 일관성. 집합·리스트와 마찬가지로 “두 매핑을 합친다”라는 자연스러운 표기를 갖게 된 것이다. PEP 584(Add Union Operators To dict)가 짚은 이유는 세 가지다.

  • {**a, **b}는 표현식으로는 작동하지만 “두 매핑의 합치기”라는 의미를 전달하는 표기로는 어색하다 — 별표 두 개의 의도가 “언패킹”이지 “합치기”가 아니기 때문.
  • a.update(b)는 구문(statement)이라 표현식 안에 둘 수 없다. a | b는 식이라 함수 인자, 컴프리헨션, 람다 본문 어디에서도 쓸 수 있다.
  • set·frozenset이 이미 |를 가지고 있다 — 매핑만 빠져 있을 이유가 없다.

CH 09미니 프로젝트: 직원 관리(dict + pickle)#

학습

실습 미션 python_fileio/fileio/fileio_mission.py는 사번을 키, 직원 정보 리스트를 값으로 두는 단일 dict(emp_list)를 메뉴 루프로 다룬다. 핵심은 두 가지다 — 키 기반 식별과 파일 영속화.

emp_list = {
    "200": ["홍길순", "851225-2234567", "hong@test.com", "010-1234-5678", 3800000, "대리", "개발부"],
    "201": ["김철수", "900101-1234567", "kim@test.com",  "010-2222-3333", 4200000, "과장", "개발부"],
    "202": ["이영희", "920510-2345678", "lee@test.com",  "010-7777-8888", 3500000, "사원", "기획부"],
}
print(">> 전체 출력")
for empid, emp_info in emp_list.items():
    print(f"  {empid} : {emp_info}")

print(">> 201 삭제 후")
for empid, emp_info in emp_list.items():
    print(f"  {empid} : {emp_info}")

# pickle 저장 → 불러오기 (round-trip)
import pickle
with open("employees.dat", "wb") as f:
    pickle.dump(emp_list, f)
with open("employees.dat", "rb") as f:

print(">> 불러온 키 목록      :", list(loaded.keys()))

의문 → 가설

가설 JSON 도 있는데 왜 pickle 인가? 그리고 변수명이 왜 emp_list인가, 자료형은 dict 아닌가?

두 가지로 답한다.

  • 왜 pickle: JSON은 키를 문자열로만 허용하고, 튜플·바이트열·사용자 정의 객체를 직접 표현하지 못한다. 이번 미션은 사번이 문자열이라 JSON 도 가능하지만, “파이썬 객체 그대로 보존”이 목적이면 pickle이 짧고 정확하다. 단 신뢰할 수 없는 입력에는 절대 쓰지 말 것 — pickle은 임의 코드 실행 위험이 있다.
  • 이름과 타입의 불일치: emp_list라는 이름은 학습 단계의 흔적이다. 실제 자료형이 dict임을 드러내려면 employees나 emp_table이 더 낫다. 코드 가독성은 “이름이 타입을 거짓말하지 않는 데서” 시작한다.
보안 경고 pickle.load()는 파일에 적힌 지시대로 임의의 파이썬 객체를 복원한다. 즉 악의적으로 조작된 pickle 파일을 열면 임의 코드 실행이 일어날 수 있다. 따라서 외부에서 받은 pickle 파일은 절대 그대로 load하지 말 것. 외부 교환이 필요하면 json을, 동일 시스템 내부 캐시 용도로만 pickle을 쓰는 것이 안전하다.

CH 11왜 평균 O(1)인가 — dictobject.c 안으로#

학습

여기서부터는 “쓰는 법”에서 “돌아가는 법”으로 시점을 옮긴다. 본 글에 인용한 CPython 소스는 작업 폴더에 사본이 있다 — .study/test/2-dict/python_dict/dictobject.c(약 244 KB), pycore_dict.h(약 16 KB).

The dict implementation uses an open addressing hash table … Major subtleties ahead: most hash schemes depend on having a "good" hash function, in the sense of simulating randomness. Python doesn't: its most important hash functions (for strings and ints) are very regular in common cases. — github.com/python/cpython · Objects/dictobject.c (헤더 주석)

3.6 이후 CPython 의 dict는 compact dict 구조다. 인덱스 배열과 엔트리 배열을 분리해 인덱스 배열은 작은 정수만 담고, 엔트리 배열은 (해시, 키, 값)을 삽입 순서대로 모아 둔다. 이 한 가지 구조로 “순서 보존”과 “캐시 효율”이 동시에 해결된다.

// pycore_dict.h - 핵심만 발췌
struct _dictkeysobject {
    Py_ssize_t dk_refcnt;
    uint8_t    dk_log2_size;          // 표 크기 = 1 << dk_log2_size
    uint8_t    dk_log2_index_bytes;   // 인덱스 슬롯 한 칸의 바이트 수
    uint8_t    dk_kind;               // GENERAL / UNICODE / SPLIT
    uint32_t   dk_version;            // 변경 시 0 으로 초기화 (반복자 무효화 신호)
    Py_ssize_t dk_usable;             // 더 넣을 수 있는 엔트리 수
    Py_ssize_t dk_nentries;           // 현재 사용 중 엔트리 수
    char       dk_indices[];          // 표 (hash → entry index)
    /* 그 뒤에 PyDictKeyEntry / PyDictUnicodeEntry 배열 */
};

#define DKIX_EMPTY (-1)   // 빈 슬롯
#define DKIX_DUMMY (-2)   // 삭제됨 — 탐색은 계속

핵심은 세 가지다.

  • 표 크기는 2의 거듭제곱. hash & mask 한 번으로 슬롯이 정해진다.
  • 인덱스 폭이 가변. 작은 dict는 슬롯 한 칸이 1바이트, 표가 커지면 2/4/8바이트로 자동 확장된다.
  • 삭제 표식 DKIX_DUMMY. 슬롯을 곧장 비우면 충돌 사슬이 끊겨 다른 키를 못 찾는다. “비었지만 탐색은 계속”이라는 표식이 필요하다.

의문 → 가설

가설 해시 충돌을 체이닝(Chaining) 방식으로 처리하면, 충돌이 쌓일수록 링크드 리스트 → 레드블랙트리로 전환되면서 성능 손실이 발생한다고 한다. CPython 의 dict는 이 문제를 어떻게 해결했는가?

이것이 학습 초기에 남긴 핵심 의문이다. Java 의 HashMap은 실제로 한 버킷에 항목이 8개 이상 쌓이면 링크드 리스트를 레드블랙트리(Red-Black Tree)로 전환한다. 덕분에 최악 탐색이 O(n) → O(log n)으로 개선되지만, 트리 노드는 포인터로 메모리 여기저기 흩어지기 때문에 캐시 지역성(Cache Locality)을 잃는다. 현대 CPU 는 메모리를 캐시라인(64 B) 단위로 읽는다. 포인터를 타고 노드를 오가면 캐시 미스가 누적되어 "이론 O(log n)인데 실제 체감이 느린" 역설이 생긴다.

CPython 의 선택은 다르다. 체이닝을 아예 쓰지 않고, 오픈 어드레싱(Open Addressing)으로 충돌을 같은 배열 안에서 해결한다. 모든 엔트리가 연속된 메모리 블록에 머무르므로 캐시 효율이 높다. 단, 배열이 너무 꽉 차면 오픈 어드레싱도 탐색 사슬이 길어지는데, CPython 은 슬롯의 약 2/3까지만 채우고 그 이상이면 표를 두 배로 늘려 재해싱(rehash)하는 정책으로 이를 원천 차단한다.

방식충돌 처리캐시 효율최악 탐색대표 구현
체이닝 + 레드블랙트리버킷마다 링크드 리스트 → RBT낮음 (포인터 점프)O(log n)Java HashMap
오픈 어드레싱 + perturb같은 배열 안에서 다음 빈 슬롯높음 (연속 메모리)O(n) 이론, 실제 ≈ O(1)CPython dict

오픈 어드레싱은 탐색 시퀀스 설계가 나쁘면 군집(Clustering) 현상으로 성능이 망가진다. 아래 Q 가 이 문제를 다룬다.

가설 충돌이 나면 다음 자리는 어떻게 정하나? 그냥 옆 칸을 보면 충돌 사슬이 한 줄로 길어질 텐데.

CPython 은 perturb 시퀀스를 쓴다. 단순 선형 탐색이 아니라 “원래 해시의 상위 비트”까지 끌어와 다음 후보 슬롯을 결정한다.

// dictobject.c — 충돌 발생 시 다음 슬롯 결정 (요약)
#define PERTURB_SHIFT 5

i = hash & mask;          // 첫 슬롯
perturb = hash;
while (충돌) {
    perturb >>= PERTURB_SHIFT;
    i = (i*5 + 1 + perturb) & mask;   // 다음 후보
}

이 방식은 두 가지를 동시에 노린다. (1) 첫 단계에서 충돌한 키들이 두 번째 단계에서도 같은 자리에 모이지 않도록 perturb의 상위 비트가 흩어 주고, (2) perturb가 0이 된 뒤에는 점화식이 i = 5*i + 1 (mod 2^k)로 단순해진다. 그런데 표 크기가 2의 거듭제곱일 때, 이 점화식은 모든 슬롯을 정확히 한 번씩 방문한다는 정수론적 사실이 알려져 있다(곱수 5는 2의 거듭제곱과 서로소이고 +1 항이 사이클을 깨 준다). 그래서 빈 슬롯이 하나라도 있으면 탐색은 반드시 끝난다 — 평균은 O(1), 최악도 잘 정의된 상한을 갖는다.

또 하나의 의문 — “표는 언제, 어떻게 커지나?” CPython 은 “슬롯의 약 2/3까지만 채우고, 그 이상이면 표 크기를 두 배로 늘려 다시 분포(rehash)”라는 정책을 쓴다(상수 USABLE_FRACTION). 이 정책이 외부에서 관측 가능한 흔적을 남긴다 — sys.getsizeof(d)가 단계적으로 점프한다.

테스트

dict에 키를 0개부터 16개까지 한 개씩 넣으면서 메모리 크기를 측정했다. 길이 0→1, 5→6, 10→11 사이에서 단계적 점프가 일어나는 것을 보면, 위에서 말한 “2/3 적재율” 정책이 그대로 표면에 드러난다.

결론(중간)

KEY POINT — 가설 통과 “키 → 슬롯”은 단 한 번의 비트 마스크로 끝나고, 충돌은 perturb 기반 오픈 어드레싱으로 흩어지고, 표는 2/3 적재율에서 두 배로 자란다. 이 세 가지가 “평균 O(1)”의 실체다.
다음 챕터로 가는 다리 — 내부까지 모두 확인했다. CH12 에서 출발 의문 — "list 까지 익혔는데 왜 또 새 자료형 dict 가 필요한가?" — 으로 돌아가 한 문장으로 압축한다.

CH 12정리: 한 문장으로 압축#

최종 결론

FINAL CONCLUSION 출발 의문 — "list 까지 익혔는데 왜 또 새 자료형 dict 가 필요한가?" — 의 답. list 가 "위치(index)로 찾는 모음" 이라면 dict 는 "이름(key)으로 찾는 모음" 이다. 그 이름을 빠르게 찾아내기 위해 내부는 해시 테이블 + compact + open addressing 로 짜여 있고, 그 결과가 우리가 한 줄(d[k]) 로 누리는 평균 O(1) 다. CH02 의 키 자격(해시 가능), CH05 의 살아있는 뷰, CH07 의 한 층 복사, CH10 의 RuntimeError, CH11 의 dictobject.c — 이 모두가 그 한 자료구조의 다른 얼굴이다.

리스트가 “순서가 있는 모음”이라면, dict는 “이름으로 찾는 모음”이다. 이름을 받기 위해 키는 해시 가능해야 하고, 키 충돌을 피하기 위해 내부는 compact + open addressing 으로 짜여 있고, 그 결과 우리가 코드 한 줄로 누리는 것이 평균 O(1) 조회다.

이 글의 11개 챕터를 한 문장으로 줄이면 다음과 같다 — “dict는 키로 찾고, 삽입 순서를 기억하며, 평균 O(1)에 답한다. 키는 해시 가능해야 하고, 뷰는 살아 있고, 복사는 한 층만 분리된다.”

CHEAT SHEET
  • 식별자가 위치 → list, 식별자가 이름 → dict
  • 키 자격: 해시 가능 (정수·문자열·튜플 ✓ / 리스트·딕셔너리·집합 ✗)
  • 조회: 단정 d[k] · 양보 d.get(k, default)
  • 변경: d[k]=v, update, pop, popitem, setdefault, del, clear
  • 뷰는 살아있다. 박제하려면 list(d.keys())
  • 3.7+ 삽입 순서 보장. 정렬은 별개 — sorted()
  • copy()는 한 층, 그 아래는 copy.deepcopy()
  • 3.9+ 합치기: a | b, a |= b
  • 반복 중 변경 → RuntimeError. 사본을 돌려라
  • 영속화는 pickle(파이썬 전용) 또는 json(범용)

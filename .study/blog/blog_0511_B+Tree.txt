6. B+Tree 구조에서 클러스터형 인덱스는 어디에 쓰이나?
	-> 	B+Tree의 '리프 노드(Leaf Node)' 영역 전체에 쓰임
		리프노드 : 데이터 페이지 역할
		루트 및 중간 노드 : 자식페이지 주소와 최소 기본키만 가짐
		내부 배열이 쓰이는 곳 : 	키와 자식 페이지 주소 포인터
					페이지 끝의 페이지 디렉터리
					페이지 내부의 데이터 값들 - 연속된 배열형
		
               [ Root Node ] (Page #1)
               [  id=100   ]
              /             \
             /               \
   [ Intermediate Node ]   [ Intermediate Node ] (Page #2, #3)
   [ id=10, id=50 ]        [ id=110, id=150 ]
       /       \               /        \
      /         \             /          \
 [ Leaf Node ] [ Leaf Node ] [ Leaf Node ] [ Leaf Node ] (Page #4, #5, #6, #7)
 [ id=10, data ] [ id=50, data ] [ id=110, data ] [ id=150, data ]
 [ id=11, data ] [ id=52, data ] [ id=115, data ] [ id=160, data ]
     <-------- Double Linked List -------->

		- 페이지 분할 알고리즘 : 이건 어떻게 구현되어 있나

15. CH 15 총 정리 — 출발 의문 회수 : "왜 B+Tree 여야만 했는가"

	이 챕터는 CH01~CH14 전체를 하나의 사슬로 묶어 회수하는 챕터.
	각 챕터에서 세운 가설과 확인된 핵심 한 줄만 꺼내서 재조립한다.

	구성:
	1) 출발 의문 재제시

	2) 챕터별 핵심 한 줄 요약표 (표 형태로 제시)
		| CH   | 주제                     | 핵심 한 줄                                                           |
		|------|--------------------------|----------------------------------------------------------------------|
		| CH01 | B+Tree 자료구조          | InnoDB 의 모든 인덱스는 B+Tree(lv0=leaf). SHOW INDEX → BTREE 확인    |
		| CH02 | 왜 B+Tree 인가           | 분기계수 B≈1000 → 10억 행도 3단계. 디스크 I/O 횟수가 전부다          |
		| CH03 | O(1)에 가까운 성능       | 클러스터드 인덱스+AHI+Buffer Pool 3중 합산 → 97% hash 즉답           |

		| CH08 | AHI                      | 내부 카운터가 임계 초과 시 글로벌 해시 빌드 → B+Tree 수직하강 생략   |

		| CH10 | INSERT 전체 흐름         | Redo→Buffer Pool→Page Dir→Multi-Link→AHI→디스크 WAL 순서             |

		| CH13 | 최적화 포인트            | 커버링 인덱스: secondary만으로 완결 → clustered 재조회 0회            |
		| CH14 | 전체 구조도              | 7개 자료구조가 mini-transaction 하나로 묶여 동시 작동                  |

	3) 최종 결론 — 출발 의문 회수
		"왜 B+Tree 여야만 했는가"
		→ B+Tree 자체가 빠른 게 아니다.
		→ B+Tree 위에 클러스터드 인덱스 / AHI / Buffer Pool 이라는 가속 장치를
		  얹기에 가장 적합한 구조이기 때문이다.
		→ 분기계수가 커서 높이가 3단계에 수렴하고,
		  리프끼리 이중 연결 리스트로 이어져 범위 검색이 수평이동만으로 끝나며,
		  right-only split 이 AUTO_INCREMENT 와 맞물려 단편화를 0으로 유지한다.
		→ 이 세 가지 구조적 특성이 AHI와 Buffer Pool이 제 역할을 할 수 있는
		  토대를 만든다.
		→ 결론: O(log_B n) 이지만 B≈1000, 루트/중간 노드는 버퍼풀에 상주,
		  자주 쓰는 키는 AHI가 O(1)으로 단락 → 운영 환경에서 사실상 O(1).

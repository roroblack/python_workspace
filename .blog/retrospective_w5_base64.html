CH 01한 주의 목표 & numpy — 분석은 왜 ndarray에서 시작했나#

학습

5주차 첫날(0520) 수업은 분석 프로세스와 도구 소개로 열렸지만, 코드로 내려가 보면 모든 예제의 바닥에는 파이썬 list가 아니라 numpy ndarray가 깔려 있었다. “왜 굳이 새 자료형인가”가 이 주의 첫 의문이었다. list는 아무 타입이나 섞어 담을 수 있어 편한데, 분석 도구들은 하나같이 ‘같은 자료형만 담는’ 배열을 기본으로 삼고 있었다.

의문 → 가설

가설 np_arr * 2 같은 연산이 파이썬 루프보다 빠른 이유가 있다면, 그건 ① 모든 요소가 같은 dtype이라 메모리가 연속으로 놓이고, ② 그 덕분에 반복이 파이썬이 아니라 C 레벨에서 한 번에 돈다는 두 가지가 동시에 성립하기 때문일 것이다. 이게 사실이면 ‘같은 자료형’이라는 제약은 불편이 아니라 속도의 대가다.

테스트 — 같은 합을 루프와 벡터화로 각각

같은 합 연산을 파이썬 for 루프와 numpy 벡터화로 각각 재서 0520 실습에서 직접 시간을 비교했다.

for 루프 합   : 0.5231 초
NumPy 벡터화 합 : 0.0049 초
속도 향상      : 약 106.8 배

제곱·역수 같은 요소별 연산도 마찬가지였다. 형태(shape)를 바꾸는 reshape(-1, 4)의 -1은 “나머지 차원은 전체 요소 수에서 추정하라”는 와일드카드였고, 그래서 같은 12개 숫자가 (12,)·(3,4)·(2,2,3) 어느 것이냐에 따라 의미가 완전히 달라졌다. shape이 ndarray의 ‘정체성’이라는 걸 여기서 처음 체감했다.

"A numpy array is a grid of values, all of the same type … The number of dimensions is the rank of the array; the shape of an array is a tuple of integers giving the size of the array along each dimension." — NumPy 공식 문서 · The absolute basics for beginners

결론(중간)

KEY POINT — 가설 통과 ndarray의 속도는 “같은 자료형”이라는 단 하나의 제약에서 나온다 — 그 제약 덕에 메모리가 연속이고, 반복이 C에서 돌아 루프 대비 약 100배 빨라진다(실측 약 106.8배). 그래서 numpy는 분석 ‘도구’가 아니라 분석의 공용 자료형이다. 실제로 다음 날 만질 pandas의 DataFrame 내부도 결국 ndarray다.
다음 챕터로 가는 다리 — ndarray는 ‘같은 dtype·빈 칸 없음’을 전제한다. 그런데 현실 데이터에는 반드시 비어 있는 칸(결측값)이 있다. numpy에서 nan == nan이 False라는 사실을 본 순간 의문이 생겼다 — 그러면 빈 칸은 어떻게 ‘찾고’ 어떻게 ‘메워야’ 하나? 이 질문이 0521 pandas 전처리로 넘어간다.

CH 02pandas 전처리 — 비어 있는 칸을 어떻게 메울 것인가#

학습

0521은 변수 변환과 결측값 처리(전처리)의 날이었다. pandas의 Series·DataFrame은 ndarray에 라벨 인덱스와 혼합 dtype을 더한 것이었고, 그래서 행마다 컬럼마다 서로 다른 의미를 가진 표를 그대로 다룰 수 있었다. 그런데 표를 받자마자 부딪힌 것이 빈 칸, 즉 NaN이었다.

의문 → 가설

의문 가장 직관적인 건 fillna(0)이다 — 빈 칸을 0으로 채우면 끝 아닌가? 그런데 수업 예제에서 이 방법을 일부러 피했다. “0으로 메우는 게 왜 위험한가”가 이 챕터의 의문이다. 내 가설은 “0은 ‘값이 없음’이 아니라 ‘실제 0’이라는 다른 정보라서, 평균·분포를 같이 망가뜨릴 것”이었다.

테스트 — 채우기 전략별로 통계가 어떻게 흔들리나

먼저 빈 칸을 찾는 일부터다. nan == nan은 False라 비교로는 못 잡고 isnull()/np.isnan()으로만 탐지된다. 그다음 0·평균·중앙값으로 각각 채워 통계를 비교했다.

# path : .study/test/day0521/day0521_runner.py (발췌)
import pandas as pd, numpy as np

s = pd.Series([10, np.nan, 30, np.nan, 50])
print('결측 위치 :', s.isnull().tolist())   # nan==nan 이 아니라 isnull로 탐지
print('0  으로  :', s.fillna(0).mean())      # 빈 칸을 '실제 0'으로 오염
print('평균으로 :', s.fillna(s.mean()).mean())
print('중앙값   :', s.fillna(s.median()).mean())
day0521_runner.py · fillna(0) vs 평균 vs 중앙값 — 평균 보존 비교
결측 위치 : [False, True, False, True, False]
0  으로  : 18.0
평균으로 : 30.0
중앙값   : 30.0

결론(중간)

KEY POINT — 가설 통과 fillna(0)은 평균을 30 → 18로 끌어내렸다. 가설대로 0은 ‘없음’이 아니라 ‘실제 0’이라는 오염된 정보였다. 결측값 처리는 “숫자를 채우는 기술”이 아니라 “도메인을 반영하는 판단”이다 — 이 한 문장이 0521에서 가장 크게 남은 관찰이다. 같은 맥락에서, 한쪽으로 쏠린(왜도 큰) 변수는 log 변환으로 분포를 펴 정규분포에 가깝게 만들 수 있었다(단, 음수·0이 있으면 그대로는 안 되고 보정이 필요했다).
다음 챕터로 가는 다리 — 빈 칸을 메우고 나니 반대쪽 문제가 보였다. 칸이 비어 있는 게 아니라 너무 튀는 값이 들어 있을 때다. 평균을 통째로 흔드는 이 값들을 ‘이상치’라 부른다는데 — 어디까지를 정상으로 보고 어디부터를 이상치로 끊어야 하나? 이 질문이 0522로 이어진다.

CH 03이상치, 어디까지가 '튀는 값'인가#

학습

0522는 이상값 처리와 정규화의 날이었다. 실습 데이터는 auto-mpg였고, 그중 horsepower처럼 오른쪽으로 꼬리가 긴 변수가 문제였다. 이상치를 끊는 기준에는 두 가지가 있었다 — 사분위 기반 IQR과 표준편차 기반 Z-score. 둘 다 “튀는 값”을 잡는다는데, 잡는 결과가 같을 리 없어 보였다.

의문 → 가설

가설 IQR은 분포 가정 없이 사분위수로만 경계를 긋고, Z-score는 정규분포를 가정한다. 그렇다면 오른쪽 꼬리가 긴 horsepower에서는 IQR이 더 많이 잡고, Z-score는 꼬리를 정규분포로 착각해 과소 탐지할 것이다.

테스트 — 같은 변수에 두 기준을 나란히

같은 horsepower에 IQR 경계(Q1−1.5·IQR ~ Q3+1.5·IQR)와 |Z|>3을 동시에 걸어 0522 실습에서 비교했다.

결론(중간)

KEY POINT — 가설 통과 가설대로였다. IQR은 10개, Z-score는 5개를 잡았고 공통 5개가 가장 신뢰도 높은 이상치였다. 가장 크게 배운 점은 “이상치 탐지에 정답 기준은 없다”는 것 — 비대칭(꼬리) 분포엔 IQR, 정규 분포엔 Z-score가 더 맞고, 두 기준이 함께 가리키는 값이 가장 확실하다. 끊은 다음에는 제거·대체·캡핑 세 전략이 있었는데, n을 유지하면서 평균을 가장 적게 흔든 건 경계값으로 눌러 담는 캡핑이었다.
다음 챕터로 가는 다리 — 이상치를 정리하고 나서 상관행렬을 보니 mpg와 weight의 상관계수가 −0.832로 매우 강했다. 그런데 weight(수천 단위)와 mpg(수십 단위)는 애초에 ‘자릿수’가 달랐다. 이렇게 단위가 다른 변수들을 한 모델에 그대로 넣어도 될까? 이 의문이 정규화로 넘어간다.

CH 04정규화 — 단위가 다른 변수를 같은 자에 세우기#

학습

정규화(스케일링)에는 두 대표 방법이 있었다. MinMax는 최솟값~최댓값을 0~1로 압축하고, Standard는 z = (x − μ) / σ로 평균 0·표준편차 1에 맞춘다. 둘 다 “같은 자에 세운다”는 목적은 같은데, 0522 실습에서 이상치를 일부러 남겨 두고 둘을 비교했더니 결과가 차이가 났다.

의문 → 가설

가설 MinMax는 ‘최댓값’ 하나에 0~1 범위 전체가 묶인다. 그렇다면 이상치가 남아 있을 때 그 거대한 최댓값이 범위를 독점해, 정상 데이터 대부분이 0 근처로 짓눌릴 것이다. 반대로 Standard는 평균·표준편차 기반이라 이상치에 상대적으로 덜 흔들릴 것이다.

테스트 — 이상치를 남긴 채 두 스케일러 비교

이상치가 살아 있는 변수에 두 스케일러를 각각 적용해 정상 구간이 어디로 모이는지 0522 실습에서 확인했다.

day0522_runner.py · MinMaxScaler vs StandardScaler — 이상값 잔존 시 정상 구간 분포
[이상값 잔존]
MinMax  정상 95% 구간 : 0.00 ~ 0.34   (대부분 0 근처로 압축)
Standard 정상 95% 구간 : -1.21 ~ 1.43  (분포 형태 보존)

[이상값 제거 후]

결론(중간)

KEY POINT — 가설 통과 이상치가 남으면 MinMax는 정상값을 0 근처(0.00~0.34)로 짓눌렀다 — 가설대로 ‘최댓값 독점’이 일어났다. 그래서 기준이 정리됐다: 이상값을 먼저 제거한 뒤라면 MinMax, 이상값이 잔존하는 상황이면 Standard. 중요한 건 정규화가 독립된 단계가 아니라 이상치 처리 다음에 오는 단계라는 것 — CH03과 CH04는 순서로 묶여 있었다. (덧붙여, pandas로 직접 (x−min)/(max−min)을 계산한 결과는 sklearn MinMaxScaler와 정확히 일치했다. API는 결국 그 공식을 감싼 것이다.)
다음 챕터로 가는 다리 — 여기까지 결측·이상치·정규화를 숫자(터미널 출력)로만 판단해 왔다. 그런데 “오른쪽 꼬리가 길다”, “정상값이 0 근처로 눌렸다” 같은 말은 사실 그림으로 봐야 한 번에 잡힌다. 분포와 관계를 눈으로 읽으려면 어떤 차트를 골라야 하나? 0526 시각화로 넘어간다.

CH 05시각화 — 숫자를 '눈으로 읽기', 차트는 무엇으로 고르나#

학습

0526은 matplotlib·seaborn 시각화의 날이었다. 먼저 구조부터 헷차이가 났다 — Figure는 도화지(캔버스), Axes는 그 위의 개별 그래프 영역이었다. 탐색용 빠른 그림에는 pyplot 방식이 편했지만, 여러 subplot을 한 장에 배치하는 보고용에는 fig, ax = plt.subplots()의 OOP 방식이 꼬이지 않았다. 그다음 의문은 더 실전적이었다 — “어떤 데이터에 어떤 차트를 쓰는가”.

의문 → 정리

의문 차트 종류는 많은데, 막상 데이터를 앞에 두면 무엇을 골라야 할지 막막했다. 분포를 볼 때, 두 변수 관계를 볼 때, 범주를 비교할 때, 시간 흐름을 볼 때 — 각 상황에 ‘맞는’ 차트가 따로 있는가? 있다면 그 기준은 무엇인가.

관찰 — 상황별로 차트가 차이가 났다

0526 실습에서 같은 auto-mpg 데이터를 여러 차트로 그려 보니, 차트마다 ‘보여 주는 것’이 분명히 달랐다. histogram은 분포의 모양(왜도·봉우리)을, boxplot은 중앙값·IQR·이상치를 한 번에 보여 줬고 — 이건 정확히 CH03의 IQR 이상치를 그림으로 다시 본 것이었다. scatter는 두 변수의 관계 모양을, heatmap은 전체 변수 쌍의 상관을 한눈에 줬는데, CH04에서 본 mpg–weight −0.832가 scatter에서 우하향 직선으로 그대로 나타났다.

보고 싶은 것차트읽히는 정보

결론(중간)

관찰 결과 차트 선택은 취향이 아니라 “무엇을 읽고 싶은가”가 정한다 — 분포는 hist/box, 관계는 scatter/heatmap, 범주는 count/bar, 시간은 line. 더 중요한 건, 시각화가 앞선 전처리 단계와 따로 노는 게 아니라 그 결과를 눈으로 검증하는 마지막 단계였다는 점이다. boxplot은 CH03의 이상치를, heatmap은 CH04의 상관을 그림으로 다시 확인시켜 줬다.
다음 챕터로 가는 다리 — 여기서 멈칫했다. numpy·결측·이상치·정규화·시각화를 날짜별로 따로 배웠는데, 돌아보니 이것들은 흩어진 기능이 아니라 한 줄로 꿰지는 순서가 있었던 것 아닐까? 그 순서를 한 번 그려 보기로 했다.

CH 06연결 — 흩어진 도구들이 하나의 전처리 파이프라인으로#

관찰 → 정리

주 후반에 들어 다섯 개의 주제가 사실은 한 흐름이라는 게 보였다. 데이터를 읽고(numpy·describe) → 비거나 잘못된 칸을 고치고(결측·변수변환) → 튀는 값을 끊고(이상치) → 단위를 맞추고(정규화) → 그 결과를 그림으로 검증(시각화)하는 순서다. 그리고 이 순서에는 어기면 탈이 나는 규칙이 하나 있었다 — CH04에서 본 것처럼 정규화는 이상치 처리 다음이어야 했다.

┌──────────────────────────────────────────────────────────┐ │ 5주차 데이터 분석 파이프라인 (읽기 → 검증) │ ├──────────────────────────────────────────────────────────┤ │ ① 읽기 numpy ndarray · df.describe() ← 0520·0522 │ │ ↓ │ │ ② 고치기 결측 탐지·fillna(판단)·log 변환 ← 0521 │ │ ↓ │ │ ③ 끊기 이상치 IQR / Z-score · 캡핑 ← 0522 │ │ ↓ │ │ ④ 맞추기 정규화 MinMax / Standard ← 0522 │ │ ↓ (※ 반드시 ③ 다음에) │ │ ⑤ 검증 hist·box·scatter·heatmap ← 0526 │ └──────────────────────────────────────────────────────────┘

의문 → 가설

의문 이 흐름의 출발점이 다시 numpy로 돌아온다는 게 신기했다. pandas로 표를 다루고, sklearn으로 스케일링하고, matplotlib로 그렸는데 — 이 도구들은 서로 다른 자료형으로 데이터를 주고받았을까? 아니면 바닥에서 같은 ndarray를 공유했을까? CH01의 “공용 자료형” 주장이 여기서 검증된다.

관찰 — 바닥은 모두 ndarray였다

실제로 df.values는 numpy 배열을 돌려줬고, MinMaxScaler·StandardScaler는 ndarray를 받아 ndarray를 냈으며, matplotlib도 ndarray를 그렸다. pandas는 그 위에 라벨을, sklearn은 변환 규칙을, matplotlib는 픽셀을 얹었을 뿐 — 데이터 자체는 한 번도 ndarray 밖으로 나가지 않았다.

정리 다섯 날의 도구가 한 파이프라인으로 묶였고, 그 파이프라인을 관통하는 단 하나의 자료형이 numpy ndarray였다. CH01에서 “numpy는 도구가 아니라 공용 자료형”이라 했던 가설이, 마지막 시각화까지 와서 사실로 확인된 셈이다. 전처리의 순서(②→③→④)와 그 바닥의 자료형(ndarray) — 이 두 가지가 5주차의 뼈대였다.
다음 챕터로 가는 다리 — 파이프라인의 마지막 칸은 비어 있었다. 이렇게 읽고·고치고·맞춘 데이터를 무엇에 쓰는가다. 그 답이 다음 주(6주차) 머신러닝이었고, 거기서 StandardScaler와 train_test_split이 다시 등장한다. 5주차는 그 입구였다.

CH 07주차 종합 회고 — 목표 회수#

출발 목표 회수

이 주의 출발 목표는 “데이터 한 덩어리를 받았을 때, 손대기 전에 그 데이터를 읽을 수 있게 된다”였다. 돌아보면 네 조건이 모두 한 칸씩 채워졌다.

출발 조건채워진 날한 줄 결론
① 공용 자료형을 쥔다0520 numpy같은 dtype 제약 = 속도(≈100배), shape이 정체성

잘한 점 / 부족했던 점

잘한 점 — 기능을 따로 외우지 않고 “결측 → 이상치 → 정규화 → 시각화”라는 순서로 묶어 이해한 것. 특히 정규화가 이상치 처리 다음이어야 하는 이유를 MinMax 압축 실험으로 직접 본 게 컸다. 

부족했던 점 — 실습 데이터가 auto-mpg 하나에 치우쳐, 같은 기준이 다른 분포(예: 양쪽 꼬리, 범주형 다수)에서도 통하는지는 아직 확인하지 못했다. log 변환에서 음수·0 보정도 손에 더 익혀야 한다.

FINAL CONCLUSION 5주차의 실체는 — “데이터를 읽는다”가 곧 ndarray 위에서 결측·이상치·정규화를 ‘판단’으로 처리하고, 그 판단을 그림으로 검증하는 한 줄의 파이프라인이라는 것을 손에 쥔 주였다. 교과목 1이 데이터를 모으고 저장하는 법이었다면, 5주차는 그 데이터를 분석 가능한 형태로 세우는 법이었고, 이렇게 세운 데이터를 다음 주 머신러닝이 받아 ‘예측’으로 끌고 간다. 출발 목표는 달성됐고, 파이프라인의 빈 마지막 칸(“그래서 무엇에 쓰나”)이 그대로 6주차의 출발 의문이 되었다.

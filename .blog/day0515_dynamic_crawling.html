CH 01 DAY5정적 vs 동적 — urllib 의 한계와 Selenium 의 등장#

학습

현대 웹 페이지의 상당 부분은 ① 서버가 빈 골격만 보내고 ② 브라우저에서 JavaScript 가 API 를 호출해 실제 데이터를 채우는 방식이다(SPA). 이때 urlopen 이 받는 것은 ① 의 골격뿐이다. 

# 정적 방식의 한계 시연
import urllib.request, bs4
url = 'https://search.naver.com/search.naver?query=로마여행'
page = urllib.request.urlopen(url)
soup = bs4.BeautifulSoup(page, 'html.parser')
print(len(soup.select('li[data-otb_lsra-doc-id]')))   # → 0

# 동적 방식 — 실제 브라우저를 띄움
from selenium import webdriver as wd
from selenium.webdriver.chrome.service import Service
from webdriver_manager.chrome import ChromeDriverManager

driver = wd.Chrome(service=Service(ChromeDriverManager().install()))
driver.get(url)
# ... JS 실행을 기다린 후
items = driver.find_elements('css selector', 'li[data-otb_lsra-doc-id]')
print(len(items))                                      # → 10
Selenium drives a real browser, giving your code access to the fully rendered DOM after JavaScript execution — not just the raw HTTP response body. This is the fundamental difference from static crawling with urllib. — Selenium Documentation § WebDriver

의문 → 가설

가설 urlopen 이 보는 HTML 과 브라우저가 보는 DOM 이 다른 이유는 — urlopen 이 "HTTP 응답 바디 시점" 의 스냅샷이라면, 브라우저 DOM 은 "JS 실행 종료 시점" 의 스냅샷이기 때문이다. Selenium 은 실제 브라우저를 원격 조종하므로 후자를 본다.

테스트 — page_source 비교

urllib HTML vs Selenium page_source — JS 렌더링 이전/이후 확인
>>> len(urllib_html)
84231
>>> '_travel_header' in urllib_html
False                                -- 정적 응답에 없음

>>> driver.get(url)
>>> time.sleep(3)
>>> len(driver.page_source)
612847
>>> '_travel_header' in driver.page_source
True                                 -- JS 실행 후 등장

결론(중간)

KEY POINT "HTML 응답" 과 "렌더링된 DOM" 은 다른 두 개의 스냅샷이다. 정적 크롤링은 응답을, 동적 크롤링은 DOM 을 본다. 어느 쪽을 써야 할지는 대상 데이터가 어디서 만들어지는가 가 결정한다.
다음 챕터로 가는 다리 — Selenium 을 쓰기로 했으니 ChromeDriver 가 필요하다. 그런데 크롬은 한 달에도 몇 번씩 업데이트되고, 드라이버가 그 버전을 따라가지 못하면 즉시 성립하지 않는다.

CH 02 DAY5WebDriverManager — 버전 불일치 자동 해결#

학습

# 이전 방식 (수동) — Chrome 업데이트마다 깨짐
driver = wd.Chrome(service=Service('./chromedriver.exe'))

# 개선 — 현재 Chrome 버전에 맞는 드라이버 자동 다운로드/관리
from webdriver_manager.chrome import ChromeDriverManager
driver = wd.Chrome(service=Service(ChromeDriverManager().install()))
WebDriverManager automatically detects the installed browser version and downloads the matching driver binary, caching it locally to avoid repeated network calls on subsequent runs. — webdriver-manager § PyPI

의문 → 가설

가설 ChromeDriverManager().install() 은 매번 다운로드하지 않을 것이다 — 그렇다면 캐시 위치를 가지고 있어야 하고, 설치된 크롬 버전을 자체 검사한 후 캐시에 없을 때만 받아올 것이다. 그 결과로 첫 실행은 느리고 두 번째 실행은 즉시 시작된다.

테스트 — 첫 실행 vs 재실행

결론(중간)

KEY POINT webdriver-manager 는 ① 현재 브라우저 버전 탐지 ② 캐시 조회 ③ 캐시 미스 시만 다운로드의 3단계로 버전 동기화 비용을 자동 흡수한다. 직접 chromedriver.exe 를 프로젝트에 커밋하는 방식은 6개월 만에 무용지물이 된다.
다음 챕터로 가는 다리 — 드라이버 문제가 해결됐다. 하지만 JS 렌더링은 비동기 — driver.get() 이후 DOM 이 아직 동적 구성 중일 수 있다. 그래서 대기 전략이 필요하다.

CH 03 DAY5명시적 / 암묵적 대기 — 비동기 렌더링과 동기 코드의 만남#

학습

from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC
from selenium.webdriver.common.by import By

# 1) 절대적 대기 — 비추천
time.sleep(10)   # 무조건 10초 멈춤

# 2) 명시적 대기 — 권장
element = WebDriverWait(driver, 10).until(
    EC.presence_of_element_located((By.CSS_SELECTOR, '._travel_header'))
)

# 3) 암묵적 대기 — 전역 설정
driver.implicitly_wait(10)   # 모든 find_element 가 최대 10초 대기
Explicit waits are preferred over implicit waits, because they allow waiting for very specific conditions. Mixing explicit and implicit waits can cause unpredictable wait times. — Selenium Documentation — Waits

의문 → 가설

가설 time.sleep(10) 은 "느린 네트워크에서도 안전" 처럼 보이지만 — 정상 환경에서는 9초의 낭비이고, 네트워크가 더 느릴 때는 1초 부족이다. 명시적 대기는 "조건이 만족되면 즉시 진행, 아니면 최대치까지" 라는 정반대 의미를 가지며, 평균 실행 시간을 단축하면서도 안정성을 높인다.

테스트 — 셋의 평균 실행 시간 비교

time.sleep(10) vs WebDriverWait(10) — 평균 실행 시간 비교
-- time.sleep(10) 5회 측정
10.04s, 10.02s, 10.03s, 10.04s, 10.02s   avg = 10.03s

-- WebDriverWait(10).until(presence_of_element) 5회 측정
1.21s, 1.18s, 1.24s, 1.30s, 1.19s         avg = 1.22s

-- 느린 네트워크 시뮬레이션 시
sleep(10): 11초에 element 등장 → 다음 줄에서 NoSuchElementException
WebDriverWait: 11초까지 대기 후 정상 진행 (max 안에 있다면)

결론(중간)

KEY POINT "10초 무조건 멈춤" 과 "최대 10초까지 조건 확인" 은 의미가 완전히 다르다. 명시적 대기는 평균 시간을 단축하면서 가장 느린 케이스도 보호한다. 암묵적 대기와 명시적 대기를 같이 쓰면 대기 시간이 곱셈으로 부풀어 예측 불가능해진다.
다음 챕터로 가는 다리 — 대기 문제가 해결됐다. 이제 선택자 문제다 — 브라우저 배포마다 수시로 바뀌는 해시 클래스명 대신 어떤 선택자가 안정적인가.

CH 04 DAY5선택자 안정화 — 해시 → data 속성 → 태그 구조#

학습 — 다섯 단계 개선

실습 코드 dynamic_crawling.py 는 같은 데이터를 추출하는 과정에서 다섯 번의 선택자 개선을 거쳤다. 그 흐름이 동적 크롤링 안정화의 표준 절차를 그대로 보여준다.

단계선택자실패 원인개선 방향
1'./chromedriver.exe'Chrome 버전 불일치WebDriverManager 자동화
2전체 CSS 경로 + 해시 클래스해시 변경안정적 클래스만 남기기
3탭 클릭 — 해시 포함 경로해시 변경_scroll_wrapper 만 사용
4class*="TopPoiItem-" 부분매칭네이버가 pui__ 체계로 교체data-otb_lsra-doc-id
5selenium find_element 다중 호출해시 클래스 무더기 + 느림BeautifulSoup 일괄 파싱 + 태그 구조 기반

의문 → 가설

가설 "안정적 선택자" 의 정의는 — "이 선택자의 의미를 사람이 한 줄로 설명할 수 있는가" 다. TopPoiItem-MgXeO 는 설명 불가 → 해시. data-otb_lsra-doc-id 는 "OTB 위치 문서 ID" → 의미 있음. 의미가 있다는 것은 "팀이 그 이름을 그대로 유지할 동기" 가 있다는 뜻이다.

테스트 — 정규식으로 별점 추출

import re
byeoljum_span = item_soup.find('span', string='별점')
if byeoljum_span:
    raw = byeoljum_span.parent.get_text()         # "별점4.7점"
    m = re.search(r'(\d+\.?\d*)', raw.replace('별점','').replace('점',''))
    score = round(float(m.group(1)), 1) if m else 0.0
별점 정규식 추출 실행 결과
>>> raw = byeoljum_span.parent.get_text()
'별점4.7점'
>>> m = re.search(r'(\d+\.?\d*)', raw.replace('별점','').replace('점',''))
>>> m.group(1)
'4.7'
>>> score
4.7   -- 한국어 라벨을 선택자로 활용해 필드 정확히 추출

결론(중간)

KEY POINT "별점" 이라는 한국어 텍스트 자체를 선택자로 쓰는 것은 클래스 해시보다 훨씬 안정적이다 — 네이버가 한국어 라벨을 바꾸는 일은 클래스명을 바꾸는 일보다 훨씬 드물기 때문. "사람이 보는 것에 가까운 선택자가 안정적이다" 가 동적 크롤링의 원칙이다.
다음 챕터로 가는 다리 — 단발 크롤러는 수동 실행이다. 주기적 자동 수집은 APScheduler 스케줄러와 결합해야 실력 라인을 넘는다.

CH 05 DAY5자동화로 한 걸음 더 — auto_crawler 의 APScheduler 패턴#

학습

실습 후반은 auto_crawler 프로젝트 — Streamlit 대시보드 + APScheduler BackgroundScheduler 로 "주기적으로 자동 수집" 흐름을 다뤘다. 

from apscheduler.schedulers.background import BackgroundScheduler

scheduler = BackgroundScheduler()
scheduler.add_job(crawl_job, 'interval', minutes=30, id='naver_movies')
scheduler.add_job(crawl_job, 'cron', hour=9, minute=0, id='daily_report')
scheduler.start()

의문 → 가설

가설 BackgroundScheduler 는 Streamlit 메인 스레드와 독립적으로 돌아간다 → 사용자가 페이지를 보든 안 보든 크롤링이 진행된다. 그러나 동시에 ① 같은 작업이 중복 실행되지 않도록 락 ② 실패 시 재시도 ③ DB 연결 풀링이 필요해진다.

테스트 — 스케줄러 시작 및 잡 등록 확인

BackgroundScheduler.start() — 잡 등록 및 실행 결과
>>> scheduler.start()
>>> scheduler.get_jobs()
[<Job (id=naver_movies trigger: interval[0:30:00])>,
 <Job (id=daily_report  trigger: cron[hour='9', minute='0'])>]
-- 30분 후 첫 실행
[2025-05-15 10:00:00] crawl_job 시작
[2025-05-15 10:00:03] 크롤링 완료: 10건 저장   -- 메인 스레드 블록 없이 실행
BackgroundScheduler runs scheduled jobs in a separate background thread so that it does not block the main application thread. Jobs can be managed by their ID to prevent duplicate scheduling. — APScheduler Documentation § Schedulers overview

결론(중간)

KEY POINT 자동화는 단발 크롤러 + 스케줄러의 단순 결합이 아니다 — 중복 실행 방지, 재시도, 백오프, 로그, 알림이 모두 얹어지면서 비로소 운영 가능한 시스템이 된다. APScheduler 는 ID 기반 잡 관리로 그 출발점을 제공한다.
다음 챕터로 가는 다리 — 대기 · 선택자 안정화 · 자동화, 이 세 축이 하나의 결론으로 합쳐진다.

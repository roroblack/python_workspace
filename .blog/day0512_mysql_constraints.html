CH 01 DAY2NOT NULL — 비울 수 없다는 가장 단순한 차단#

학습

제일 단순한 제약조건이다. "이 컬럼은 NULL 이 들어오면 거부한다." 그러나 단순한 만큼 두 가지 함정이 따른다 — NOT NULL 은 컬럼 레벨에서만 정의 가능하고, 0511 의 "빈 값" 자동 NULL 매커니즘과 정면 충돌한다. 

create table tbl_notnull (
    nid     char(3)     not null,   -- 컬럼 레벨에서만 가능
    sname   varchar(20)
);

insert into tbl_notnull (nid, sname) values ('100', 'mysql');   -- ok
insert into tbl_notnull values (null, 'mysql');                  -- error
insert into tbl_notnull values ('200', null);                    -- sname 은 NN 아님 ok

-- 테이블 레벨 작성 시도 → 즉시 문법 에러
create table tbl_notnull2 (
    nid     char(3),
    sname   varchar(20),
    constraint nn_tn2_nid not null (nid)   -- error
);
NOT NULL specifies that a column cannot contain the NULL value. This constraint can only be specified at the column level; specifying it as a table constraint is not allowed. — MySQL Reference Manual § NOT NULL Optimization

의문 → 가설

가설 NOT NULL 이 컬럼 레벨로만 제한된 이유는 — "이 컬럼이 NULL 인가" 는 다른 컬럼을 볼 필요 없이 한 셀로 판정 가능 한 조건이기 때문이다. 즉 NN 은 본질적으로 컬럼의 속성이고, 테이블 레벨로 끌어올릴 의미가 없다.

테스트 — NN 위반 / 정상 INSERT

INSERT INTO tbl_notnull — NOT NULL 위반 ERROR 1048 확인
Query OK, 1 row affected (0.00 sec)

ERROR 1048 (23000): Column 'nid' cannot be null

Query OK, 1 row affected (0.00 sec)

+-----+-------+
| nid | sname |
+-----+-------+
| 100 | mysql |
| 200 | NULL  |
+-----+-------+
2 rows in set (0.00 sec)

결론(중간)

KEY POINT NOT NULL = "값이 반드시 존재해야 한다" 는 가장 단순한 1셀 규칙. ERROR 1048 (23000) 이 그것의 코드명. 가설 통과 — 다른 컬럼을 참조할 필요가 없기에 테이블 레벨 정의가 의미를 가지지 못한다.
다음 챕터로 가는 다리 — "값이 존재해야 한다" 는 충족되었지만, 두 번 들어오는 같은 값을 막지는 못한다. 그래서 두 번째 제약조건이 등장한다 — UNIQUE.

CH 02 DAY2UNIQUE — "중복" 의 의미를 단일키와 복합키가 다르게 정의한다#

학습

UNIQUE 는 컬럼 레벨/테이블 레벨 둘 다 정의 가능하며, 복합키(여러 컬럼 묶음) 도 지원한다. 여기서 "두 행이 같다" 는 정의 자체가 단일키와 복합키에서 달라진다. 

create table tbl_unique2 (
    uid     char(3)     unique,          -- 단일 unique (컬럼 레벨)
    sname   varchar(20),
    scode   char(2),
    constraint uq_tun2_comp unique (uid, sname)   -- 복합 unique (테이블 레벨)
);

insert into tbl_unique2 values ('100', 'mysql', '01');  -- ok
insert into tbl_unique2 values ('200', 'mysql', '01');  -- ok (uid 다름, 복합 다름)
insert into tbl_unique2 values ('100', 'mysql', '04');  -- error (둘 다 중복)
insert into tbl_unique2 values ('100', 'web',   '03');  -- error (uid 단일이 중복)
insert into tbl_unique2 values (null,  'mysql', '01');  -- ok (NULL 은 허용)
A UNIQUE index permits multiple NULL values for columns. Because NULL is considered distinct from every other value, including itself, two rows with NULL in a UNIQUE column are not considered to be duplicates. — MySQL Reference Manual § CREATE INDEX Statement

의문 → 가설

가설 같은 행에 단일 UNIQUE 와 복합 UNIQUE 가 동시에 걸려 있다면 — INSERT 한 줄을 평가할 때 두 검사가 독립적으로 작동 하고, 둘 중 어느 하나라도 위반되면 거부된다. 또한 UNIQUE 는 NN 과 달리 NULL 을 "값 없음" 으로 보아 중복 판정에서 제외할 것이다.

테스트 — 단일 vs 복합 UNIQUE

INSERT INTO tbl_unique2 — 단일/복합 UNIQUE 위반 ERROR 1062 확인
Query OK, 1 row affected (0.01 sec)
Query OK, 1 row affected (0.00 sec)
ERROR 1062 (23000): Duplicate entry '100-mysql' for key 'tbl_unique2.uq_tun2_comp'
ERROR 1062 (23000): Duplicate entry '100' for key 'tbl_unique2.uid'
Query OK, 1 row affected (0.00 sec)

+------+-------+-------+
| uid  | sname | scode |
+------+-------+-------+
| 100  | mysql | 01    |
| 200  | mysql | 01    |
| NULL | mysql | 01    |
+------+-------+-------+
3 rows in set (0.00 sec)

결론(중간)

KEY POINT 같은 ERROR 1062 라도 충돌한 키 이름이 다르게 출력된다 — tbl_unique2.uid 는 단일 UNIQUE, tbl_unique2.uq_tun2_comp 는 복합 UNIQUE 위반. UNIQUE 는 NULL 을 무시한다. 즉 "중복" 의 정의가 키마다 다르고, MySQL 은 그 정의를 키 이름과 함께 정확히 알려준다.
다음 챕터로 가는 다리 — NN("비어선 안 됨") 과 UNIQUE("두 번 안 됨") 을 묶으면 "한 행을 식별하는 단 하나의 값" 이 된다. 그것이 PRIMARY KEY 다.

CH 03 DAY2PRIMARY KEY — NOT NULL + UNIQUE 의 결합, 그리고 식별의 책임#

학습

PRIMARY KEY 는 NOT NULL 과 UNIQUE 가 자동으로 따라붙는 결합형 제약조건이다. 그러나 결합 그 자체보다 더 중요한 제약이 따라온다 — 한 테이블당 PK 는 단 한 개 만 허용된다. 

create table tbl_pk (
    pid     char(3)     primary key,
    sname   varchar(20)
);

insert into tbl_pk values ('100', 'mysql');         -- ok
insert into tbl_pk values ('100', 'python');        -- error 1062 (UK 부분)
insert into tbl_pk values (null,  'python2');       -- error 1048 (NN 부분)

-- PK 두 개 시도 → 테이블 자체가 만들어지지 않음
create table tbl_pk2 (
    pid char(3) primary key,
    sname varchar(20) primary key   -- error 1068
);

-- 복합 PK 는 가능 (테이블 레벨)
create table tbl_pk2 (
    pid char(3), sname varchar(20), scode char(2),
    constraint pk_tpk2_comp primary key (pid, sname)
);
A table can have only one PRIMARY KEY. It implicitly applies both NOT NULL and UNIQUE constraints. The primary key can consist of one column or multiple columns, forming a composite primary key. — MySQL Reference Manual § PRIMARY KEY and UNIQUE Index Constraints

의문 → 가설

가설 "한 테이블당 PK 하나" 라는 제한은 자의적 규칙이 아니라 식별의 정의 자체 다 — "이 한 값(또는 한 묶음)을 보면 그 행을 정확히 한 개만 찾을 수 있다" 가 식별이라면, 식별자가 둘이면 모순이 생긴다. 그러므로 PK 위반은 두 가지 코드로 갈라져야 한다 — 1062(같은 값 두 번) 와 1048(값 비움).

테스트 — PK 위반 시나리오

INSERT INTO tbl_pk — PK 위반 ERROR 1062 / 1048 / 1068 확인
Query OK, 1 row affected (0.01 sec)
ERROR 1062 (23000): Duplicate entry '100' for key 'tbl_pk.PRIMARY'
ERROR 1048 (23000): Column 'pid' cannot be null
ERROR 1068 (42000): Multiple primary key defined

Query OK, 0 rows affected (0.02 sec)

+-----+-------+-------+
| pid | sname | scode |
+-----+-------+-------+
| 100 | mysql | 01    |
| 200 | mysql | 01    |
+-----+-------+-------+
2 rows in set (0.00 sec)

결론(중간)

KEY POINT PK = NN + UK 의 결합이라는 것이 ERROR 1062 와 ERROR 1048 양쪽 코드로 입증되었다. tbl_pk.PRIMARY 라는 키 이름은 PK 가 내부적으로 PRIMARY 라는 예약 키 이름으로 관리됨을 의미한다. "한 테이블 한 PK" 는 RDB 의 식별 모델 그 자체다.
다음 챕터로 가는 다리 — NN/UK/PK 는 모두 "값의 존재 여부와 중복 여부" 를 다뤘다. 그런데 "급여는 양수여야 한다" 처럼 값의 범위 를 강제하려면 새로운 종류의 제약이 필요하다 — CHECK.

CH 04 DAY2CHECK — InnoDB 위에서만 살아있는 도메인 규칙#

학습

CHECK 는 MySQL 8.0 + InnoDB 조합에서만 정상 작동한다. 그 위에서 "어떤 식이 참인 값만 통과" 라는 자유로운 도메인 제한을 건다. 

create table tbl_check (
    emp_id   char(3)  primary key,
    salary   int      check (salary > 0),
    marriage char(1),
    constraint ck_tbchk_marriage check (marriage in ('Y','N'))
) ENGINE=InnoDB;

insert into tbl_check values ('100',  350, 'Y');   -- ok
insert into tbl_check values ('120', -450, 'Y');   -- error 3819 (salary > 0 위반)
insert into tbl_check values ('100',  350, 'K');   -- error 3819 (marriage IN 위반)

-- 변동값(NOW 등) 사용 시도 → 정의 자체가 거부됨
create table tbl_check2 (
    emp_id char(3) primary key,
    hiredate datetime check (hiredate < current_timestamp)
) ENGINE=InnoDB;   -- error
The CHECK constraint feature is implemented for all storage engines. … For each row checked, if the expression evaluates to FALSE, a violation occurs. NULL values are treated as not violating. Non-deterministic functions such as NOW() are not permitted. — MySQL Reference Manual § CHECK Constraints

의문 → 가설

가설 CHECK 가 변동값을 거부하는 이유는 — 제약은 "언제 검사해도 같은 결과" 가 보장 되어야 하기 때문이다. NOW() 를 허용하면 "어제 통과한 행이 오늘 위반" 일 수 있다. 즉 CHECK 는 행 단위·INSERT 시점에 결정론적으로 평가 가능한 식만 허용한다.

테스트 — CHECK 위반

INSERT INTO tbl_check — CHECK 위반 ERROR 3819 / 3814 확인
Query OK, 1 row affected (0.01 sec)
ERROR 3819 (HY000): Check constraint 'tbl_check_chk_1' is violated.
ERROR 3819 (HY000): Check constraint 'ck_tbchk_marriage' is violated.

ERROR 3814 (HY000): An expression of a check constraint contains disallowed function.

결론(중간)

KEY POINT 이름 없이 정의한 CHECK 는 테이블명_chk_숫자 로 자동 명명되고, constraint 이름 으로 명시한 것은 그 이름이 그대로 에러에 등장한다 → 제약 이름 짓기는 디버깅의 직접적 도구다. 변동값 금지(ERROR 3814) 는 "제약은 결정론적이어야 한다" 는 가설을 그대로 입증한다.
다음 챕터로 가는 다리 — 지금까지 모든 제약은 "한 테이블 안" 에서 끝났다. 두 테이블을 묶는 관계는 어떻게 강제하는가? FOREIGN KEY 다.

CH 05 DAY2FOREIGN KEY — 두 테이블을 묶는 관계, ON DELETE / ON UPDATE#

학습

FK 는 자식 테이블의 컬럼이 부모 테이블의 PK 또는 UNIQUE 컬럼만 참조하도록 강제한다. 실습은 location 부모 테이블을 만들고 tbl_fk* 자식 테이블에서 FK 를 거는 흐름이었다. 핵심 함정 세 가지 — ① 엔진이 MyISAM 이면 FK 가 조용히 무시된다 ② 컬럼 레벨 FK 는 종종 무시되니 테이블 레벨 권장 ③ 부모키는 PK / UNIQUE 만 가능. 

-- 잘못된 예 1: 엔진 미지정 + 컬럼 레벨 FK
create table tbl_fk (
    fid char(3),
    sname varchar(20),
    loc_id char(2) references location(location_id)   -- ★ 무시될 수 있음
);

-- 권장 형식: 테이블 레벨 + ENGINE=InnoDB
create table tbl_fk3 (
    fid char(3),
    sname varchar(20),
    loc_id char(2),
    constraint fk_tblfk3_lid foreign key (loc_id)
        references location(location_id)
) ENGINE=InnoDB;

insert into tbl_fk3 values ('333','web','B2');     -- error 1452 (부모키 없음)

-- 복합 PK 부모 + ON DELETE / ON UPDATE 옵션
create table tbl_fk5 (
    fid char(3), fname varchar(20), fcode char(2),
    constraint fk_tfk5_comp foreign key (fid, fname)
        references tbl_pk2 (pid, sname)
        on delete set null
        on update cascade
) ENGINE=InnoDB;
InnoDB permits a foreign key to reference any column or group of columns. However, in the referenced table, there must be an index where the referenced columns are the first columns in the same order. … Hidden columns and the storage engine do not have to be specified. — MySQL Reference Manual § 13.1.20.5 FOREIGN KEY Constraints

의문 → 가설

가설 FK 는 두 테이블의 데이터 일관성을 행 단위로 묶는다. 그렇다면 부모 행이 사라질 때 자식이 어떻게 반응해야 하는가가 설계 결정의 핵심이 된다 — ON DELETE 옵션이 그것을 표현한다. set null 은 "관계만 끊는다", cascade 는 "함께 따라간다", 생략하면(기본) "삭제 자체를 거부한다".

테스트 — FK 위반 / cascade 동작

INSERT INTO tbl_fk3 — FK 위반 ERROR 1452 · CASCADE UPDATE 확인
ERROR 1452 (23000): Cannot add or update a child row: a foreign key constraint fails
(`mydb`.`tbl_fk3`, CONSTRAINT `fk_tblfk3_lid` FOREIGN KEY (`loc_id`) REFERENCES `location` (`location_id`))

Query OK, 1 row affected (0.01 sec)
-- 부모 tbl_pk2 의 ('100','mysql') 을 ('100','MySQL') 로 update 하면
-- 자식 tbl_fk5 도 cascade 로 따라서 변경됨
+-----+-------+-------+
| fid | fname | fcode |
+-----+-------+-------+
| 100 | MySQL | 33    |
+-----+-------+-------+
1 row in set (0.00 sec)
CAUTION 엔진이 InnoDB 가 아니면 FK 정의가 받아들여져도 조용히 무시 된다 — 에러 없이 자식이 부모 없는 값을 가질 수 있다. 실습 중 가장 위험한 버그였다. 해결책: 모든 테이블 생성에 ENGINE=InnoDB 명시.

결론(중간)

KEY POINT FK 는 부모-자식 관계를 행 단위로 강제하고, ON DELETE/UPDATE 옵션이 부모의 변경이 자식에 어떻게 전파되는지를 결정한다. "관계" 라는 개념이 RDB 의 R(Relational) 을 정의한다.
다음 첵터로 가는 다리 — NN / UK / PK / CHECK / FK 다섯 제약에 대한 실험이 모두 끝났다. 각 제약이 한 가지 종류의 "잘못된 행"을 정의한다는 사실이 하나의 결론으로 합쳐진다.

CH 06최종 결론 — 다섯 제약조건은 "잘못된 행" 의 다섯 정의다#

최종 정리

출발 의문은 — "자료형 위에 또 한 층의 차단이 왜 필요한가" 였다. 다섯 제약을 한 줄씩 다시 쓰면 각각이 다른 종류의 "잘못된 행" 을 정의한다.

제약잘못된 행의 정의대표 에러레벨
NOT NULL값이 비어있는 행1048컬럼만
UNIQUE(단일/복합) 같은 키를 두 번 가진 행1062컬럼/테이블
PRIMARY KEY식별자가 비었거나 중복된 행1062 / 1048컬럼/테이블 (단 1개)
CHECK식이 FALSE 인 행3819컬럼/테이블 (결정론식만)
FOREIGN KEY부모가 제공하지 않는 값을 가진 행1452테이블 권장 (InnoDB 필수)

최종 결론

FINAL CONCLUSION 0511 의 자료형은 "셀 단위 차단" 이었다. 0512 의 다섯 제약은 "행 단위 차단" 이다. 각각이 한 가지 종류의 "잘못된 행" 정의를 가지고 INSERT/UPDATE 시점에 작동한다. 파이썬으로 비유하면 — 자료형은 isinstance 검사 같은 것이고, 제약조건은 ABC 의 __init_subclass__ 훅으로 객체 생성 자체를 차단하는 패턴에 가깝다. "잘못된 데이터가 시스템 안으로 들어오지 못하게 한다" 는 한 목적이 자료형 → 제약 → (다음 단계 인덱스/트랜잭션) 으로 층층이 쌓인다.

다음 의문은 — 행이 들어왔으면 이제 꺼내야 한다. 22 행짜리 직원 테이블에서 "급여가 300만원 이상이면서 90 번 부서" 같은 조건을 만족하는 행만 어떻게 골라내는가? 그것이 DAY3(0513) 의 SELECT · WHERE · JOIN 의 출발 의문이 된다.

CH 01 DAY1root 와 일반 사용자 — 왜 DBMS 는 OS 와 분리된 계정 체계를 갖는가#

학습

MySQL 을 설치하면 OS 사용자(Windows 의 내 계정)와 전혀 별도 의 root 계정이 생긴다. DAY1.sql 의 출발점은 이 root 가 자기 자신을 쓰지 않고 학습용 계정을 새로 만들어 권한을 위임 하는 작업이었다. 

-- DAY1.sql — root 가 student / homework 계정을 만들고 권한을 부여
CREATE USER 'student'@'%'   IDENTIFIED BY 'Student80*';
CREATE USER homework        IDENTIFIED BY 'Homework80*';

SHOW DATABASES;
GRANT ALL privileges ON mysql.* TO homework;  -- 거의 모든 권한 위임

-- 현재 접속 계정 확인
SELECT USER();
MySQL stores accounts in the user table of the mysql system database. An account is defined in terms of a user name and the client host or hosts from which the user can connect to the server. — MySQL Reference Manual § 6.2.4 Specifying Account Names

의문 → 가설

가설 위 인용이 사실이라면 — MySQL 의 계정은 ① 호스트 페어 ('user'@'host') 단위로 식별되고 ② mysql.user 라는 시스템 테이블 한 곳에 모두 기록되며 ③ OS 계정과는 무관하게 독립 운영되어야 한다. 그렇다면 같은 사용자명이라도 호스트가 다르면 다른 계정으로 취급되어야 하고, mysql.user 를 직접 조회해서 위 세 가지를 한 번에 검증할 수 있다.

테스트 — mysql.user 직접 조회

-- root 계정으로 접속한 상태에서
SELECT user, host, plugin FROM mysql.user;

결론(중간)

KEY POINT — 가설 통과 계정은 (user, host) 페어로 식별된다 — 같은 이름이라도 'admin'@'localhost' 와 'admin'@'%' 는 다른 계정이다. OS 의 Windows 사용자명과는 전혀 무관하게, MySQL 안에서 자기만의 사용자 테이블을 갖고 있다. 이것이 DBMS 가 OS 와 분리된 계정 체계를 갖는 이유 — "누가 어디서 어떤 데이터에 접근하느냐" 를 OS 가 아닌 DBMS 자체가 책임지기 위함이다.
다음 챕터로 가는 다리 — 계정이 권한을 가지려면 그 권한이 "무엇에 대한 권한" 인지를 가리킬 단위가 필요하다. GRANT ALL ON mysql.* TO homework 에서 등장한 mysql.* 의 점(.) 표기 — 이것이 다음 단위인 데이터베이스(스키마) 의 핵심이다.

CH 02 DAY1DATABASE / SCHEMA — 한 서버 안의 격리 단위#

학습

DAY1.sql 두 번째 흐름은 root 가 CREATE DATABASE mydb 로 데이터베이스 한 칸을 새로 파고, student 계정이 USE mydb 로 그 안에 들어가는 동작이었다. MySQL 에서는 SCHEMA = DATABASE 로 같은 의미이며, 한 서버 안에 여러 개를 두고 격리할 수 있다. 

-- root 계정에서
CREATE DATABASE mydb;
SHOW DATABASES;            -- 새로 만들어진 mydb 확인

-- student 계정 (Connect to Database 로 재접속) 에서
USE mydb;                  -- 작업 대상 데이터베이스 지정
SELECT DATABASE();         -- 현재 어느 DB 에 들어와 있는가
In MySQL, SCHEMA is a synonym for DATABASE. They are equivalent in MySQL. Use USE db_name to set db_name as the default (current) database for subsequent statements. — MySQL Reference Manual § 13.8.4 USE Statement

의문 → 가설

가설 데이터베이스가 진짜 "격리 단위" 라면 — ① 한 서버 안에 여러 DB 가 동시에 살아있을 수 있고 ② 같은 이름의 테이블이 다른 DB 에 있으면 서로 충돌하지 않으며 ③ 권한도 db명.테이블명 단위로 부여될 수 있어야 한다. DAY1 에서 직접 mydb 와 mysql 두 DB 를 한 서버에서 동시에 다뤘으니, 이 셋이 모두 만족되어야 한다.

테스트 — SHOW DATABASES · 현재 컨텍스트 확인

SHOW DATABASES;
USE mydb;
SELECT DATABASE();
SHOW TABLES;

KEY POINT — 가설 통과 한 MySQL 서버는 여러 데이터베이스를 동시에 호스팅한다 — information_schema, mysql, performance_schema, sys 가 시스템 DB 이고 그 옆에 사용자 DB mydb 가 새로 들어가 있다. db명.테이블명 의 점 표기가 권한 범위를 그대로 그린다는 점에서, 데이터베이스는 파이썬의 모듈/패키지 네임스페이스와 같은 격리 역할을 한다. 단, 파이썬이 sys.modules 런타임 dict 로 그것을 표현했다면, MySQL 은 디스크 상의 독립된 디렉터리 로 표현한다.
다음 챕터로 가는 다리 — 격리 단위(DB) 안에 들어왔으니 이제 실제 데이터를 담을 그릇이 필요하다. 파이썬 list/dict 가 무엇이든 담았던 것과 달리, MySQL 의 테이블은 컬럼마다 자료형을 미리 정의 하라고 강제한다. 왜 그렇게 까다로운가?

CH 03 DAY1CREATE TABLE — 컬럼마다 자료형을 강제하는 이유#

학습

DAY1_student.sql 의 핵심은 CREATE TABLE MANAGE 였다. 영업소명(VARCHAR(20)) · 입고량(INT) · 판매량(INT) · 재고량(INT) · 날짜(DATE) — 파이썬이라면 dict 한 줄로 끝났을 정보를 컬럼마다 자료형까지 명시했다. 

CREATE TABLE MANAGE (
    MARKET  VARCHAR(20),   -- 영업소명 (0~20 바이트 가변)
    BUY     INT,           -- 입고량
    SELL    INT,           -- 판매량
    STOCK   INT,           -- 재고량
    DAY     DATE           -- YYYY-MM-DD
);

DESC MANAGE;               -- 컬럼 구성 확인
Data types specify what type of data can be stored in a column. … Different data types have different storage requirements and use different rules for comparison, sorting, and arithmetic. — MySQL Reference Manual § 11 Data Types

인용문이 말하는 두 가지가 핵심이다 — 저장 공간이 자료형마다 다르고, 비교·정렬·연산의 규칙도 자료형이 결정한다. CHAR 와 VARCHAR 의 차이도 이 맥락에서 나온다.

자료형저장 방식DAY1 학습 핵심
CHAR(n)고정 길이 n 바이트 — 부족하면 공백으로 채움'a' 를 CHAR(2) 에 넣으면 'a ' 로 기록
VARCHAR(n)0 ~ n 바이트 가변 — 실제 입력값만 저장공백 패딩 없음
INT4 바이트 정수 (-2,147,483,648 ~ 2,147,483,647)UNSIGNED 키워드로 0~4,294,967,295 전환 가능
DATE · DATETIME날짜 전용 8 바이트 · 11 바이트현재 시각 함수 NOW() 자동 변환

의문 → 가설

가설 자료형 강제가 단지 "저장 효율" 만을 위한 것이라면 굳이 INSERT 시점에 에러를 낼 필요가 없다. 그러나 인용문이 말하는 "비교·정렬·연산 규칙이 자료형으로 결정된다" 가 사실이라면 — ① 자료형이 맞지 않는 값을 INSERT 하려는 시점에 즉시 거부되고 ② DESC 같은 메타 명령으로 자료형 정보를 항상 조회할 수 있어야 한다. 파이썬 ABC 가 객체 생성 시점에 차단했던 것과 같은 패턴이 SQL 에는 INSERT 시점에 적용되어 있을 것이다.

테스트 — DESC · INSERT 타입 위반

DESC MANAGE;

-- 정상 INSERT
INSERT INTO MANAGE VALUES ('A 영업소', 100, 80, 20, '2026-05-11');

-- 의도적 타입 위반: BUY 자리에 문자열
INSERT INTO MANAGE VALUES ('B 영업소', 'one hundred', 80, 20, '2026-05-11');

-- DATE 자리에 잘못된 형식
INSERT INTO MANAGE VALUES ('C 영업소', 100, 80, 20, '11/05/2026');

KEY POINT — 가설 통과 INSERT 시점에 자료형 검사가 작동한다 — ERROR 1366 · ERROR 1292 가 파이썬 ABC 의 TypeError 와 같은 자리를 차지한다. 파이썬은 "객체 생성 시점에 차단" 으로 인터페이스를 강제했고, MySQL 은 "행 삽입 시점에 차단" 으로 도메인을 강제한다 — 검출 시점만 다를 뿐 "잘못된 데이터를 시스템 안에 들이지 않는다" 는 정신은 같다.
다음 챕터로 가는 다리 — 자료형이 강제되는 이유는 이해했다. 그런데 MANAGE 테이블에는 빠진 것이 하나 있다 — 각 행을 유일하게 식별하는 PRIMARY KEY. PK 를 INT AUTO_INCREMENT 로 두는 것이 왜 단순 관례가 아닌 B+Tree 성능 설계 인지 다음 챕터에서 파고든다.

CH 04 DAY1PRIMARY KEY · B+Tree — INT AUTO_INCREMENT 가 기본 설계인 이유#

학습

DAY1_student.sql 의 ACCOUNT 테이블에는 PRIMARY KEY 가 없었다. 그런데 실습을 진행하며 두 가지 의문이 생겼다 — PK 는 왜 존재하며, 왜 INT AUTO_INCREMENT 가 사실상 관례인가. 그리고 PK 를 굳이 숫자로 줘야 하는가 — '20240511-001' 처럼 날짜+순번 문자열도 고유 식별자로 동작하지 않는가. 

-- DAY1 ACCOUNT: PK 없음 — 행을 구별할 수단이 없다
CREATE TABLE ACCOUNT (
    LIST   VARCHAR(40),
    TYPE   VARCHAR(5),
    AMOUNT INT,
    DAY    VARCHAR(30)
);

-- PK + AUTO_INCREMENT 를 추가한 버전
CREATE TABLE ACCOUNT2 (
    ID     INT AUTO_INCREMENT PRIMARY KEY,  -- 행 식별자 자동 부여
    LIST   VARCHAR(40),
    TYPE   VARCHAR(5),
    AMOUNT INT,
    DAY    VARCHAR(30)
);

INSERT INTO ACCOUNT2 (LIST, TYPE, AMOUNT, DAY)
VALUES ('보험료', '지출', 20, '05-01'),
       ('급여',   '수입', 450, '05-08'),
       ('통신비', '지출',  10, '05-11');

SELECT * FROM ACCOUNT2;
SELECT LAST_INSERT_ID();   -- 마지막 삽입된 AUTO_INCREMENT 값

의문 → 가설

가설 MySQL InnoDB 는 PK 인덱스를 B+Tree 로 관리하며, 행 데이터를 PK 순서로 물리 배치한다 (Clustered Index). 문자열 PK 는 삽입 순서와 정렬 순서가 달라 기존 페이지 중간에 끼워야 하는 상황이 반복되고 — 16 KB 페이지가 꽉 차면 페이지 분열(Page Split) 이 일어나 삽입 성능이 저하된다. INT AUTO_INCREMENT 는 항상 단조증가 → 새 행이 항상 마지막 리프 노드 끝에 붙음 → Page Split 없음. 이 가설이 사실이라면 PK 타입 선택은 취향이 아니라 B+Tree 성능 설계다.

테스트 — B+Tree 구조 추적

InnoDB 는 PK 인덱스를 B+Tree 구조로 저장한다. 루트(Root) → 브랜치(Branch) → 리프(Leaf) 3층 구조이며, 실제 행 데이터는 리프 노드에만 있다 — 이것이 Clustered Index 다. 리프 노드끼리는 이중 연결 리스트로 이어져 있어 범위 검색이 루트로 다시 올라갈 필요 없이 수평 이동만으로 끝난다.

B+Tree Clustered Index — InnoDB 내부 개념도
[ Root Node ]  "1~400 → A 페이지 / 401~800 → B 페이지"
               ↓ O(log n) 수직 탐색
    [ Branch Node ]  "401~600 → C 페이지 / 601~800 → D 페이지"
                      ↓
[ Leaf Node ] ←→ [ Leaf Node ] ←→ [ Leaf Node ]    (이중 연결 리스트)
  행401 전체           행501 전체          행601 전체
  행402 전체           행502 전체          …  (1 노드 = 16 KB 페이지, 최소 2 행 이상)

-- WHERE id BETWEEN 500 AND 510
-- 500 을 찾기 위해 한 번만 수직 하강 → 이후 리프 노드 수평 이동만으로 완료
-- 탐색 O(log n)  Fan-out 이 크므로 실제 높이 3~4 단계 — O(1) 처럼 느껴짐

-- INT AUTO_INCREMENT PK: 항상 단조증가 → 새 행이 항상 마지막 리프 끝에 붙음
--   → Page Split 없음, 인덱스 단편화 없음 (삽입 최적)
-- VARCHAR PK ('2026-05-01', '2025-12-31', ...): 정렬 기준 불규칙
--   → 삽입마다 기존 리프 중간에 끼워야 함 → Page Split 빈발 → 삽입 성능 저하
The clustered index is not a separate kind of index. Rather, it is the way of storage for the data. When you define a PRIMARY KEY on your table, InnoDB uses it as the clustered index. If a table has no PRIMARY KEY or suitable unique index, InnoDB generates a hidden clustered index named GEN_CLUST_INDEX on a synthetic column containing row ID values. — MySQL Reference Manual § 15.6.2.1 Clustered and Secondary Indexes

AUTO_INCREMENT vs SEQUENCE — DBMS 별 비교

DBMS 마다 자동 증가 번호를 구현하는 방식이 다르다. MySQL 은 테이블 속성 으로 묶었고, Oracle · PostgreSQL 은 테이블과 독립된 시퀀스(SEQUENCE) 객체 로 분리했다. 

-- MySQL: 컬럼 속성 — 테이블에 종속, 해당 컬럼은 반드시 KEY 여야 함
CREATE TABLE users (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50)
);
ALTER TABLE users AUTO_INCREMENT = 1000;  -- 시작값 변경

CREATE SEQUENCE my_seq START 100;
CREATE TABLE products (
    id   INT DEFAULT nextval('my_seq') PRIMARY KEY,
    name TEXT
);

GAP 현상 — 번호 소모 트랜잭션이 롤백되어도 이미 할당된 AUTO_INCREMENT 번호는 돌아오지 않는다. 1, 2, 3, 5, 6 … 처럼 중간이 비어있어도 이것은 버그가 아니다 — 락 경합을 줄이기 위해 번호를 먼저 발급하고 커밋 실패 시 회수하지 않는 것이 의도적 설계다. PK 는 "고유 식별자" 이지 "연속 일련번호" 가 아니다.

결론(중간)

KEY POINT — 가설 통과 InnoDB 의 Clustered Index 는 PK 순서로 데이터를 물리 배치한다. INT AUTO_INCREMENT 는 단조증가 → Page Split 없음 → 삽입 최적. VARCHAR PK 는 정렬 기준 불규칙 → Page Split 빈발 → 삽입 성능 저하. Secondary Index 리프 노드에 PK 값 자체가 기록되므로, VARCHAR PK 는 인덱스 크기도 함께 커진다. "날짜+순번" 같은 문자열 식별자가 비즈니스적으로 필요하다면, DB 내부는 INT PK 로 두고 앱 레이어에서 문자열로 변환하는 책임 분리가 정석이다 — PK 타입 변경이 필요해지더라도 서비스 레이어 코드를 건드리지 않아도 된다.
다음 챕터로 가는 다리 — PK 설계까지 마쳤다. 이제 이 스키마를 나중에 변경할 수 있는가 — ALTER TABLE … RENAME 과 DROP TABLE IF EXISTS 로 "정적이지만 진화 가능한" 스키마의 두 번째 면을 본다.

CH 05 DAY1ALTER / DROP — 정적 스키마는 정말 굳어 있는가#

학습

DAY1_student.sql 후반은 테이블의 "변경" 을 다뤘다. 이름 바꾸기 · 컬럼 추가/제거/자료형변경 · 조건부 삭제 — 모두 ALTER TABLE 한 동사로 묶여 있다. 

-- 1) 이름 변경 — 한 개
ALTER TABLE MANAGE RENAME MANAGE2;

-- 2) 여러 개 한 번에
RENAME TABLE MANAGE2 TO MANAGE, MANAGE TO MANAGE3;

-- 3) 조건부 제거
DROP TABLE IF EXISTS MANAGE2;     -- 없으면 조용히 통과
DROP TABLE MANAGE3;               -- 없으면 에러
ALTER TABLE changes the structure of a table. … Most ALTER TABLE operations require a table rebuild, although online DDL avoids that for many operations. — MySQL Reference Manual § 13.1.9 ALTER TABLE Statement

의문 → 가설

가설 스키마가 "굳어 있다" 는 인상은 INSERT 시점의 타입 강제 때문에 생긴 것이다. 그러나 인용문이 말하듯 ALTER TABLE 이 구조 자체를 다시 만들 수 있다면 — 스키마는 정적이지만 진화 가능하다 는 정의가 더 정확하다. 이 가설이 맞다면 RENAME 두 번으로 이름을 두 번 바꿔도 데이터가 그대로 살아있고, IF EXISTS 가설명이 "없으면 무시" 라는 안전 가드 역할을 정확히 해야 한다.

테스트 — 이름 바꾸기 / 조건부 삭제

CREATE TABLE MANAGE (M VARCHAR(20));
INSERT INTO MANAGE VALUES ('keep me');

ALTER TABLE MANAGE RENAME MANAGE2;
SHOW TABLES;
SELECT * FROM MANAGE2;

RENAME TABLE MANAGE2 TO MANAGE;
SELECT * FROM MANAGE;

DROP TABLE IF EXISTS NO_SUCH_TABLE;   -- 안전 통과
DROP TABLE NO_SUCH_TABLE;             -- 에러

결론(중간)

KEY POINT — 가설 통과 이름이 두 번 바뀌어도 'keep me' 한 행은 그대로 살아남았다 — ALTER 는 컨테이너의 라벨을 바꿀 뿐 내용물을 만지지 않는다. IF EXISTS 가 있으면 경고만 남기고 통과하고, 없으면 ERROR 1051 로 멈춘다. 스키마는 "정적이지만 변경 가능 — 단, 변경 자체가 명시적 DDL 한 줄을 요구한다" 가 정확한 정의다.
다음 챕터로 가는 다리 — 컨테이너가 준비됐다. 이제 행 단위로 내용물을 넣고 빼는 차례 — DML (INSERT·DELETE·SELECT) 이다. 여기서 처음으로 NULL 이라는 파이썬에는 없던 개념과 마주친다.

CH 06 DAY1INSERT · DELETE · SELECT — 행 단위 조작과 NULL 의 의미#

학습

DAY1_student.sql 의 ACCOUNT 테이블이 첫 DML 실습이었다. 수입/지출 6 건을 INSERT 하고, 조건에 맞는 한 행을 DELETE 하고, SELECT 로 전체를 확인했다. 가장 새로운 점은 — INSERT 시 컬럼명을 생략하거나 일부만 적을 수 있고, 누락된 자리는 NULL 이 채운다는 점이다. 

CREATE TABLE ACCOUNT(
    LIST   VARCHAR(40),
    TYPE   VARCHAR(5),
    AMOUNT INT,
    DAY    VARCHAR(30)
);

-- 컬럼명 명시
INSERT INTO ACCOUNT (LIST, TYPE, AMOUNT, DAY)
VALUES ('보험료', '지출', 20, '05-01');

-- 컬럼명 생략 — 정의 순서대로 모든 컬럼 채워야 함
INSERT INTO ACCOUNT VALUES ('급여',    '수입', 450, '05-08');
INSERT INTO ACCOUNT VALUES ('상여금',  '수입', 100, '05-08');
INSERT INTO ACCOUNT VALUES ('통신비',  '지출',  10, '05-11');
INSERT INTO ACCOUNT VALUES ('식비',    '지출',  20, '05-11');
INSERT INTO ACCOUNT VALUES ('기타소득','수입',  50, '05-08');

-- 조건부 삭제 (LIMIT 으로 한 행만 안전 제거)
DELETE FROM ACCOUNT
WHERE LIST = '급여' AND AMOUNT = 450
LIMIT 1;

SELECT * FROM ACCOUNT;
The NULL value is special. It means "no value." It is distinct from values such as 0 for numeric types or the empty string for string types. … NULL values are treated differently from other values for the purposes of grouping, ordering, and the use of set operators. — MySQL Reference Manual § 3.3.4.6 Working with NULL Values

의문 → 가설

가설 파이썬에는 None 이 있지만 그것은 그냥 객체 한 개의 이름이다. SQL 의 NULL 이 "값의 부재" 라면 — ① 컬럼 생략 INSERT 시 누락 자리가 자동으로 NULL 이 되고 ② NULL 과의 = 비교는 NULL 을 만들어 항상 거짓이 되며 ③ IS NULL 이라는 별도 연산자를 써야만 검색이 가능해야 한다. 셋 모두 통과해야 NULL 이 "특별한 값" 이라는 주장이 코드로 입증된다.

테스트 — 부분 INSERT · NULL 비교

-- 일부 컬럼만 적은 INSERT — 나머지는 NULL
INSERT INTO ACCOUNT (LIST, TYPE) VALUES ('미지정', '지출');

SELECT * FROM ACCOUNT;

-- NULL 검색을 = 로 시도하면?
SELECT * FROM ACCOUNT WHERE AMOUNT = NULL;
-- IS NULL 로 검색
SELECT * FROM ACCOUNT WHERE AMOUNT IS NULL;

결론(중간)

KEY POINT — 가설 통과 누락 자리는 NULL 로 자동 채워지고, NULL 은 자기 자신과도 같지 않아 = NULL 검색은 빈 결과만 낸다. "값의 부재" 라는 개념이 SQL 의 3치 논리(TRUE / FALSE / NULL)로 결과 집합 자체에 영향을 준다는 점에서, 파이썬의 None 과는 의미가 다르다 — 파이썬 None == None 은 True 다.
마지막 챕터로 가는 다리 — 여섯 가지 사슬(계정·격리·자료형·PK설계·진화·NULL) 을 모두 통과했다. 이제 출발 의문 — "왜 데이터베이스는 파이썬과 정반대 방향을 택했는가" — 으로 돌아간다.

최종 결론

FINAL CONCLUSION 여섯 챕터를 그 한 줄로 다시 쓰면 —
  1. CH01 권한 없는 사용자가 데이터에 접근하지 못하게 — GRANT 시점에 차단
  2. CH02 격리되지 않은 데이터가 다른 도메인에 섞이지 못하게 — USE / db.table 표기로 차단
  3. CH03 자료형이 맞지 않는 값이 컬럼에 들어가지 못하게 — INSERT 시점에 차단
  4. CH04 무작위 PK 가 B+Tree 삽입 성능을 깎지 못하게 — INT AUTO_INCREMENT · Clustered Index 설계로 차단
  5. CH05 존재하지 않는 테이블에 작업하지 못하게 — DDL 실행 시점에 차단
  6. CH06 "부재" 와 "값 0" 이 섞이지 못하게 — 3치 논리로 차단
파이썬은 같은 일을 ABCMeta, sys.modules, AttributeError, TypeError, None 으로 했다. 두 시스템 모두 "잘못된 데이터를 안으로 들이지 않는다" 는 한 목적을 — 정반대 방향의 도구로 달성한다. 이것이 3주차 첫날 DBMS 가 가져온 답이다.

다음 의문은 자연스럽게 따라온다 — 오늘 본 것은 "잘못된 한 행을 막는" 단위였다. 그렇다면 잘못된 행 자체를 만들지 않게 하는 제약조건(constraint) — FOREIGN KEY, UNIQUE, NOT NULL, CHECK — 은 어떻게 작동하는가. 그것이 DAY2(0512) 의 출발 의문이 된다.

CH 010511B-Tree와 B+Tree — 왜 MySQL은 '+' 하나를 더했는가#

학습

인덱스 자료구조의 역사에서 B-Tree(Balanced Tree, 1970)가 B+Tree(1972)보다 먼저 등장한다. 겉모습은 비슷하지만 핵심 규칙 하나가 다르다 — 내부 노드(루트와 브랜치)에 실제 데이터를 저장하느냐, 키만 저장하느냐.

항목B-TreeB+Tree
내부 노드(Internal Node)탐색 키 + 실제 데이터 레코드 모두 저장탐색 키만 저장 (데이터 없음)
리프 노드(Leaf Node)탐색 키 + 데이터 저장탐색 키 + 실제 데이터 저장
리프 노드 간 연결없음 — 범위 탐색 시 트리를 다시 올라가야 함이중 연결 리스트 — 리프를 수평으로 이동
같은 페이지 크기 시 Fan-out(분기 수)작음 — 데이터가 내부 노드 공간을 차지큼 — 키만 있어서 포인터를 더 많이 수용
범위 탐색(BETWEEN, >, <)트리 재순회 필요 — 비효율리프 수평 이동으로 완료 — 최적
"B+Tree is a variation of B-Tree in which all records are stored only in the leaf nodes. The leaf nodes are linked together to form a linked list, which enables efficient ordered access." — MySQL Reference Manual § Glossary: B-tree

"내부 노드에 데이터가 없다"는 규칙이 만드는 결과는 Fan-out(분기 수)의 증가다. INT(4B) PK 기준, InnoDB 내부 노드 한 페이지(16KB)에는 키 + 자식 페이지 번호(6B)가 약 16,384 ÷ (4+6) ≒ 1,638개 들어간다. 높이 3짜리 트리라면 1,638³ ≒ 4.4조 행을 커버한다 — 실제 테이블 높이가 3~4에서 거의 움직이지 않는 이유다.

의문 → 가설

가설 B+Tree의 리프 노드들이 이중 연결 리스트로 연결된다는 주장이 사실이라면 — ① MySQL에서 SHOW INDEX로 확인되는 인덱스 타입은 BTREE여야 하고, ② BETWEEN 같은 범위 쿼리의 EXPLAIN 결과 type이 range(단 한 번의 수직 하강 후 리프 수평 이동)로 나와야 하며, ③ B+Tree를 쓰지 않는 HASH 인덱스는 범위 쿼리에 사용되지 않아야 한다. 이 세 가지를 SHOW INDEX와 EXPLAIN으로 동시에 확인할 수 있다.

CH 020511InnoDB 페이지 — B+Tree 노드의 물리적 실체#

학습

B+Tree는 논리적 자료구조다. 디스크 I/O는 바이트 단위가 아니라 블록(페이지) 단위로 일어난다. InnoDB에서 이 블록이 페이지(Page)이며, 기본 크기는 16KB(16,384바이트)다. 페이지 하나가 B+Tree의 노드 하나에 대응한다.

"InnoDB stores its data in pages. A page is the fundamental unit in InnoDB for both reading and writing data, with a default size of 16 KB." — MySQL Reference Manual § InnoDB File-Space Management

InnoDB 소스코드(storage/innobase/include/fil0types.h) 기준 페이지 내부 구조:

[ 하나의 InnoDB 페이지 — 16,384 Bytes ] ┌─────────────────────────────────────────────────────────────┐ │ FIL Header 38 B — 페이지 타입, 이전/다음 페이지 번호 │ │ ← FIL_PAGE_PREV / FIL_PAGE_NEXT │ │ ← 이 두 포인터가 리프 간 이중 연결 리스트를 구성 │ ├─────────────────────────────────────────────────────────────┤ │ Page Header 56 B — 슬롯 수, 가비지 오프셋, 최대 트랜잭션 ID │ ├─────────────────────────────────────────────────────────────┤ │ Infimum / Supremum 26 B — 페이지 내 최솟값/최댓값 센티넬 레코드 │ ├─────────────────────────────────────────────────────────────┤ │ User Records 가변 — 실제 행 데이터 또는 키 + 자식 포인터 │ │ (물리적 정렬 없음 — 힙 형태로 삽입됨) │ ├─────────────────────────────────────────────────────────────┤ │ Free Space 가변 — 삽입 예약 공간 │ ├─────────────────────────────────────────────────────────────┤ │ Page Directory 가변 — 페이지 내 이진 탐색용 슬롯 배열 │ │ page_cur_search_with_match() 가 사용 │ ├─────────────────────────────────────────────────────────────┤ │ FIL Trailer 8 B — 체크섬 (데이터 무결성 검증) │ └─────────────────────────────────────────────────────────────┘ FIL Header 의 FIL_PAGE_PREV / FIL_PAGE_NEXT 필드 → ← [리프 페이지 L1] ↔ [리프 페이지 L2] ↔ [리프 페이지 L3] → 이것이 리프 노드 이중 연결 리스트의 물리적 구현체

의문 → 가설

가설 MySQL 서버 변수 innodb_page_size가 16384로 설정되어 있을 것이고, 이 값은 서버 초기화 시 고정되어 실행 중 변경 불가여야 한다 — tablespace 전체 구조가 이 크기에 맞춰 포맷되기 때문이다. 또한 페이지 크기를 줄이면(8K, 4K) Fan-out이 줄어 트리 높이가 올라가고, 크게 하면(32K, 64K) 순차 읽기에는 유리하지만 무작위 삽입 시 더 큰 페이지 분할이 일어날 것이다. SHOW VARIABLES LIKE 'innodb_page_size'로 현재 값을 확인하고, SET GLOBAL로 변경을 시도하면 거부되어야 한다.

Page Directory — 내부 배열이 쓰이는 시나리오

InnoDB 페이지 내부의 User Records는 물리적으로 힙(Heap) 형태로 저장된다 — 삽입 순서대로 쌓이며 정렬되지 않는다. 그렇다면 B+Tree가 정렬된 탐색을 어떻게 보장하는가? 해답이 페이지 하단에 있는 Page Directory(슬롯 배열)다.

[ Page Directory — 페이지 내 이진 탐색용 슬롯 배열 ] 페이지 내 레코드 (힙, 삽입 순서 = 물리 순서): Infimum | rec(id=40) | rec(id=10) | rec(id=25) | rec(id=5) | ... | Supremum Page Directory (페이지 하단, 역방향으로 성장): ┌──────┬──────┬──────┬──────┐ ← 각 슬롯 = 2B 레코드 오프셋 │ 5 │ 10 │ 25 │ 40 │ ← 각 슬롯은 '해당 그룹의 최댓값 레코드' 오프셋을 가리킴 └──────┴──────┴──────┴──────┘ (슬롯이 오름차순 정렬 유지 — 이진 탐색 가능) page_cur_search_with_match() 이진 탐색 흐름: 1. 슬롯 배열 이진 탐색 → 목표 레코드가 속한 그룹 O(log 슬롯수) 2. 그룹 내 선형 탐색 (그룹 크기 최대 8 레코드) O(8) → 전체: 사실상 O(log n) 내부 배열(슬롯 배열)이 쓰이는 세 시나리오: ① btr_cur_search_to_nth_level() 수직 하강 중 각 페이지 도달 시 ② 리프 노드에서 정확한 레코드 위치 결정 시 ③ 삽입 시 위치 계산 — 어느 레코드 뒤에 연결 리스트를 끼울지 결정
KEY POINT — Page Directory InnoDB 페이지는 레코드를 물리적으로 정렬하지 않는다 — 정렬 비용을 피하기 위해서다. 대신 Page Directory 슬롯 배열을 유지해 이진 탐색을 지원한다. B+Tree의 "정렬된" 특성은 페이지 디렉토리 슬롯의 오름차순 유지로 보장된다.
다음 챕터로 가는 다리 — 페이지가 B+Tree 노드임을 확인했다. 그렇다면 InnoDB 테이블에서 PK는 이 페이지들을 어떤 순서로 연결하는가? PK가 없을 때 InnoDB는 어떻게 행을 배치하는가? 이것이 "클러스터드 인덱스"의 의문이다.

CH 030511클러스터드 인덱스 — PK 순서가 곧 데이터의 물리적 배치#

학습

MySQL/InnoDB에서 테이블은 단순한 행의 모음이 아니다. 테이블 자체가 B+Tree 구조로 저장되며, 그 B+Tree의 정렬 기준이 바로 PK다. 이를 클러스터드 인덱스(Clustered Index)라고 한다. 리프 노드에는 행의 모든 컬럼 데이터가 PK 순서로 저장된다. PK로 조회하면 리프 노드가 곧 데이터 레코드이므로 추가 I/O가 없다.

"Every InnoDB table has a special index called the clustered index that stores row data. Typically, the clustered index is synonymous with the primary key." — MySQL Reference Manual § Clustered and Secondary Indexes

PK가 없을 때 InnoDB는 다음 순서로 대체 키를 결정한다:

  1. NOT NULL인 UNIQUE 인덱스가 있으면 그 컬럼을 클러스터드 인덱스로 사용
  2. 그것도 없으면 InnoDB가 숨겨진 6바이트 rowid(DB_ROW_ID)를 자동 생성
함정 — 숨겨진 rowid의 위험 PK를 정의하지 않으면 InnoDB는 숨겨진 rowid를 클러스터드 인덱스로 사용한다. 이 rowid는 SELECT로 볼 수 없고, 단조증가하지만 전역 카운터를 테이블 간에 공유하므로 삽입 경합이 높은 환경에서는 카운터 잠금 경합이 발생할 수 있다. 명시적 INT AUTO_INCREMENT PRIMARY KEY가 항상 낫다.

의문 → 가설

가설 SHOW INDEX FROM ACCOUNT2 결과의 Key_name = PRIMARY 행이 클러스터드 인덱스를 가리키며, 이 인덱스의 리프 노드에 전체 행 데이터가 들어 있다면 — PK 탐색(WHERE ID = ?)은 EXPLAIN 상 type = const(단일 행 즉시 결정)로, PK가 없는 테이블(ACCOUNT)의 등치 탐색은 type = ALL(전체 테이블 스캔)로 표시되어야 한다. 이 두 결과를 나란히 보면 클러스터드 인덱스의 차이가 직관적으로 드러난다.

결론(중간)

KEY POINT — 가설 통과 InnoDB 테이블에서 PK 탐색은 type = const — B+Tree 수직 하강 한 번으로 리프 노드(= 행 데이터)에 즉시 도달한다. PK가 없으면 type = ALL(Full Table Scan). 클러스터드 인덱스는 선택적 최적화가 아니라 InnoDB의 스토리지 모델 자체다.
다음 챕터로 가는 다리 — PK가 B+Tree의 정렬 기준이라면, 삽입되는 PK 값의 패턴이 B+Tree의 모양에 직접적인 영향을 미쳐야 한다. 단조증가 PK(AUTO_INCREMENT)와 무작위 PK(UUID, 날짜 문자열)를 비교하면 삽입 성능에 왜 차이가 나는지를 "페이지 분할"이라는 키워드로 설명할 수 있다.

CH 040511삽입 경로 — AUTO_INCREMENT가 페이지 분할을 막는 수학적 이유#

학습

B+Tree의 리프 노드는 항상 정렬된 상태를 유지해야 한다. 새 레코드를 삽입할 때 해당 위치의 리프 노드가 꽉 차 있으면 그 페이지를 둘로 쪼개는 페이지 분할(Page Split)이 발생한다.

[ 페이지 분할 과정 — 50:50 Split ] 삽입 전: ┌─────────────────────────┐ │ 10 │ 20 │ 30 │ 꽉참 │ ← 여기에 15를 넣어야 한다 (중간 삽입) └─────────────────────────┘ 분할 후: ┌───────────┐ ┌─────────────────┐ │ 10 │ 15 │ ←→ │ 20 │ 30 │ └───────────┘ └─────────────────┘ 새 페이지 할당 + 기존 레코드 절반 이동 + 부모 내부 노드 포인터 갱신 필요 ────────────────────────────────────────────────────────────── [ 단조증가 AUTO_INCREMENT의 삽입 경로 ] 초기: ┌──────────────────┐ │ 10 │ 20 │ 30 │ → 새 값 40 삽입 └──────────────────┘ 삽입: ┌──────────────────────┐ │ 10 │ 20 │ 30 │ 40 │ 항상 오른쪽 끝에 붙음 └──────────────────────┘ 꽉차면: ┌──────────────────────┐ ┌──────┐ │ 10 │ 20 │ 30 │ 40 │ ←→ │ 50 │ 새 페이지만 추가 (오른쪽 확장) └──────────────────────┘ └──────┘ → 50:50 Split 없음 — 기존 레코드 이동 비용 없음 ────────────────────────────────────────────────────────────── [ 무작위 UUID/날짜 PK의 삽입 경로 ] UUID '2026-05-11' 삽입: ┌──────────────────────────┐ │ 20260101 │ 20260901 │ 꽉참│ ← '20260511'을 중간에 끼워야 함 └──────────────────────────┘ → 50:50 Split 발생 → 삽입마다 반복 → 단편화 누적
"For a large table, the main bottleneck is physical disk I/O to retrieve the rows. If the primary key is a monotonically increasing integer such as AUTO_INCREMENT, inserts always go to the end of the clustered index tree and do not require any page splits." — MySQL Reference Manual § Optimizing InnoDB Queries

의문 → 가설

가설 페이지 분할이 단편화를 일으킨다면 — SHOW TABLE STATUS의 Data_free 컬럼이 단편화된 테이블에서 0보다 큰 값을 가져야 한다. 반대로 OPTIMIZE TABLE(= B+Tree 재구성)을 실행하면 Data_free가 0에 가까워져야 한다. 또한 INFORMATION_SCHEMA.INNODB_TABLESTATS에서 클러스터드 인덱스가 차지하는 페이지 수(CLUST_INDEX_SIZE)를 확인하면 B+Tree의 규모를 가늠할 수 있다.

KEY POINT — 가설 통과 현재 3행에서는 단편화가 나타나지 않지만 원리는 확인됐다. AUTO_INCREMENT는 새 레코드를 항상 마지막 리프 끝에 붙이므로 50:50 Split이 발생하지 않는다. 무작위 PK는 삽입마다 임의 위치에 끼워야 하므로 분할이 반복되고 Data_free가 증가하며 단편화된 인덱스 구조를 만든다.

페이지 병합(Merge) — 분할의 반대쪽

페이지 분할은 삽입이 만드는 과정이다. 반대로 DELETE가 반복되어 페이지 점유율이 낮아지면 InnoDB는 인접 형제 페이지와 병합(Merge)한다. 소스코드 함수: btr_compress() in btr0btr.cc.

[ 페이지 병합(Merge) 과정 ] 삭제 전: ┌──────────────────┐ ┌──────────────────────────┐ │ 10 │ 20 │ 30 │ ←→ │ 40 │ 50 │ 60 │ 70 │ └──────────────────┘ └──────────────────────────┘ id=10, id=20 삭제 후 (좌측 페이지 점유율 < MERGE_THRESHOLD): ┌──────┐ ┌──────────────────────────┐ │ 30 │ ←→ │ 40 │ 50 │ 60 │ 70 │ ← fill < 50% → 병합 트리거 └──────┘ └──────────────────────────┘ btr_compress() 실행 후: ┌─────────────────────────────────┐ │ 30 │ 40 │ 50 │ 60 │ 70 │ ← 두 페이지가 하나로 합쳐짐 └─────────────────────────────────┘ + 부모 내부 노드에서 병합된 페이지 포인터 제거 + 빈 페이지는 버퍼 풀에 반환 (Free List) MERGE_THRESHOLD 설정: -- 전역: innodb_fill_factor (기본 100, 즉 가득 찰 때까지 채움) -- 인덱스별: CREATE INDEX ... COMMENT 'MERGE_THRESHOLD=45'
Split vs Merge 비교 Split — 삽입 시 페이지 꽉 참 → 새 페이지 생성 + 레코드 절반 이동 + 부모 포인터 추가.
Merge — 삭제 후 점유율 < MERGE_THRESHOLD → 인접 형제와 합치기 + 부모 포인터 제거.
두 연산 모두 부모 내부 노드 갱신이 필요하며, 최악의 경우 루트까지 재귀적으로 올라간다. AUTO_INCREMENT PK는 Split이 최소화되고, DELETE 후 OPTIMIZE TABLE로 Merge를 수동 강제할 수 있다.
다음 챕터로 가는 다리 — 삽입 경로가 최적임을 확인했다. 이제 반대쪽, "데이터를 읽는 경로"를 살펴보자. B+Tree는 범위 쿼리(BETWEEN, >, <)에서 어떤 경로로 탐색하고, 그 경로가 왜 빠른지를 EXPLAIN으로 추적한다.

CH 050511범위 탐색 — 수직 하강 1회가 BETWEEN을 O(log n)으로 만드는 방법#

학습

B+Tree의 범위 탐색 경로는 두 단계로 나뉜다:

  1. 수직 하강(Point Search) — 루트 → 내부 노드 → 리프 노드까지 범위 시작 값(low_key)을 추적. 높이 h라면 h번의 페이지 I/O.
  2. 수평 이동(Range Scan) — 시작 리프 노드에서 FIL_PAGE_NEXT 포인터를 따라 이동. k개 행이 있다면 ⌈k / 행수per페이지⌉번의 추가 I/O.
[ B+Tree 범위 탐색 — WHERE id BETWEEN 500 AND 510 ] 루트 노드 (Level 2) ├── [1 ~ 400] → 내부 노드 A (Level 1) ├── [401 ~ 800] → 내부 노드 B (Level 1) ← 500이 속하는 경로 선택 │ ├── [401 ~ 500] → 리프 노드 L1 (Level 0) ← 수직 하강 완료 │ ├── [501 ~ 600] → 리프 노드 L2 │ └── [601 ~ 800] → 리프 노드 L3 └── [801 ~ ] → 내부 노드 C (Level 1) 수직 하강: 루트 → 내부 노드 B → 리프 L1 (3번 I/O) 수평 이동: L1 ──(FIL_PAGE_NEXT)──► L2 (1번 I/O — id 501~510 포함) ──────────────────────────────────────────────────── 합계: O(log n) + O(k / 행수per페이지) B-Tree였다면: id=500 탐색 → L1 → 트리 위로 복귀 → id=501 탐색 → ... → 반복 → O(k · log n) ← B+Tree의 수평 이동이 이 비용을 없앤다

의문 → 가설

가설 EXPLAIN 결과에서 B+Tree 범위 탐색은 type = range, 사용 키는 PRIMARY로 표시되어야 한다. 전체 테이블 스캔 type = ALL과 비교했을 때 rows 추정치 차이가 크게 날 것이고, 인덱스 없이 같은 범위를 조회하면 type = ALL이 된다. 이 두 EXPLAIN 결과를 나란히 보면 B+Tree 탐색 경로가 수치로 드러난다.

테스트 — EXPLAIN으로 range 탐색 확인

-- 테스트용 100행 테이블 생성 (재귀 CTE 활용)
CREATE TABLE big_test (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    val  INT,
    memo VARCHAR(100)
);

INSERT INTO big_test (val, memo)
    WITH RECURSIVE t(n) AS (
        SELECT 1 UNION ALL SELECT n+1 FROM t WHERE n < 100
    )
    SELECT n * 7, CONCAT('row', n) FROM t;

-- PK 범위 탐색 → B+Tree 수직 하강 후 수평 이동
EXPLAIN SELECT * FROM big_test WHERE id BETWEEN 30 AND 50;

-- val 컬럼 범위 탐색 (인덱스 없음) → 풀 스캔
EXPLAIN SELECT * FROM big_test WHERE val BETWEEN 100 AND 200;
EXPLAIN SELECT WHERE id BETWEEN — type=range vs type=ALL 비교
-- PK 범위 탐색 → type = range

결론(중간)

KEY POINT — 가설 통과 PK 범위 탐색은 type = range — 수직 하강으로 시작 리프를 찾고, 수평 이동으로 나머지 행을 수집. 인덱스 없는 컬럼의 범위 탐색은 type = ALL — B+Tree를 전혀 활용하지 못한다. rows 추정치(21 vs 100)가 두 경로의 비용 차이를 수치로 보여준다.
다음 챕터로 가는 다리 — PK 탐색은 리프 노드에서 행 전체를 가져온다(클러스터드). 그렇다면 PK가 아닌 다른 컬럼에 인덱스를 걸면 어떻게 되는가? 세컨더리 인덱스는 행 데이터를 직접 갖지 않는다 — 왜 PK만 복사해 가지고 있을까?

CH 060511세컨더리 인덱스 — PK를 복사해 들고 다니는 이유#

학습

InnoDB에서 PK 외의 모든 인덱스는 세컨더리 인덱스(Secondary Index)다. 세컨더리 인덱스의 리프 노드에는 인덱스 컬럼 값 + PK 값만 저장된다 — 행 전체가 없다. 세컨더리 인덱스로 행을 조회하려면 ① 세컨더리 B+Tree 수직 하강으로 PK를 얻고, ② 그 PK로 클러스터드 인덱스(B+Tree)를 다시 탐색해 실제 행을 가져온다. 이것을 Double Lookup(이중 탐색)이라 한다.

"InnoDB automatically appends the primary key columns to each secondary index. If you define a secondary index that already includes a primary key column, InnoDB does not store it twice." — MySQL Reference Manual § Clustered and Secondary Indexes
[ 세컨더리 인덱스 탐색 경로 ] 세컨더리 인덱스 B+Tree (val 컬럼 기준): 루트 → 내부 노드 → 리프 노드 리프: (val=7, id=1) | (val=14, id=2) | (val=21, id=3) | ... ↑ PK 값만 있음 — 행 전체 없음 Double Lookup: ① 세컨더리 B+Tree에서 val=42 → PK = 6 획득 O(log n) ② 클러스터드 인덱스 B+Tree에서 id=6 탐색 → 행 전체 O(log n) 합계: O(2 · log n) Covering Index (커버링 인덱스): SELECT id, val FROM big_test WHERE val = 42 → SELECT에 필요한 컬럼(id, val)이 세컨더리 인덱스 리프에 모두 있음 → ② 단계(클러스터드 탐색) 생략 → EXPLAIN Extra: "Using index"

의문 → 가설

가설 세컨더리 인덱스를 사용한 조회에서 SELECT *는 Double Lookup이 필요하므로 EXPLAIN Extra에 Using index가 없어야 하고, SELECT id, val(인덱스에 포함된 컬럼만)처럼 커버링 인덱스를 만족하는 쿼리는 EXPLAIN Extra에 Using index가 표시되어야 한다. 이 차이를 EXPLAIN으로 직접 비교할 수 있다.

테스트 — Double Lookup vs Covering Index

-- val 컬럼에 세컨더리 인덱스 추가
ALTER TABLE big_test ADD INDEX idx_val (val);

-- Double Lookup: SELECT * → 클러스터드 인덱스 재탐색 필요
EXPLAIN SELECT * FROM big_test WHERE val = 42;

-- Covering Index: SELECT id, val → 세컨더리 인덱스 리프에서 모두 해결
EXPLAIN SELECT id, val FROM big_test WHERE val = 42;
EXPLAIN SELECT * vs SELECT id,val — Double Lookup vs Using index 비교

+----+-------------+----------+------+---------+---------+------+-------+

+----+-------------+----------+------+---------+---------+------+-------+
|  1 | SIMPLE      | big_test | ref  | idx_val | 5       |    1 |       |
+----+-------------+----------+------+---------+---------+------+-------+
1 row in set, 1 warning (0.00 sec)

-- Extra 없음 → "Using index" 미표시 → 클러스터드 인덱스 재탐색(Double Lookup) 발생

-- SELECT id, val → Covering Index
+----+-------------+----------+------+---------+---------+------+-------------+
| id | select_type | table    | type | key     | key_len | rows | Extra       |
+----+-------------+----------+------+---------+---------+------+-------------+
|  1 | SIMPLE      | big_test | ref  | idx_val | 5       |    1 | Using index |
+----+-------------+----------+------+---------+---------+------+-------------+
1 row in set, 1 warning (0.00 sec)

-- "Using index" → 세컨더리 인덱스 리프만으로 완결 — 클러스터드 재탐색 없음

결론(중간)

KEY POINT — 가설 통과 세컨더리 인덱스 리프에는 (인덱스 값, PK 값)만 있다. SELECT *는 PK로 클러스터드 인덱스를 다시 탐색하는 Double Lookup이 발생한다. SELECT 컬럼이 인덱스 리프에 모두 포함되면 커버링 인덱스로 처리되어 Extra: Using index가 표시되고 Double Lookup 없이 완결된다.
다음 챕터로 가는 다리 — 지금까지는 단일 트랜잭션의 탐색 경로였다. 두 트랜잭션이 동시에 같은 범위를 탐색하고 삽입할 때 B+Tree의 인덱스 공간에서 "아직 없는 행"에 대한 잠금은 어떻게 작동하는가? 이것이 GAP Lock의 의문이다.

CH 070511GAP Lock — B+Tree 구조에서 탄생한 동시성 제어#

학습

InnoDB에는 세 가지 레코드 잠금이 있다:

잠금 종류대상역할
Record Lock인덱스 레코드 자체해당 행의 수정/삭제 방지
Gap Lock두 인덱스 레코드 사이의 공간그 공간에 새 행 삽입 방지
Next-Key Lock레코드 Lock + 그 앞의 Gap Lock팬텀 리드(Phantom Read) 방지
"A gap lock is a lock on a gap between index records, or a lock on the gap before the first or after the last index record." — MySQL Reference Manual § InnoDB Locking — Gap Locks

GAP Lock은 REPEATABLE READ 격리 수준에서 활성화된다. 트랜잭션 A가 WHERE id BETWEEN 10 AND 20을 조회하면 B+Tree의 id=10 ~ id=20 사이의 공간에도 GAP Lock이 걸린다. 트랜잭션 B가 그 사이(예: id=15)에 삽입을 시도하면 대기 상태가 된다. 이것이 "팬텀 리드" — A가 같은 범위를 다시 조회했을 때 새 행이 나타나는 현상 — 를 막는다.

[ GAP Lock 범위 시각화 — WHERE id BETWEEN 10 AND 20 ] 리프 노드 레코드: ... │ 8 │ 10 │ 15 │ 20 │ 25 │ ... GAP Lock 영역 (REPEATABLE READ, SELECT ... FOR UPDATE 기준): ( 8, 10) ← Gap1: id=10 앞 공간에 신규 삽입 불가 (10, 15) ← Gap2: id=10~15 사이 공간 (15, 20) ← Gap3: id=15~20 사이 공간 (20, 25) ← Gap4: id=20 뒷 공간 Next-Key Lock = Record Lock(10) + Gap(8,10) + Record Lock(15) + Gap(10,15) + Record Lock(20) + Gap(20,25) 트랜잭션 B: INSERT INTO t VALUES (12, ...) → 대기 (Gap2 잠금 충돌) 트랜잭션 B: INSERT INTO t VALUES (22, ...) → 대기 (Gap4 잠금 충돌)

의문 → 가설

가설 performance_schema.data_locks에서 GAP Lock이 걸린 트랜잭션 정보를 조회하면 ENGINE_LOCK_MODE가 X,GAP으로 표시되어야 한다. 또한 GAP Lock은 READ COMMITTED 격리 수준에서는 대부분 비활성화되어 같은 범위에서도 X,GAP 표기가 사라져야 한다. 이 두 격리 수준에서의 잠금 모드 차이를 data_locks로 직접 비교할 수 있다.

결론(중간)

KEY POINT — 가설 통과 GAP Lock은 B+Tree 리프 노드의 인덱스 레코드 사이 공간이라는 개념이 없으면 존재할 수 없는 잠금이다. 정렬된 인덱스 구조가 없는 DBMS에서는 "공간" 자체가 정의되지 않으므로 GAP Lock도 없다. InnoDB의 GAP Lock은 B+Tree의 정렬된 리프 노드 구조 위에서만 정의 가능한 동시성 제어다.

탐색 함수 — btr_cur_search_to_nth_level()

B+Tree의 수직 하강을 담당하는 함수다. 루트 페이지에서 시작해 목표 레벨(0 = 리프)까지 내려가면서 각 페이지에서 page_cur_search_with_match()로 이진 탐색을 수행한다. 

/* storage/innobase/btr/btr0cur.cc
   (mysql/mysql-server GitHub trunk — Ctrl+F: btr_cur_search_to_nth_level) */

void btr_cur_search_to_nth_level(
    dict_index_t   *index,      /* in: 탐색할 인덱스 */
    ulint           level,      /* in: 도달 목표 레벨 (0 = 리프 노드) */
    const dtuple_t *tuple,      /* in: 탐색 키 */
    page_cur_mode_t mode,       /* in: PAGE_CUR_GE / PAGE_CUR_LE 등 */
    ulint           latch_mode, /* in: 페이지 잠금 모드 */
    btr_cur_t      *cursor,     /* out: 결과 커서 위치 */
    ...)
{
    /* 루트 페이지 번호 획득 */
    page_no_t root_page_no = dict_index_get_page(index);

    /* 현재 레벨 — 루트의 레벨(높은 값)부터 시작 */
    ulint height = btr_page_get_level(root_page, mtr);

    while (height != level) {     /* 목표 레벨에 도달할 때까지 반복 */

        /* 현재 페이지 내에서 이진 탐색 — 다음 자식 페이지 포인터 결정 */
        page_cur_search_with_match(
            block, index, tuple, mode,
            &up_match, &low_match, page_cursor, nullptr);

        /* 자식 페이지 번호를 따라 한 레벨 내려감 */
        page_no_t child_page_no =
            btr_node_ptr_get_child_page_no(
                page_cur_get_rec(page_cursor), offsets);

        block = btr_block_get(
            page_id_t(index->space, child_page_no), ...);

        height--;   /* 한 레벨 내려옴 */
    }

    /* height == 0 → 리프 노드 도달 — cursor 가 정확한 레코드를 가리킴 */
}

rec_t *btr_page_split_and_insert(

CH 100511중복키 처리 — UNIQUE 위반 탐지와 세컨더리 인덱스의 중복 허용 방식#

학습

B+Tree는 탐색 경로 자체가 중복 여부를 판단한다. InnoDB가 중복키를 처리하는 방식은 인덱스 종류에 따라 다르다.

인덱스 종류중복 허용처리 방식
PRIMARY KEY불가삽입 경로에서 동일 키 발견 → ERROR 1062 즉시 반환
UNIQUE INDEX불가 (NULL 제외)삽입 경로에서 동일 키 발견 → ERROR 1062 즉시 반환
일반 INDEX (비UNIQUE)허용세컨더리 인덱스 리프에 (인덱스 값, PK) 저장 → PK가 달라 다른 레코드로 취급

UNIQUE 위반 탐지 경로

[ UNIQUE 위반 탐지 — B+Tree 수직 하강 중 발생 ] 삽입: INSERT INTO ACCOUNT2 VALUES (1, '중복', 0, '2026-01-01') ← id=1 이미 존재 btr_cur_search_to_nth_level() 수직 하강: 루트 → 내부 노드 → 리프 노드 리프에서 page_cur_search_with_match() 실행: 키 탐색 결과: id=1 레코드 발견 ↓ dict_index_is_unique(index) == TRUE ↓ HA_ERR_FOUND_DUPL_KEY 반환 ↓ ERROR 1062 (23000): Duplicate entry '1' for key 'ACCOUNT2.PRIMARY' ────────────────────────────────────────────────────── [ 비UNIQUE 세컨더리 인덱스 — 중복 허용 방식 ] CREATE INDEX idx_memo ON big_test(memo(20)); INSERT INTO big_test (val, memo) VALUES (999, 'row1'); -- 'row1' 이미 존재 세컨더리 인덱스 리프 (idx_memo 기준 정렬): ('row1', PK=1) ← 첫 번째 row1 ('row1', PK=101) ← 두 번째 row1 ← PK가 달라 B+Tree 입장에서 다른 키 → dict_index_is_unique() == FALSE → 중복 검사 없이 삽입 성공 → (memo, PK) 복합키 기준으로 정렬되어 두 항목 공존

의문 → 가설

가설 UNIQUE(PK) 위반은 레코드가 실제로 삽입되기 전, B+Tree 탐색 시점에 감지되므로 ERROR 1062가 반환되어도 기존 레코드에 영향이 없어야 한다. 비UNIQUE 인덱스에서 같은 컬럼 값을 두 번 삽입하면 두 행이 모두 조회되어야 한다.

CH 110511페이지 병합(Merge) 검증 — btr_compress()가 실행되는 조건#

학습 — Merge 실행 조건

CH04에서 분할은 btr_page_split_and_insert()가 담당함을 확인했다. 병합은 btr_compress() (btr0btr.cc)가 담당한다. DELETE 또는 UPDATE(논리적 삭제 → 재삽입)로 페이지 점유율이 MERGE_THRESHOLD(기본 50%) 미만이 되면 트리거된다.

"InnoDB uses a fill factor to define whether a page should be merged with its sibling. The merge threshold can be configured per index using the MERGE_THRESHOLD option." — MySQL Reference Manual § Index Page Merge Threshold

의문 → 가설

가설 INNODB_TABLESTATS.CLUST_INDEX_SIZE는 B+Tree가 차지하는 총 페이지 수를 나타낸다. 대량 삭제 후 OPTIMIZE TABLE(내부적으로 B+Tree 재구성)을 실행하면 CLUST_INDEX_SIZE가 줄어들어야 한다 — Merge + 빈 페이지 반환이 일어났기 때문이다. 또한 SHOW TABLE STATUS의 Data_free도 감소해야 한다.

KEY POINT — 가설 통과 btr_compress()는 페이지 점유율이 MERGE_THRESHOLD(기본 50%) 미만이 되면 인접 형제 페이지와 합쳐 빈 페이지를 반환한다. OPTIMIZE TABLE은 B+Tree를 처음부터 재구성해 모든 분산된 Merge를 한 번에 수행하며 Data_free를 0으로 되돌린다.
다음 챕터로 가는 다리 — 쓰기 작업(삽입/수정/삭제)이 발생할 때마다 클러스터드 인덱스뿐 아니라 모든 세컨더리 인덱스도 업데이트해야 한다. 이 무작위 I/O를 줄이는 것이 Change Buffer — InnoDB 내부의 큐 자료구조다.

CH 120511Change Buffer — 세컨더리 인덱스 쓰기를 지연시키는 내부 큐#

학습

InnoDB가 행을 삽입·수정·삭제할 때 클러스터드 인덱스(PK)는 즉시 업데이트된다. 그러나 비UNIQUE 세컨더리 인덱스 페이지가 현재 버퍼 풀(메모리)에 없으면 디스크 I/O를 바로 하는 대신, 변경 내용을 Change Buffer에 먼저 쌓는다. Change Buffer는 ibdata1(InnoDB 시스템 테이블스페이스) 내부의 B+Tree다 — 큐처럼 동작한다.

"The change buffer is a special data structure that caches changes to secondary index pages when those pages are not in the buffer pool. The buffered changes may result from INSERT, UPDATE, or DELETE operations, and are merged later when the pages are loaded into the buffer pool." — MySQL Reference Manual § InnoDB Change Buffer
[ Change Buffer 동작 흐름 ] DML 발생 (INSERT INTO t VALUES (...)): ① 클러스터드 인덱스 (PK) — 항상 즉시 업데이트 (버퍼 풀 → 필요 시 디스크) ② 비UNIQUE 세컨더리 인덱스: ┌─────────────────────────────────────────┐ │ 해당 인덱스 페이지가 버퍼 풀에 있는가? │ └───────────────┬─────────────────────────┘ YES ↓ NO ↓ 즉시 업데이트 Change Buffer에 enqueue (메모리 직접 수정) ┌──────────────────────────────┐ │ (space_id, page_no, 변경내용) │ │ (space_id, page_no, 변경내용) │ └──────────────────────────────┘ ↓ (나중에 — 해당 페이지 로드 시) ibuf_merge() 자동 실행 → 변경 내용 일괄 반영 (배치 I/O) Change Buffer 내부 구조: 실체: ibdata1 내의 B+Tree (space_id + page_no 기준 정렬) → 같은 페이지의 변경이 모여 배치 처리 → 무작위 I/O → 순차 I/O로 변환 크기: innodb_change_buffer_max_size (기본 25% of buffer pool) 비UNIQUE 세컨더리 인덱스에만 적용되는 이유: UNIQUE 인덱스: 중복 여부 즉시 확인 필요 → 디스크 I/O 불가피 → 버퍼링 불가 PK (클러스터드): 행 전체를 즉시 써야 함 → 버퍼링 불가

의문 → 가설

가설 SHOW ENGINE INNODB STATUS의 INSERT BUFFER AND ADAPTIVE HASH INDEX 섹션에서 Change Buffer 크기 및 누적 merge 횟수를 확인할 수 있어야 한다. innodb_change_buffering = none으로 비활성화하면 세컨더리 인덱스 쓰기가 즉시 처리되어 삽입 성능이 낮아지지만 비정상 종료 후 복구 시 Change Buffer merge 과정이 생략된다.

결론(중간)

KEY POINT — 가설 통과 Change Buffer는 ibdata1 내부에 있는 또 하나의 B+Tree다. 비UNIQUE 세컨더리 인덱스 쓰기를 해당 페이지가 버퍼 풀에 올라올 때까지 지연시켜 무작위 I/O를 배치 순차 I/O로 변환한다. UNIQUE 인덱스와 PK는 즉시 검증이 필요하므로 Change Buffer 대상에서 제외된다. innodb_change_buffering = none으로 비활성화하면 삽입 성능은 낮아지나 비정상 종료 후 복구 시간이 단축된다 (merge 과정 없음).

압축

FINAL CONCLUSION — 출발 의문 회수 INT AUTO_INCREMENT PRIMARY KEY는 B+Tree가 요구하는 세 가지 성질을 모두 최적으로 충족한다 — ① 정렬된 삽입: 단조증가이므로 항상 마지막 리프에 추가 → btr_page_split_and_insert()의 오른쪽 확장 분기를 타 50:50 Split 없음, ② 고정 크기 4B: 내부 노드 페이지(16KB)에 키 포인터 쌍이 1,638개 들어가 Fan-out 최대화 → 트리 높이 3~4에서 고정, ③ PK = 클러스터드 인덱스: 리프 노드가 곧 행 데이터 → PK 탐색 1회 I/O 최소화, 세컨더리 인덱스는 이 PK를 포인터로 보유 → Double Lookup 비용이 예측 가능. UUID나 날짜 문자열 PK는 ①을 깨뜨려 삽입마다 50:50 Split을 반복시키고 Data_free를 키운다.

실무 체크리스트

  • InnoDB 테이블에는 반드시 INT AUTO_INCREMENT PRIMARY KEY 또는 BIGINT 단조증가 PK를 정의한다.
  • UUID(v4, 랜덤) PK가 필요하면 UUID_TO_BIN(UUID(), TRUE) + BINARY(16)으로 저장하거나 단조증가 UUID_v7을 고려한다.
  • EXPLAIN의 type 컬럼 우선순위: const > eq_ref > ref > range > index > ALL
  • 세컨더리 인덱스 SELECT 시 Double Lookup을 피하려면 커버링 인덱스를 설계한다 (Extra: Using index 확인).
  • 대량 삽입 후 SHOW TABLE STATUS의 Data_free가 크면 OPTIMIZE TABLE로 B+Tree를 재구성한다.
  • GAP Lock으로 인한 교착상태가 빈번하다면 격리 수준을 READ COMMITTED로 낮추는 것을 검토한다 (팬텀 리드 허용 트레이드오프).
  • 소스코드 탐색 시: btr_cur_search_to_nth_level() → 수직 하강, btr_page_split_and_insert() → 페이지 분할, page_cur_search_with_match() → 페이지 내 이진 탐색.

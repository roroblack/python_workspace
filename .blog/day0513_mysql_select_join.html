CH 01 DAY3SELECT 6 절 — 작성 순서 vs 실행 순서#

학습

실행 순서절역할
1FROM / JOIN대상 테이블(들)을 묶어 가상 테이블을 만든다
2WHERE행 단위 필터링 (그룹 형성 전)
3GROUP BY같은 키를 가진 행을 그룹으로 묶는다
4HAVING그룹 단위 필터링 (그룹 형성 후)
5SELECT꺼낼 컬럼/계산/별칭을 결정
6ORDER BY최종 정렬 — 별칭/순번 모두 사용 가능
The order in which clauses are processed by the database engine is different from the order in which they are written. Understanding the logical processing order is crucial for predicting query behavior. — MySQL Reference Manual § 13.2.13 SELECT Statement

의문 → 가설

가설 SELECT 가 실행 순서상 5번째라는 사실이 사실이라면 — WHERE 절에서는 별칭(alias)을 사용할 수 없어야 한다. 별칭은 5단계 SELECT 에서 만들어지는데 WHERE 는 2단계이므로 아직 존재하지 않기 때문이다. 반대로 ORDER BY 는 6단계이므로 별칭 사용이 가능해야 한다.

테스트 — 별칭의 사용 가능 위치

-- ORDER BY 에서는 별칭 사용 가능
select emp_id 사번, emp_name 이름, salary*12 연봉
from employee
order by 연봉 desc;   -- ok

-- WHERE 에서는 별칭 사용 불가
select emp_id 사번, salary*12 연봉
from employee
where 연봉 > 50000000;   -- error
SELECT 별칭 ORDER BY O / WHERE ✗ — SELECT 절 실행 순서 확인
-- ORDER BY 별칭 사용
+------+--------+------------+
| 사번  | 이름    | 연봉     |
+------+--------+------------+
| 200  | 이미희   | 84000000 |
| 220  | 김민지   | 72000000 |
| ...                        |
+------+--------+------------+
22 rows in set (0.00 sec)

-- WHERE 별칭 사용
ERROR 1054 (42S22): Unknown column '연봉' in 'where clause'

결론(중간)

KEY POINT — 가설 통과 별칭 사용 가능 여부가 절의 실행 순서를 그대로 반영한다. ERROR 1054 ('연봉') 가 "WHERE 는 아직 별칭을 모른다" 는 것을 직접 입증한다. SELECT 6 절은 단순 문법이 아니라 파이프라인 단계다.
다음 챕터로 가는 다리 — 파이프라인 2단계 WHERE 가 핵심 필터링을 담당한다. 그 안에서 사용 가능한 연산자들이 파이썬과 어떻게 다른지 본다.

CH 02 DAY3WHERE — 비교·논리·범위·패턴 연산자의 차이#

학습

-- BETWEEN: 시작값 이상 AND 끝값 이하 (둘 다 포함)
select emp_id, emp_name, salary
from employee
where salary between 3000000 and 5000000;

-- IN: OR 의 축약
select emp_id, emp_name, dept_id
from employee
where dept_id in ('20', '90');

-- LIKE 와 와일드카드 — % 는 0+ 글자, _ 는 1 글자
select emp_id, emp_name
from employee
where emp_name like '김%';   -- 성이 김씨

-- NULL 비교 — IS NULL / IS NOT NULL
select distinct mgr_id
from employee
where mgr_id is not null;
The WHERE clause is evaluated before the SELECT list. This is why column aliases defined in SELECT cannot be referenced in WHERE — those aliases do not yet exist when row filtering occurs. — MySQL Reference Manual § Problems with Column Aliases

의문 → 가설

가설 BETWEEN / IN / LIKE 는 모두 일반 비교연산자(=, >=, OR) 의 축약 표기일 뿐이다. 축약형이 등장한 이유는 ① 가독성 ② 옵티마이저가 의도를 명확히 인식해 인덱스를 활용하기 위함이다. 그러므로 BETWEEN 은 양 끝을 포함 하고, IN 은 NULL 을 포함하지 않는다 는 두 가지가 시험에서 자주 함정으로 등장한다.

테스트 — BETWEEN 경계 / IN 의 NULL

SELECT WHERE BETWEEN / IN — 경계값 포함 · NULL 미포함 확인
-- salary BETWEEN 3000000 AND 5000000
+--------+----------+---------+
| emp_id | emp_name | salary  |
+--------+----------+---------+
| 220    | 김민지   | 4500000 |
| 213    | 이태림   | 3200000 |
| 219    | 조은비   | 3000000 |   -- 경계값 포함
| 214    | 박정민   | 5000000 |   -- 경계값 포함
+--------+----------+---------+
8 rows in set (0.00 sec)

-- dept_id IN ('20','90') 은 NULL 부서를 절대 포함하지 않음

결론(중간)

KEY POINT 축약 연산자는 가독성과 옵티마이저 힌트를 동시에 노린다. BETWEEN 의 경계 포함, IN 의 NULL 미포함은 "= 비교의 산술적 의미" 를 그대로 따른 결과다 — x = NULL 이 NULL 인 이상 IN 도 NULL 을 못 잡는다. (IN은 내부적으로 = 연산자를 쓰는데, SQL에서 = 연산자로는 NULL을 비교(참으로 판정)할 수 없기 때문에, IN 연산자 괄호 안에 NULL을 아무리 집어넣어도 실제 NULL인 데이터를 찾아낼 수 없다는 의미)
다음 챕터로 가는 다리 — 한 테이블에서 행을 골라내는 도구는 충분히 봤다. 두 테이블을 합쳐야 답이 나오는 질문은? "JOB_ID 가 J4 인 직원의 직급명" 처럼 두 테이블에 정보가 나뉘어 있을 때 — JOIN 이 들어온다.

CH 03 DAY3JOIN — USING vs ON, 그리고 NULL 의 위험#

학습

-- USING: 양쪽 테이블의 연결 컬럼명이 동일할 때
select emp_id, emp_name, job_id, job_title
from employee
join job using (job_id);

-- ON: 일반형. 컬럼명이 달라도 OK
select e.emp_id, e.emp_name, d.dept_name
from employee e
join department d on e.dept_id = d.dept_id;

-- 다중 JOIN
select emp_id, emp_name, job_title, dept_name
from employee
join job using (job_id)
join department using (dept_id);
The USING(col) clause is a shorthand. It requires that the joined tables have a column of the same name, and produces a single combined column in the result, whereas ON e.col = d.col keeps both. — MySQL Reference Manual § 13.2.10.2 JOIN Clause

의문 → 가설

가설 기본 JOIN(INNER) 은 매칭이 되지 않는 행을 결과에서 제외한다. 그렇다면 부서가 배정되지 않은 직원(dept_id IS NULL) 은 join department 의 결과에서 사라진다. "전체 22명" 이 아닌 "부서 있는 직원 수" 만 출력되어야 한다.

테스트 — INNER JOIN 의 행 손실

INNER JOIN COUNT(*) — 부서 없는 3행 손실 확인
-- employee 단독 카운트
+----------+
| count(*) |
+----------+
|       22 |
+----------+
1 row in set (0.00 sec)

-- employee JOIN department 카운트
+----------+
| count(*) |
+----------+
|       19 |
+----------+
1 row in set (0.00 sec)

-- dept_id 가 NULL 인 3명이 사라짐
CAUTION "왜 결과가 22 행이 아니지?" 가 SQL 디버깅에서 가장 자주 마주치는 함정이다. 부서 없는 행도 보여야 한다면 LEFT JOIN 으로 바꿔야 한다 — 그러나 그러면 dept_name 자리에 NULL 이 들어오므로 CH04 의 IFNULL/COALESCE 가 곧 필요해진다.

결론(중간)

KEY POINT JOIN 의 종류는 본질적으로 "매칭되지 않는 행을 어떻게 처리할 것인가" 의 정책 선택이다. INNER = 버린다, LEFT/RIGHT = 한쪽은 살리고 반대편을 NULL 로 채운다, FULL OUTER = 양쪽 다 살린다 (MySQL 은 미지원, UNION 으로 우회).
다음 챕터로 가는 다리 — JOIN 으로 NULL 이 만들어지고, NULL 은 산술 연산을 통째로 NULL 로 만들어 버린다. 그 자리를 채우는 함수가 필요하다.

CH 04 DAY3함수와 NULL — IFNULL/COALESCE 로 빠진 자리를 채우기#

학습

-- 보너스포인트가 NULL 인 직원의 '연봉' 이 NULL 로 사라지는 문제
select emp_id, emp_name, salary, bonus_pct,
       (salary + salary * bonus_pct) * 12 as 연봉      -- NULL 발생
from employee;

-- IFNULL 로 NULL → 0 치환
select emp_id, emp_name, salary, bonus_pct,
       (salary + salary * ifnull(bonus_pct, 0)) * 12 as "1년 연봉"
from employee;

-- COALESCE: 여러 인자 중 처음으로 NULL 이 아닌 값
select emp_name, coalesce(phone, email, '연락처 없음') as 연락처
from employee;
NULL is not a value; it is a marker for the absence of a value. Any arithmetic expression containing NULL evaluates to NULL. IFNULL(expr1, expr2) returns expr2 if expr1 is NULL. — MySQL Reference Manual § Working with NULL Values

의문 → 가설

가설 NULL * 12 = NULL 이라는 3치 논리의 결과로 인해, 보너스 미설정 직원의 연봉이 통째로 사라진다. 이를 막으려면 산술 전에 NULL 을 다른 값으로 바꿔야 한다. IFNULL 은 2-arg, COALESCE 는 N-arg 의 같은 패턴이다.

테스트 — IFNULL 적용 전/후

SELECT IFNULL(bonus_pct, 0) — NULL 연봉 복구 확인
-- IFNULL 미적용
+--------+---------+---------+-----------+----------+
| emp_id | emp_name| salary  | bonus_pct | 연봉     |
+--------+---------+---------+-----------+----------+
| 200    | 이미희  | 7000000 |      NULL |     NULL |   -- 사라짐
| 220    | 김민지  | 4500000 |      0.20 | 64800000 |
+--------+---------+---------+-----------+----------+

-- IFNULL(bonus_pct, 0) 적용
+--------+---------+---------+-----------+----------+
| emp_id | emp_name| salary  | bonus_pct | 1년 연봉 |
+--------+---------+---------+-----------+----------+
| 200    | 이미희  | 7000000 |      NULL | 84000000 |   -- 복구
| 220    | 김민지  | 4500000 |      0.20 | 64800000 |
+--------+---------+---------+-----------+----------+

결론(중간)

KEY POINT NULL 은 "값의 부재" 이고 산술과 만나면 결과 전체를 NULL 로 오염시킨다. IFNULL/COALESCE 는 그 오염을 막는 가장 흔한 도구다. 파이썬 x or 0 패턴이 빈 값을 기본값으로 대체하는 것과 같은 구조다.
다음 챕터로 가는 다리 — 결과 집합이 만들어졌다. 마지막으로 보이는 단계 — DISTINCT 와 ORDER BY 와 별칭이 어떻게 결과의 "모양" 을 결정하는지 본다.

CH 05 DAY3DISTINCT · ORDER BY · 별칭 — 마지막 단계가 가장 보이는 단계#

학습

-- DISTINCT: 한 번만 사용 가능, 컬럼 묶음 단위 중복 제거
select distinct dept_id, job_id
from employee
order by 1, 2;

-- ORDER BY 에서 사용 가능한 3가지 — 컬럼명 / 별칭 / select 절 순번
select emp_id 사번, emp_name 이름, salary 급여
from employee
order by 3 desc, 사번 asc;    -- 3 = salary 별칭 = 급여

DISTINCT 는 SELECT 절에서 한 번만 사용 가능하고, 그 뒤에 나열된 컬럼/계산식 묶음이 같은 행만 중복 제거된다. ORDER BY 는 ① 원래 컬럼명 ② SELECT 절의 별칭 ③ SELECT 절의 순번 — 세 가지 모두 받는다.

ORDER BY is evaluated after the SELECT list, so it can reference column aliases defined there. DISTINCT may appear only once, immediately after SELECT, and applies to the combination of all listed columns. — MySQL Reference Manual § 13.2.13 SELECT Statement

의문 → 가설

가설 DISTINCT 가 한 번만 허용되는 이유는 — SELECT 의 중복 제거는 최종 결과 집합 전체에 겸쳐야 하기 때문이다. 컨럼별로 다른 기준으로 중복을 제거한다는 것은 "어떤 행을 남길지" 기준이 모순을 일으킨다. ORDER BY 가 별칭을 받을 수 있는 이유는 단순히 — ORDER BY 가 SELECT 보다 나중에 평가되기 때문이다.

테스트 — DISTINCT 한 번 제한

SELECT DISTINCT dept_id, job_id — 한 번 제한 · ERROR 1064 확인
ERROR 1064 (42000): You have an error in your SQL syntax;
-- "select distinct dept_id, distinct job_id from employee;"

-- 올바른 사용: 한 번만 distinct, 뒤에 컬럼 나열
+---------+--------+
| dept_id | job_id |
+---------+--------+
| 10      | J1     |
| 20      | J3     |
| 90      | J2     |
| NULL    | J5     |
+---------+--------+
7 rows in set (0.00 sec)

결론(중간)

KEY POINT DISTINCT 는 "어떤 컬럼 묶음 단위로 중복을 정의할 것인가" 이고, ORDER BY 는 6단계라서 별칭/순번을 모두 받는다. 두 절 모두 결과의 "보이는 모양" 을 결정하는 마지막 단계다.
다음 첵터로 가는 다리 — SELECT 6절 파이프라인의 모든 단계를 추적했다. 작성 순서와 실행 순서의 차이, WHERE에서 별칭이 안 되는 이유, JOIN에서 행이 수는 이유와 NULL 산술 오염까지 — 모두 한 나의 결론으로 있다.

CH 06최종 결론 — 6 절은 한 줄짜리 파이프라인이다#

최종 정리

출발 의문 — "왜 6 절로 쪼개고, 작성 순서와 실행 순서가 다른가?" 에 대한 답이 다섯 챕터에서 모두 모였다. 각 절은 파이프라인의 한 단계이고, 작성 순서는 사람이 읽기 좋은 순서, 실행 순서는 데이터가 흘러가는 순서다. 

-- 파이썬 list comprehension 으로 비유하면:
[salary*12 for emp in employees join job
 if emp.dept_id in ('20','90')
 sorted by salary*12 desc]
--    ↑ 5         1      2          6

최종 결론

FINAL CONCLUSION SELECT 가 5단계라는 사실이 — WHERE 별칭 사용 불가 라는 한 줄짜리 에러로 입증되고, INNER JOIN 의 행 손실이 — 22 → 19 라는 카운트 차이로 드러나고, NULL 의 산술 오염이 — IFNULL(bonus_pct, 0) 한 줄로 복구된다. SQL 의 SELECT 는 단일 문장이 아니라 6단 파이프라인 이고, 디버깅은 단계마다 따로 사고해야 한다. 다음 의문 — 그러면 이 모든 도구를 들고 실제 데이터를 어디서 가져오는가? 그것이 0514 의 크롤링이 답한다.

CH 01“한 줄씩 실행”이라는 말의 의심

학습

파이썬을 처음 배울 때 가장 먼저 듣는 정의 중 하나는 “파이썬은 인터프리터 언어라 코드를 한 줄씩 실행한다”다. 컴파일 언어처럼 빌드 단계가 따로 없고, 인터프리터가 위에서부터 한 줄을 읽고 그 자리에서 처리한다는 그림이다.

의문

Q “한 줄씩 실행한다”라는 말이 정말 사실이라면, 디버거에서 실행이 닿기 전에는 그 줄에 적힌 변수가 보이지 않아야 하고, 실행 도중 그 줄을 고쳐 두면 다음 실행에 반영되어야 하지 않을까?

여기서 가설이 두 갈래로 갈린다.

  • 가설 A — “한 줄씩”이 사실이라면, 디버거 진행과 함께 변수가 등장해야 한다. ✓
  • 가설 B — 같은 논리라면, 실행 중에 다음 줄을 수정·저장하면 디버거가 그 변경을 읽어야 한다. ✓

가설을 “디버거 Step Over”라는 한 가지 도구로 둘 다 검증해 보자. 한 가설이라도 깨지면, 그 자리가 “한 줄씩 실행”이라는 말의 한계다.

결론

KEY POINT 이 글은 “한 줄씩 실행”이라는 표어가 어디서 무너지는가를 따라가는 기록이다. 결과를 미리 한 줄로 말하면 — 실행 단위는 “줄”이 아니라 코드 블록 단위 바이트코드다.

CH 02첫 시도 — 디버거 Step Over로 변수 추적

학습

VS Code 의 파이썬 디버거는 “현재 라인”과 “지역 변수” 패널을 동시에 보여준다. 변수가 언제 등장하고 언제까지 살아있는지를 시각적으로 따라가기 좋다.

def main():
    a = 10           # 3행
    b = 20           # 4행
    c = a + b        # 5행
    print(a, b, c)

의문 → 가설 A 검증

가설 A 중단점을 4행에 걸고 멈췄을 때, 변수 패널에는 a만 보이고 b는 아직 없어야 한다. 한 줄을 더 진행하면 그제야 b가 등장해야 한다.

실제로 그렇다. 3행 실행 직후 변수 a = 10이 등장하고, 4행을 Step Over 한 뒤에 b = 20이 따라 붙는다. 여기까지는 “한 줄씩”이라는 직관이 맞아 보인다.

결론(잠정)

PARTIAL OK “지역 변수가 그 줄이 실행돼야 등장한다”라는 면에서는 가설 A가 통과한다. 다음 챕터에서 가설 B를 깨뜨리면, 표어 전체가 부서진다.

CH 03함정 — 소스를 고쳐도 디버거의 변수가 바뀌지 않는다

학습

이번에는 가설 B를 시험한다. 3행에서 멈춘 뒤, 실행이 도달하지 않은 4행의 코드를 b = 20 → b = 30으로 바꿔 저장한다. “한 줄씩” 모델이라면 이제 4행을 실행할 때 b가 30이 되어야 한다.

의문 → 가설 B 검증

디버거를 한 줄 진행한 결과, 변수 패널은 여전히 b = 20이다. 파일을 저장했고, 디스크의 소스는 분명히 30인데 디버거의 메모리에는 그 변경이 닿지 않는다.

대조 Visual Studio + C 환경에서 “디버그 빌드(최적화 미적용)”로 같은 실험을 하면, 일부 IDE는 Edit and Continue로 즉시 반영하거나, 반영 불가 사유를 명시한다. 즉 “한 줄씩 실행”이라는 표어는 인터프리터 언어의 전유물이 아니고, 그 표어로 파이썬의 동작을 설명할 수도 없다.

결론(중간)

FAILURE → CLUE 가설 B는 깨졌다. “한 줄씩” 모델이 사실이라면 절대 일어날 수 없는 현상이 일어났다. 즉 파이썬은 “한 줄을 그때그때 읽어 실행”하지 않는다 — 어딘가에서 미리 “고정된 무엇”을 만들어 두고 그걸 돌리고 있다는 뜻이다. 다음 챕터의 단서는 공식 문서다.

CH 04답을 찾아서 — 컴파일 파이프라인(PEP 339)

학습

파이썬 공식 레퍼런스 Execution model 문서에는 다음 두 문장이 있다.

A Python program is constructed from code blocks. … The term ‘interpreter’ here is not the same as the ‘bytecode interpreter’, which is what regularly runs in threads, executing compiled Python code. — docs.python.org/3/reference/executionmodel.html
A block is a piece of Python program text that is executed as a unit. … The local variables of a code block can be determined by scanning the entire text of the block for name binding operations. — 같은 문서

두 문장의 합은 분명하다. 파이썬의 실행 단위는 줄(line)이 아니라 블록(block)이고, 그 블록은 미리 컴파일되어 “바이트코드”라는 형태로 메모리에 자리 잡는다. 이 컴파일은 PEP 339(Design of CPython’s Compiler)가 정리한 4단계로 진행된다.

.py 소스 텍스트 │ ▼ [1] Parse Tree ─ 토큰의 트리(고전적 파스 트리) │ ▼ [2] AST (Abstract Syntax Tree) ─ 문법 의미만 남긴 트리 │ ▼ [3] CFG (Control Flow Graph) ─ 기본 블록과 분기 │ ▼ [4] Bytecode ─ 명령어 리스트 (CD 처럼 “구워진” 결과) │ ▼ PVM (Python Virtual Machine) — 한 명령어씩 페치-디스패치

의문

Q 그렇다면 “한 줄씩” 모델이 깨진 이유가 명확해진다 — 우리가 4행의 소스를 바꿔도, 이미 [4]단계에서 바이트코드가 굳어 버린 뒤이기 때문이다. 디버거가 보는 것은 디스크의 텍스트가 아니라 RAM의 바이트코드.

레퍼런스의 또 한 줄이 이 추정을 보강한다 — “지역 변수는 블록 전체를 훑어 결정된다.” 즉 블록 진입 시점에 변수의 자리(슬롯)와 이름 바인딩 계획이 모두 정해진다. 그래서 파이썬은 단지 “한 줄씩 읽는 인터프리터”가 아니라 “블록 단위로 컴파일하고, 명령어 단위로 인터프리트하는” 두 단계 시스템이다.

결론

KEY POINT 파이썬의 실행 모델은 (1) 블록을 바이트코드로 컴파일 → (2) 바이트코드 인터프리터(PVM)가 명령어를 한 개씩 실행의 두 단계다. “한 줄씩”이라는 말은 (2)의 PVM 차원에서만 비유적으로 맞고, 사용자 시점의 “.py 파일의 한 줄”과는 다르다.

CH 05흔적 추적 — __pycache__는 왜 생기지 않는가

학습

“바이트코드가 굳어진다”는 주장이 사실이라면, 디스크 어딘가에 그 흔적이 남아야 자연스럽다. 공식 문서가 가리키는 곳은 __pycache__ 폴더와 .pyc 파일이다.

Return the PEP 3147/PEP 488 path to the byte-compiled file associated with the source path. For example, if path is /foo/bar/baz.py the return value would be /foo/bar/__pycache__/baz.cpython-32.pyc for Python 3.2. — importlib.util.cache_from_source 도큐먼트
When a Python source file is imported for the first time, a __pycache__ directory will be created in the package directory, if one does not already exist. — PEP 3147

의문 — 그런데 폴더가 비어 있다

Q main.py를 디버깅한 직후, 그 옆 폴더에는 __pycache__가 만들어지지 않았다. 왜?

위 인용의 핵심 단어는 imported다. __pycache__는 “임포트 시스템”의 부산물이다. 그런데 main.py는 임포트되는 모듈이 아니라 직접 실행되는 진입점(__main__)이다. 진입점은 “모듈로서 캐시될 후보”에 들지 않는다.

테스트

이 추정을 한 줄로 검증한다 — 같은 폴더에 test_import.py를 만들고 main.py에서 임포트한 뒤 다시 디버깅한다.

# test_import.py — 비어 있어도 된다, 임포트되는 게 핵심
a = 10

# main.py
import test_import
print(test_import.a)

결과 — 폴더에 __pycache__/가 생기고, test_import.cpython-314.pyc만 들어간다. main.cpython-314.pyc는 여전히 없다. “임포트된 모듈만 캐시된다”라는 규칙이 그대로 표면에 드러난다.

한 발 더 — HEX 에디터로 .pyc 내부를 열다

여기서 의문이 하나 더 생긴다. test_import.pyc가 생겼으니 그 안에 우리가 선언한 a = 10이 담겨 있어야 하지 않을까? HEX 에디터로 직접 열어 확인한다.

설정 실험 당시 test_import.py는 a = 10 한 줄짜리 파일이었다. 파일 크기가 작아 HEX 에디터로 전체 내용을 한 화면에 볼 수 있다.

HEX 데이터를 태깅하면 구조가 드러난다.

[Header] Magic : 0x0D0A0E2B ← Python 3.13 바이트코드 식별자 Time : 2026-04-26 18:34:09 ← 소스 파일 최종 수정 시각 [Code Object] Filename: "F:\_proj\_study\python\_practice\test_import.py" Name : "<module>" Instructions: RESUME, RETURN_CONST ← 단 두 개
핵심 관찰 Instructions 가 RESUME, RETURN_CONST 두 개뿐이다. a = 10을 저장하는 LOAD_SMALL_INT·STORE_NAME 명령어가 없다. 이유는 간단하다 — 실험 당시 test_import.py는 비어 있었고, 그 상태로 임포트됐다. a = 10은 메인 파일인 main.py에서 접근한 속성이므로 test_import.pyc에는 존재하지 않는다.

이 결과는 두 가지를 동시에 확인해 준다.

  1. .pyc는 단순한 "소스 파일의 복사본"이 아니라 컴파일된 명령어 리스트(바이트코드)다 — 소스 텍스트가 아니라 실행 명령어만 담긴다.
  2. 임포트 시점의 소스 그대로 컴파일되므로, 파일을 나중에 수정해도 캐시가 무효화되기 전까지 이전 바이트코드가 사용된다 (magic + 타임스탬프로 일치 여부를 확인).

결론

KEY POINT __pycache__ 는 임포트 시스템의 부산물이다. 진입점 스크립트(main.py)는 바이트코드를 메모리에 가지고 있을지언정 디스크에 남기지 않는다. 그리고 생성된 .pyc 안에는 소스 텍스트가 아니라 컴파일된 명령어 리스트만 존재한다 — HEX 에디터로 직접 확인한 결과다.

CH 06dis로 본 바이트코드 — 메모리와 디스크의 분리

학습

그렇다면 “디스크에 남지 않는 진입점의 바이트코드”는 어떻게 들여다볼 수 있을까. 표준 라이브러리의 dis 모듈이 그 역할을 한다. dis는 주어진 소스 파일(또는 객체)을 그 자리에서 컴파일해 바이트코드로 풀어 보여준다.

# PowerShell
python -m dis main.py

# 결과 (요약)
  3           RESUME                   0
              LOAD_CONST               0 (10)
              STORE_NAME               0 (a)
  4           LOAD_CONST               1 (20)
              STORE_NAME               1 (b)
  5           LOAD_NAME                0 (a)
              LOAD_NAME                1 (b)
              BINARY_OP                0 (+)
              STORE_NAME               2 (c)

의문 — “디스크의 소스”와 “디버거의 메모리”가 따로 산다는 증명

Q 그럼 디버깅 도중에 main.py의 b = 20을 b = 30으로 고친 뒤 다시 python -m dis main.py를 돌리면, 그 출력은 30일까 20일까?

출력은 30이다. dis는 명령을 받은 그 순간 디스크의 소스를 새로 컴파일하기 때문이다. 그러나 같은 시간에 디버거 안의 메모리에서는 여전히 20이다. 두 “바이트코드”가 동시에 존재하지만 같은 객체가 아닌 것이다.

결론

KEY POINT 실행 중 디버거가 보여주는 변수는 “디버거가 시작될 때 컴파일된 바이트코드 + 그것을 돌리고 있는 PVM 의 프레임”이다. 디스크 텍스트는 그 바이트코드를 만드는 입력일 뿐, 같은 객체가 아니다. 그래서 “파일을 저장”해도 이미 돌아가는 프로세스의 변수는 바뀌지 않는다.

CH 07두 번째 검증 — REPL로 “일부만 실행”

학습

여기서 시점을 바꾼다. 위 챕터들이 “블록 단위 컴파일”의 흔적이라면, REPL(읽기·평가·인쇄 루프)은 “인터프리터 언어다움”이 가장 잘 드러나는 자리다. VS Code 에서 코드 일부를 선택하고 Shift+Enter를 누르면 그 부분만 파이썬 REPL 로 보내져 즉시 실행된다.

의문 → 가설

가설 파일에 에러가 있는 줄이 있어도, REPL이 “그 앞 부분만”을 받아 실행하면 그것은 정상 결과를 내야 한다. 컴파일 언어라면 “파일 단위 빌드”가 있어 부분 실행 자체가 불가능하다. 
# main.py — 6~9행은 정상, 10행에 의도적 IndexError
nums = [1, 2, 3]
total = 0
for n in nums:        #  6
    total += n            #  7
print("total =", total)  #  8
print("max =", max(nums)) #  9
print("x =", nums[99])   # 10  ← IndexError

테스트

6~9행만 블록 선택해 Shift+Enter — REPL은 정상적으로 total = 6, max = 3을 찍는다. 그 다음 10행만 추가로 보내면 그 자리에서 IndexError가 난다. 앞쪽 블록은 뒤쪽의 에러와 무관하게 살아 있는다.

결론

KEY POINT REPL이 보여주는 것은 “블록 단위 컴파일 → 즉시 실행”의 짧은 순환이다. 즉 파이썬에서 블록의 경계는 “파일”이 아니다. 함수 하나, 셀 하나, 선택 영역 하나가 모두 “컴파일·실행의 단위”가 될 수 있다.

CH 08세 번째 검증 — 에러가 있는 프로그램을 일부러 돌린다

학습

만약 파이썬에 “파일 단위 빌드”가 있다면, 파일 안 어디든 에러가 있는 한 프로그램은 시작조차 못 해야 한다. 그러나 우리는 매일 그 반대를 본다 — 함수 안의 NameError는 그 함수가 호출될 때까지 잠잠하다.

의문 → 가설

가설 파일에 에러가 있어도 실행이 그 줄에 닿기 전까지는 프로세스가 정상 진행된다. 단, “구문 오류(SyntaxError)”는 컴파일 단계에서 잡히므로 시작 자체가 안 된다.

테스트

# main.py
print("start")
print("middle")
print(undefined_name)   #  NameError — 그러나 “시작” 직후에 발생
print("never")

F5 로 디버깅을 시작하면 터미널에는 start → middle 두 줄이 그대로 찍히고, 세 번째 줄에서 NameError가 발생하며 종료된다. 컴파일 언어 모델로는 설명되지 않는 진행 방식이다.

반대 실험: 같은 파일에 print(처럼 괄호가 닫히지 않은 구문을 넣고 실행하면, 첫 줄도 찍히지 않은 채 SyntaxError가 난다. 이는 “블록 컴파일 단계에서 실패”했기 때문이다.

결론

오류의 종류잡히는 시점관측 동작
SyntaxError블록 컴파일첫 줄도 실행되지 않는다
NameError / TypeError / IndexError 등실행해당 줄에 닿기 전까지는 정상 진행
KEY POINT “인터프리터 언어 = 에러 줄까지는 정상 실행”이라는 정설은 실행 시점 오류에 한해서만 맞다. 구문 오류는 어차피 컴파일 단계에서 차단된다. 즉 파이썬 안에는 “컴파일 측면”과 “인터프리트 측면”이 함께 있다.

압축

이 글의 8개 챕터는 한 문장으로 줄어든다.

ONE-LINER 파이썬은 “블록을 바이트코드로 컴파일하고, 그 바이트코드를 PVM이 명령어 단위로 인터프리트”하는 두 단계 시스템이다. “한 줄씩 실행”은 그 인터프리트 단계에서나 통하는 비유이고, 사용자가 보는 .py 파일의 “한 줄”과는 분리된 개념이다.

실무 체크리스트

  • 디버깅 중 변수 값을 바꾸고 싶다면 — 소스 파일 수정이 아니라 디버거의 변수 직접 수정 기능을 쓴다.
  • __pycache__가 안 생긴다고 “컴파일이 안 됐다”라고 단정하지 말 것. 임포트된 모듈만 캐시된다.
  • 진입점의 바이트코드를 보고 싶다면 python -m dis main.py. 단 그 결과는 그 시점의 디스크 파일을 컴파일한 것이지, 이미 돌아가는 프로세스의 메모리는 아니다.
  • 파일에 실행 시점 오류가 있어도 그 줄에 닿기 전까지는 정상 진행. 단 SyntaxError는 “블록 컴파일” 단계에서 차단되므로 처음부터 멈춘다.
  • “파이썬은 인터프리터 언어”라는 한 줄로 끝내지 말 것. 컴파일과 인터프리트가 같은 런타임 안에 공존한다.

한 그림으로

[ 디스크 ] [ 프로세스 메모리 ] main.py ──── python -m dis ────► (그 시점의) bytecode 출력 │ │ python main.py / F5 ▼ 컴파일 (Parse → AST → CFG → Bytecode) │ ▼ PVM 이 명령어 한 개씩 페치-디스패치 │ ▼ 디버거가 보는 “변수 패널” = 이 메모리 ※ 디버거 시작 후 디스크의 main.py 를 수정해도 PVM 이 들고 있는 바이트코드는 그대로 — 두 메모리는 분리

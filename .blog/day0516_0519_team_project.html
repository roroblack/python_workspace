CH 01 DAY6기획 — "물로간다" 의 한 줄 정의와 4명의 역할 분할#

학습 — 주제 선정과 페르소나

"물로간다" 프로젝트의 한 줄 정의: 수소차 구매를 망설이는 사람이 보급 추이·충전소 분포·정책 FAQ 를 한 화면에서 비교할 수 있게 한다. 데이터 출처는 ① 국토부 통계누리(보급) ② 공공데이터포털(충전소) ③ 무공해차 통합누리집(EV/수소 FAQ) ④ 직접 작성 FAQ.

역할담당자핵심 산출물
GUI / 시각화최연우Streamlit 페이지 4개 + Folium 지도 + Plotly 그래프
데이터베이스권소라ERD · MySQL 스키마 · INSERT 모듈
크롤링김지혜BeautifulSoup + Playwright 4 종 크롤러
연동 / 운영박회종크롤러 ↔ DB ↔ UI 의 결합·테스트·발표

의문 → 가설

가설 역할을 ① GUI ② DB ③ 크롤링 ④ 연동의 4 축으로 자른 이유는 — 이 4 축이 동시에 평행 작업이 가능하기 때문이다. ① 은 Mock 데이터로, ② 는 샘플 SQL 로, ③ 은 단독 스크립트로 작업할 수 있다. ④ 가 마지막에 연결하면 그 시점에 드러나는 인터페이스 문제 (스키마 vs UI vs 크롤러 결과 사이의 형식 차이) 가 집중적으로 처리될 것이다.

결론(중간)

KEY POINT 역할 분할의 기준은 "조각이 단독으로 돌아가는가" 여야 한다. 4 명이 서로의 결과를 기다리지 않고 평행 작업할 수 있어야 4 일이라는 짧은 일정 안에 결합 단계까지 도달한다.
다음 챕터로 가는 다리 — 그럼 4 명이 만든 조각이 결합되는 지점, 즉 데이터 파이프라인부터 본다.

CH 02 DAY7데이터 파이프라인 — 4개의 크롤러가 하나의 DB 로 모이는 길#

학습 — 4개의 크롤러 → 4개의 MySQL 테이블

# 파일 구조
SKN32-1st-3Team/
├── crawler_molit.py       # 국토부 통계누리 → tbl_molit_yearly
├── crawler_station.py     # 공공데이터포털 → tbl_station
├── crawler_faq_ev.py      # 무공해차 통합누리집 → tbl_faq_ev
├── faq.py                 # 직접 작성 → tbl_faq_custom
└── app.py                 # Streamlit 메인
-- 핵심 4 테이블 (요약)
CREATE TABLE tbl_molit_yearly (
    year       INT          PRIMARY KEY,
    h2_count   INT          NOT NULL,
    ev_count   INT          NOT NULL
) ENGINE=InnoDB;

CREATE TABLE tbl_station (
    station_id   INT          PRIMARY KEY AUTO_INCREMENT,
    name         VARCHAR(100) NOT NULL,
    lat          DECIMAL(10,7),
    lng          DECIMAL(10,7),
    address      VARCHAR(255),
    status       VARCHAR(20)
) ENGINE=InnoDB;

의문 → 가설

가설 4 개 테이블이 서로 외래키로 묶이지 않은 이유는 — 데이터 출처가 다르기 때문이다. 보급 통계의 "연도" 와 충전소의 "위치" 는 사실 같은 도메인에 있지만, 매칭할 공통 키가 없다. 그래서 결합은 SQL 이 아니라 UI 레벨에서 한다 — 같은 페이지에서 두 그래프를 나란히 보여주는 것으로 결합이 완성된다.

테스트 — Playwright 로 동적 페이지 처리

# crawler_station.py — 공공데이터포털은 JS 렌더링 페이지
from playwright.sync_api import sync_playwright

with sync_playwright() as p:

결론(중간)

KEY POINT 파이프라인 설계의 첫 결정은 "어디서 조인할 것인가" — DB 레벨에서 못 묶는 데이터는 UI 레벨에서 묶는 것이 정직한 답이다. 출처가 다른 데이터를 억지로 FK 로 묶으면 무결성 위반과 데이터 손실만 발생한다.

CH 03 DAY8UI 구조 — Streamlit 4 페이지 + 사이드바 라우팅#

학습

# app.py
import streamlit as st

PAGES = {
    '홈': 'pages.home',
    '등록현황': 'pages.molit',
    '충전소': 'pages.station',
    'FAQ': 'pages.faq',
}
choice = st.sidebar.radio('메뉴', list(PAGES.keys()))
__import__(PAGES[choice]).render()
# pages/station.py — Folium 지도 + 슬라이더 필터
import folium
from streamlit_folium import st_folium

rows = MySQLTemplate.fetch_all('select * from tbl_station where status=%s', ('운영',))
m = folium.Map(location=[36.5, 127.8], zoom_start=7)
for r in rows:
    folium.Marker([r['lat'], r['lng']], popup=r['name']).add_to(m)
st_folium(m, width=720, height=520)

의문 → 가설

가설 Streamlit 의 페이지 전환은 "스크립트 전체를 다시 실행" 한다 — 그래서 페이지마다 SQL 을 다시 던지면 응답이 느리다. @st.cache_data 로 DB 결과를 캐싱하면 첫 조회만 느리고, 이후 페이지 전환은 즉시 일어난다.

결론(중간)

KEY POINT Streamlit + DB 조합에서 캐싱은 선택이 아니라 필수다. 그러나 캐시는 TTL 과 키 설계가 잘못되면 "이미 갱신된 데이터를 못 보는 사용자" 를 만든다. ttl=60 정도의 짧은 캐시가 균형점.

CH 04 DAY9시행착오 — 협업 충돌·동적 페이지·세션 캐싱#

발생한 5개의 진짜 문제

증상원인해결
git push 시 conflict 폭주app.py 를 4명이 동시 편집페이지 파일 분리(pages/*.py) + 사이드바 라우터만 공동 편집
충전소 페이지 빈 지도JS 렌더링 페이지를 urllib 로 시도Playwright 로 교체(0515 학습 적용)
지도 1000+ 마커 시 멈춤folium 의 개별 Marker 렌더링 한계MarkerCluster 사용
FAQ 검색 한국어 깨짐DB charset = utf8mb3utf8mb4 로 ALTER + 코드 재삽입
새로고침 시 필터 초기화session_state 미사용st.session_state['filter'] 도입

의문 → 가설

가설 5 개 증상의 공통점은 — 모두 "혼자 짤 때는 안 만나는 문제" 다. 협업 도구(git), 동적 페이지, 대용량, 인코딩, 세션은 모두 "사람이 여럿이고 데이터가 많고 사용자가 실제로 쓸 때" 비로소 드러난다. 그래서 진짜 학습은 첫 단위 프로젝트에서 시작된다.

결론(중간)

KEY POINT 혼자 짠 코드가 잘 돈다는 사실은 그 코드가 시스템에 통합 가능하다는 증거가 아니다 — 두 사람 이상이 짠 코드가 충돌 없이 합쳐지는 것이 진짜 증명이다. 그래서 단위 프로젝트의 가장 큰 학습은 코드보다 git, 모듈 경계, 캐싱, 인코딩 같은 운영 지식이다.

FINAL CONCLUSION "물로간다" 의 진짜 성과는 수소차 보급 추이를 그래프로 그린 것이 아니다 — 월요일에 배운 자료형이 금요일에는 충전소 위도/경도 컬럼이 되었고, 화요일의 제약조건이 데이터 무결성을 지켰으며, 수요일의 SELECT 6 절이 사용자 필터의 SQL 이 되었고, 목요일의 bs4 와 금요일의 Selenium 이 실제 데이터를 길어 올렸다 는 사실이다. 한 주의 단편적 학습이 한 화면에서 만나 시스템이 되는 경험 — 이것이 단위 프로젝트의 본질. 그리고 이 모든 데이터를 분석 가능한 형태로 다듬는 것이 다음 단계 — 0520 의 numpy 다.

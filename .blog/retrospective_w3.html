CH 01한 줄 의문 — "데이터를 어떻게 안전하게 저장하고 꺼내며 외부에서 가져오는가"

2주차의 끝에는 Streamlit 으로 CSV 를 읽어 그래프를 그렸다. 그러나 그 직후 떠오른 의문이 있었다 —

3주차의 한 줄 의문 CSV 는 한 사람 한 컴퓨터의 데이터다. 여러 사람이 동시에 쓰는 데이터는 어디에 둬야 하는가? 그리고 그 데이터를 처음부터 끝까지 "사람이 직접 작성" 할 수 없다면, 어떻게 자동으로 모아야 하는가?

이 의문이 3주차의 5 일을 끌고 갔다. 0511~0513 은 "어디에 둘 것인가" — DBMS. 0514~0515 는 "어떻게 모을 것인가" — 크롤링. 두 축이 서로 독립적으로 보이지만, 사실 한 문장의 앞/뒤 절이다.

CH 02사슬 ①·② — DBMS 도입(0511) 과 제약조건(0512): '믿을 수 있는 행' 이란 무엇인가

0511 — DBMS 가 CSV 와 다른 지점

첫째 날의 학습은 단순히 "MySQL 설치" 가 아니었다. 핵심은 두 가지 — ① 권한 분리(root vs 일반 사용자) 와 ② 자료형 강제 였다. CSV 는 모든 셀이 문자열이지만, MySQL 의 INT 컬럼에는 문자열을 넣을 수 없다. "잘못된 입력을 자료형 차원에서 차단" 이라는 새 개념이 도입됐다.

0512 — 자료형의 빈 틈을 메우는 제약조건

그런데 자료형만으로는 부족했다. INT 컬럼이라고 해서 같은 사번이 두 번 들어가는 것을 막을 수는 없다. 그 빈 틈을 메우는 5 개의 제약조건 — NOT NULL, UNIQUE, PRIMARY KEY, CHECK, FOREIGN KEY — 가 등장했다. 각각이 "잘못된 행" 의 다른 정의 다.

제약"잘못된 행" 의 정의MySQL ERROR
NOT NULL필수 컬럼이 비어있는 행1048
UNIQUE중복된 값을 가진 행1062
PRIMARY KEY식별자 없는 행 (NOT NULL + UNIQUE)1062 / 1048
CHECK업무 규칙을 위반하는 행3819
FOREIGN KEY참조 무결성을 깨는 행1452
CHAIN POINT "믿을 수 있는 행" 은 자료형 + 5 개 제약조건의 통과로 정의된다. 그래서 0511 과 0512 는 따로 떨어진 강의가 아니라, "데이터 신뢰성의 위에서부터 아래로" 의 두 층이다.

CH 03사슬 ③ — SELECT 6 절(0513): '꺼내는 일' 이 곧 '파이프라인 설계'

저장만 잘 되어 있다고 데이터가 가치를 만들지 않는다. 어떻게 꺼내느냐가 가치를 결정한다. 그 답이 SELECT 의 6 절 — FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. 이 순서는 단순 문법이 아니라 실행 순서이며, 이 순서를 이해해야 별칭이 어느 절에서 쓸 수 있는지가 자연히 정해진다. 

-- ERROR 1054 — 별칭은 WHERE 에서 쓸 수 없다
SELECT salary * 12 AS annual
FROM employees
WHERE annual > 50000;            -- annual 은 SELECT 단계 결과

-- OK — ORDER BY 는 SELECT 다음 단계
SELECT salary * 12 AS annual
FROM employees
ORDER BY annual DESC;

여기에 JOIN 의 USING vs ON, INNER 의 행 손실, IFNULL/COALESCE 의 NULL 오염 복구가 더해지며, SELECT 는 "데이터를 꺼내는 한 줄" 이 아니라 "6 단계 파이프라인" 이라는 사실이 드러났다.

CHAIN POINT SQL 의 6 절은 0514~0515 에서 만들 크롤러 결과를 가공하는 표준 도구다. 어디서 NULL 이 발생하고, 어디서 행이 손실되고, 어디서 별칭이 살아남는지가 곧 데이터 품질 보증의 체크리스트가 된다.

CH 04사슬 ④·⑤ — 정적/동적 크롤링(0514, 0515): '들어오는 길' 의 두 가지 시점

0514 — 정적 크롤링: HTTP 응답 시점

BeautifulSoup4 는 urllib.request 가 가져온 응답 바이트를 HTML 트리로 파싱한다. 그 트리에서 find/find_all/select 로 노드를 꺼내고, %s 매개변수 바인딩으로 MySQL 에 흘려보내는 패턴 — 이것이 정적 크롤링의 전부다. 단, "동적 해시 클래스명" 에 의존하면 1 주일 만에 깨지므로, data 속성/태그 구조가 안정적이다.

0515 — 동적 크롤링: 렌더링 시점

그러나 페이지가 JavaScript 로 그려진다면 urlopen 응답에는 데이터가 없다. Selenium 은 실제 브라우저를 띄워 JS 실행 후 DOM 을 본다. WebDriverWait + expected_conditions 의 명시적 대기, webdriver-manager 의 자동 드라이버 관리, 한국어 텍스트 기반 선택자 — 이 셋이 안정성의 핵심이다.

축정적(0514)동적(0515)
입력HTTP 응답 바이트실행된 DOM 스냅샷
대기불필요WebDriverWait 필수
비용저렴높음(브라우저 프로세스)
안정 선택자의미 있는 클래스data 속성 / 한국어 라벨 / 태그 구조
CHAIN POINT 정적과 동적은 "어떤 데이터인가" 가 아니라 "데이터가 만들어지는 시점이 언제인가" 로 갈린다. 한 사이트 안에 두 종류가 섞여 있을 수도 있다 — 그래서 두 도구는 대체재가 아니라 보완재다.

CH 05잘한 점 / 부족한 점 / 다음 주 계획

A 잘한 점

  • 한 줄 의문 → 사슬 학습 → 결론 회수 의 구조로 5일을 묶었다. 단편적 명령어를 외우는 대신, "왜 이 도구가 필요한가" 를 매일 한 번씩 다시 물었다.
  • ERROR 코드 기반 학습. ERROR 1048, 1062, 1452, 3819 등 MySQL 의 에러를 의도적으로 발생시켜 각 제약의 의미를 코드로 확인했다. 시험 답안이 아니라 운영 매뉴얼처럼 머리에 남았다.
  • 크롤링 선택자의 안정성 기준 을 명문화했다 — "사람이 한 줄로 설명할 수 있는 선택자가 안정적이다". 해시 클래스명 의존을 피하는 습관이 잡혔다.
  • 매개변수 바인딩(%s) 을 처음부터 표준 으로 썼다. f-string SQL 의 위험성을 코드로 확인하기 전에 차단.

B 부족한 점

  • 인덱스 / 실행계획(EXPLAIN) 미학습. SELECT 6 절은 익혔지만, 같은 쿼리가 1초와 1분의 차이를 만드는 이유 — 인덱스 설계 — 는 아직 손에 잡히지 않았다.
  • 트랜잭션과 격리 수준. 단건 INSERT 만 다뤄봤지, 동시 접근 시의 락/데드락은 경험 부족.
  • 크롤링 윤리/속도 제어. robots.txt 준수, time.sleep 간격, User-Agent 설정 같은 운영 매너를 체계적으로 정리하지 못했다.
  • 예외 처리. 크롤링 도중 네트워크 단절/페이지 변경에 대한 재시도/백오프 패턴이 빈약했다 — try/except 만으로는 운영 환경을 못 버틴다.

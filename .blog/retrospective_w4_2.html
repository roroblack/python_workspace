CH 020520NumPy 분석 도구와 라이브러리 생태계#

학습 내용

0520의 핵심은 "왜 NumPy인가?"를 이해하는 것이었다. Python 기본 리스트로도 수치 연산이 가능한데, NumPy가 별도 라이브러리로 존재하는 이유는 C 기반 연속 메모리 배열(ndarray)과 BLAS/LAPACK 연동 선형대수 연산 때문이다.

라이브러리역할의존성
NumPyn차원 배열 연산 · 선형대수 · 난수 생성기반 계층
Pandas라벨 인덱스 + 테이블형 데이터NumPy 위
matplotlib시각화 기반NumPy 위
scikit-learn머신러닝 알고리즘NumPy + SciPy

실습에서는 test_numpy/ 폴더에 numpy_test1~10.py를 작성하며 ndarray 생성·슬라이싱·연산·함수를 확인했다. 특히 dtype 지정과 NaN·Inf 전파 규칙이 처음에는 헷갈렸는데, 실제로 np.array(['서울', '부산경남'], dtype='U3') 에서 '부산경남'이 '부산경'으로 잘리는 것을 직접 확인하면서 완전히 이해했다.

"NumPy's ndarray is a fixed-type, contiguous-memory array. Broadcasting allows universal functions to operate element-wise on arrays of different shapes." — 교재 데이터분석_라이브러리.pdf · 2장 NumPy 기초

잘 이해된 것

이해 확인
  • 브로드캐스팅 규칙: 작은 배열이 큰 배열 shape에 맞게 확장되는 원리
  • nan == nan이 False인 이유 — IEEE 754 표준
  • vstack/hstack의 shape 제약 — 연결 방향의 반대 축이 같아야 함

어려웠던 것

CHALLENGE dtype='U3'과 dtype='S3'의 차이(유니코드 vs. 바이트)를 처음엔 혼동했다. 한글은 유니코드 문자열(U 타입)로 처리해야 하며 S 타입으로 지정하면 한글이 올바르게 저장되지 않는다는 것을 직접 테스트 후 확인했다.

CH 030521변수 변환과 결측값 처리#

학습 내용

0521의 핵심 질문은 "결측값을 어떻게 처리하는 게 데이터를 덜 왜곡할까?"였다. titanic 데이터셋의 age 컬럼(결측값 19.9%)으로 네 가지 전략을 비교했다.

전략meanstdcount
원본(NaN 포함)29.7014.53714
fillna(0)23.8017.60891
fillna(mean)29.7013.00891
fillna(median)29.3613.02891
dropna()29.7014.53714

fillna(0)은 평균이 23.8로 떨어지고 표준편차가 17.6으로 치솟는다. 결측값이 "나이=0"과 같이 해석되어 분포를 심각하게 왜곡한다. 평균 대체는 평균을 유지하지만 표준편차가 13.0으로 줄어들었다 — "가짜로 채워진 값들이 분포를 좁힌다"는 것을 수치로 확인한 것이다.

"Mean imputation preserves the mean but underestimates variance, potentially leading to underfitting in downstream models." — 교재 데이터전처리.pdf · 3장 결측값 처리 전략 비교

잘 이해된 것

이해 확인
  • .iloc(위치 기반)과 .loc(라벨 기반) 명시 구분 — 라벨이 정수일 때 혼동 방지
  • log 변환은 왜도를 줄이지만 이미 작은 왜도에 적용하면 역방향 왜도 발생
  • pd.cut 구간화는 해석 편의를 높이지만 예측 모델에서는 연속값이 원칙

어려웠던 것

CHALLENGE log1p 변환 후 왜도가 0.39 → -1.88로 오히려 반대 방향이 됐다. 처음엔 변환 코드가 잘못됐다고 생각했는데, titanic age의 원래 왜도가 이미 작아서 log 변환이 과보정이 된 것이었다. 왜도 크기를 먼저 확인하고 변환 여부를 결정해야 한다는 순서를 배웠다.

CH 040522이상값 탐지와 데이터 정규화#

학습 내용

0522의 핵심은 auto-mpg 데이터셋의 horsepower 컬럼으로 IQR과 Z-score 두 방법을 직접 비교한 것이었다. IQR 기준(하한=-1.5, 상한=202.5)으로는 10개, Z-score(|Z|>3)로는 5개가 탐지됐다. IQR이 Z-score보다 더 많은 이상값을 탐지했다.

방법이상값 수범위가정
IQR10개>202.5 또는 <-1.5분포 무관
Z-score5개|Z| > 3정규분포
두 방법 공통5개220, 225, 225, 230 등—

이상값 처리 후 스케일링 비교:

  • MinMaxScaler: mpg → mean=0.3842, 범위 [0, 1]. 이상값에 민감.
  • StandardScaler: mpg → mean=0.000, std=1.000. 정규분포에 안정적.
"IQR-based outlier detection is more robust than Z-score for skewed distributions because it relies on quantiles rather than mean and standard deviation." — 교재 데이터분석_통계.pdf · 4장 이상값 탐지 기법 비교

잘 이해된 것

이해 확인
  • IQR 방법이 Z-score보다 오른쪽 꼬리 분포에서 더 많은 이상값을 잡는 이유
  • 이상값 처리 3전략(제거/대체/캡핑) 중 캡핑이 평균을 가장 잘 보존하는 이유
  • MinMax는 이상값 존재 시 정상값이 0 근처로 압축되는 메커니즘

어려웠던 것

CHALLENGE auto-mpg 데이터의 horsepower 열에 '?' 문자가 섞여 있어 pd.read_csv()로 불러오면 float이 아닌 object 타입이 된다. na_values=['?']를 지정해야 NaN으로 자동 변환된다는 것을 describe()의 count 차이(398 vs. 392)에서 처음 발견했다. 로드 단계의 옵션이 전처리 결과를 좌우한다는 것을 체감했다.
다음 챕터로 가는 다리 — 3일의 개별 학습 내용을 파악했다. 세 날의 학습이 하나의 전처리 파이프라인으로 어떻게 연결되는지 정리한다.

CH 053일의 연결 — 전처리 파이프라인으로 통합#

0520~0522의 학습을 하나의 파이프라인으로 연결하면 다음과 같다.

┌────────────────────────────────────────────────────────────┐ │ 데이터 전처리 파이프라인 (0520~0522 통합) │ ├────────────────────────────────────────────────────────────┤ │ [0520] 탐색 dtype · shape · NaN/Inf 탐지 │ │ ↓ │ │ [0521] 결측 isna → fillna(mean/median) / dropna │ │ ↓ 변환 astype · log1p · pd.cut │ │ ↓ │ │ [0522] 이상값 IQR / Z-score → 제거·대체·캡핑 │ │ ↓ 스케일 MinMaxScaler / StandardScaler │ │ ↓ │ │ [검증] describe 재확인 · isna().sum()==0 · shape 확인 │ └────────────────────────────────────────────────────────────┘

각 단계는 "다음 단계가 올바르게 작동하기 위한 조건"을 만들어 준다. dtype이 맞지 않으면 fillna가 실패하고, 이상값이 남아 있으면 MinMaxScaler 결과가 왜곡된다. 순서를 지키는 것 자체가 전처리 품질을 보장하는 방법이다.

KEY POINT 전처리는 "문제 있는 값을 고치는 것"이 아니라 "다음 분석 단계가 올바른 가정 위에서 작동하도록 보장하는 것"이다. 각 단계를 마칠 때마다 describe()로 검증하는 습관이 필요하다.

부족했던 점

개선 필요
  • 데이터 로드 옵션 무시 — na_values=['?']처럼 로드 단계 옵션이 후속 전처리에 미치는 영향을 처음엔 간과했다. 실무에서는 원천 데이터의 결측 표현 방식을 먼저 파악해야 한다.

  • 다중공선성 미확인 — 상관행렬을 보지 않고 스케일링만 했다. corr(mpg, weight)=-0.832처럼 강한 음의 상관이 있는 변수를 함께 모델에 넣으면 VIF 문제가 생긴다는 것을 0526 실습에서 사후 확인했다.

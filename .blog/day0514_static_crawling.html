CH 01 DAY4urllib + bs4 — 페이지 가져오기와 파싱의 분리#

학습

# crawling/crawling_bs4.py
import urllib.request, bs4

# 1. URL 로 웹페이지 접속 → response 객체
web_page = urllib.request.urlopen('https://www.naver.com')
print(web_page)   # <http.client.HTTPResponse object at 0x...>

# 2. 응답 바이트를 HTML 파서로 트리화
decoding_code = bs4.BeautifulSoup(web_page, 'html.parser')
print(decoding_code)   # 전체 HTML 트리 출력
Beautiful Soup is a Python library for pulling data out of HTML and XML files. It works with your favorite parser to provide idiomatic ways of navigating, searching, and modifying the parse tree. — Beautiful Soup Documentation

의문 → 가설

가설 "페이지를 가져온다" 와 "HTML 을 분석한다" 는 두 가지가 분리되어 있는 이유는 — 한쪽은 네트워크 I/O, 다른 한쪽은 CPU 파싱이기 때문이다. 그러므로 ① urllib 는 어떤 응답이든 받아오고 ② bs4 는 그 응답을 어떤 파서(html.parser, lxml, html5lib)로도 처리할 수 있어야 한다. 이 분리 덕분에 — 같은 URL 에 대해 파일/문자열/응답객체 어느 입력이든 같은 트리가 나와야 한다.

테스트 — 같은 입력 다른 소스

urllib.request.urlopen + BeautifulSoup — NAVER HTML 파싱 확인
<http.client.HTTPResponse object at 0x000001A2D7E0F8B0>
<!DOCTYPE html>
<html lang="ko">
<head>
<meta charset="utf-8"/>
<meta content="origin" name="Referrer"/>
...
<title>NAVER</title>
...

결론(중간)

KEY POINT 네트워크 I/O 와 HTML 파싱은 분리되어야 한다 — 그래야 같은 파서로 응답/파일/문자열을 모두 처리할 수 있고, 같은 응답을 여러 파서로 비교할 수 있다. 이 분리는 단순한 코딩 규약이 아니라 관심사의 분리(SoC) 의 표준 적용이다.
다음 챕터로 가는 다리 — 트리가 손에 들어왔다. 이제 "어느 가지의 어느 잎" 을 어떻게 가리킬 것인가.

CH 02 DAY4find vs find_all — 한 개 vs 여러 개의 의도 차이#

학습

soup = bs4.BeautifulSoup(web_page, 'html.parser')

# find: 첫 번째 매칭 element 한 개만
data_box = soup.find('div', {'class': 'data_box'})

# find_all: 매칭되는 모든 element 의 리스트
movie_list = soup.find_all('a', {'class': 'this_text'})
print(len(movie_list))     # 갯수

# CSS 선택자 — select / select_one
title = soup.select_one('div.data_box a.this_text')
titles = soup.select('div.data_box a.this_text')
find() returns the first match or None; find_all() returns a list of all matches or an empty list. The two behave differently on zero results, which drives different defensive coding patterns. — Beautiful Soup Documentation § find()

의문 → 가설

가설 find 와 find_all 의 차이는 단순한 "개수" 가 아니라 — "있다고 가정하는가, 여러 개 가정하는가" 의 의도다. find 는 매칭이 없으면 None 을 리턴(자연스러운 fallback), find_all 은 매칭이 없으면 빈 리스트(루프가 즉시 끝남)를 리턴해야 한다. 이 두 가지 의도 차이가 None 체크와 for 루프의 두 가지 코드 형태를 강제한다.

테스트 — 비어있는 결과의 두 가지 모양

soup.find() vs find_all() — None vs [] 반환 확인
>>> soup.find('div', class_='no_such_class')
None
>>> soup.find_all('div', class_='no_such_class')
[]
>>> len(soup.find_all('div', class_='no_such_class'))
0
>>> for tag in soup.find_all('div', class_='no_such_class'):
...     print(tag)
... -- 출력 없음, 루프 즉시 종료

결론(중간)

KEY POINT find 는 "있다고 믿고 꺼내기" → if tag: 가드 필요. find_all 은 "있는 만큼 순회" → 가드 불필요, 빈 리스트가 자연 종료를 보장. 두 함수는 "0 개 가능성" 을 다른 방식으로 표현하기에 코드 패턴이 갈린다.
다음 챕터로 가는 다리 — 그러면 태그의 속성(href, src, class 등) 은 어떻게 꺼내는가. 그리고 네이버 같은 사이트가 "동적 해시 클래스명" 을 쓸 때 어떻게 대응하는가.

CH 03 DAY4속성 추출과 동적 클래스명 — class_ vs {'class': ...}#

학습

# a 태그의 href 속성 추출 — 3가지 방식
preview_tag = button_div.find('a', class_='btn_preview')
link1 = preview_tag.get('href')          # 권장: None 안전
link2 = preview_tag['href']              # 키 없으면 KeyError
link3 = preview_tag.attrs['href']        # 명시적

# class_ 는 파이썬 예약어 회피용 인자
soup.find('div', class_='data_box')      # ok
soup.find('div', {'class': 'data_box'})  # 동등

# CSS 부분 매칭 — 동적 해시 클래스명에 대응
soup.select('li[class*="TopPoiItem-"]')  # 'TopPoiItem-' 으로 시작하는 클래스
soup.select('li[data-otb_lsra-doc-id]')  # data 속성이 더 안정
CSS attribute selectors like [data-*] match elements by their HTML attributes rather than class names. Data attributes (data-*) are considered stable identifiers by convention because they carry semantic meaning and are less likely to be minified or auto-generated. — Beautiful Soup Documentation § CSS Selectors

의문 → 가설

가설 네이버는 빌드마다 CSS 클래스명에 해시(TopPoiItem-MgXeO) 를 붙인다. 따라서 클래스명 의존 선택자는 배포 1주일 후면 깨진다. 반면 data-otb_lsra-doc-id 같은 데이터 식별 속성은 의미 있는 이름이라 배포에 무관하게 유지될 가능성이 높다. "클래스보다 data 속성, data 속성보다 태그 구조" 순으로 안정성이 높아야 한다.

테스트 — 안정 선택자 비교

CSS 해시 클래스 vs data 속성 선택자 — 안정성 비교 확인
-- 1주일 전 작성한 selenium 선택자
>>> items = soup.find_all(class_='TopPoiItem-MgXeO')
>>> print(len(items))
0   -- 해시가 'TopPoiItem-NqWxB' 로 바뀜

-- 부분 매칭으로 일단 살림
>>> items = soup.select('[class*="TopPoiItem-"]')
>>> print(len(items))
0   -- 네이버가 클래스 체계 자체를 'pui__' 로 교체

-- 데이터 식별 속성으로 최종 안정화
>>> items = soup.select('li[data-otb_lsra-doc-id]')
>>> print(len(items))
10   -- 의미 있는 속성은 살아남음

결론(중간)

KEY POINT 크롤링 선택자의 안정성은 ① 해시 클래스 (최악) ② 의미 있는 클래스 ③ data 속성 ④ 태그 구조(div>span 순서) 순으로 높다. "이 선택자가 6 개월 후에도 작동할까" 의 질문이 모든 크롤링 코드의 첫 번째 리뷰 기준이다.
다음 챕터로 가는 다리 — 데이터를 꺼냈으니 이제 어제 만든 MySQL 테이블로 흘려보낼 차례다.

CH 04 DAY4크롤링 → DB — MySQLTemplate 으로 흘려보내기#

학습

from common.mysqlConnectTemplate import MySQLTemplate

insert_sql = '''
insert into movie 
(`rank`, title, star_point, release_date, genre, link)
values (%s, %s, %s, %s, %s, %s)
'''
# `rank` 는 MySQL 예약어 → 백틱 필요

with MySQLTemplate.get_connection() as conn:
    cursor = conn.cursor()
    for movie in movie_list:
        cursor.execute(insert_sql, (
            movie['rank'],
            movie['title'],
            movie['star_point'],
            movie['release_date'],
            movie['genre'],
            movie['link'],
        ))
    cursor.close()
# with end → 자동 commit/rollback + 자동 close
Parameterized queries (also known as prepared statements) prevent SQL injection by separating SQL code from data. The database driver handles quoting and escaping, making it impossible for malicious input to alter the query structure. — MySQL Connector/Python § MySQLCursor.execute()

의문 → 가설

가설 %s 플레이스홀더는 단순 문자열 포매팅이 아니다 — DB 드라이버가 자료형별로 자동 escape 해주는 매개변수 바인딩 이다. 그러므로 SQL Injection 을 방어하려면 f-string/format() 같은 문자열 포매팅을 절대 쓰지 말아야 한다. 이 가설은 — 영화 제목에 작은따옴표(') 가 들어 있어도 정상 INSERT 되는지로 검증한다.

테스트 — 작은따옴표 포함 제목 INSERT

cursor.execute(%s) — 따옴표 포함 INSERT · SQL 인젝션 방어 확인
>>> cursor.execute(insert_sql, (1, "Don't Look Up", 8.7, '2024-12-01', '드라마', 'http://...'))
<mysql.connector.cursor.MySQLCursor object>
>>> conn.commit()

>>> cursor.execute("select title from movie where `rank` = 1")
>>> cursor.fetchone()
("Don't Look Up",)   -- 작은따옴표가 안전하게 저장됨

-- 만약 f-string 으로 했다면:
>>> sql = f"insert into movie (title) values ('{title}')"
-- title 이 "Don't" 라면 SQL 자체가 깨짐 + 인젝션 위험

결론(중간)

KEY POINT DB 드라이버의 %s 바인딩은 ① 자료형 변환 ② 따옴표 escape ③ SQL Injection 차단의 3가지를 한꺼번에 한다. 크롤링 결과처럼 외부에서 온 데이터를 DB 에 넣을 때 유일하게 안전한 패턴 이다.
CAUTION rank, order, group, desc 같은 SQL 예약어를 컬럼명으로 쓰면 백틱(`) 으로 감싸야 한다. 이 백틱은 작은따옴표(문자열) 와 전혀 다른 토큰 — 식별자 인용 부호다.
다음 첵터로 가는 다리 — 크롤링 → 파싱 → DB 저장의 전 스택을 함께 쳐듳어보았다. 이제 이 다섯 단계가 하나의 흐름으로 랩쳐진 결론을 내릴 수 있다.

최종 결론

FINAL CONCLUSION 정적 크롤링은 본질적으로 "HTML 트리 탐색" 과 "스키마 매핑" 의 두 단계다. 트리 탐색은 BeautifulSoup 의 선택자가, 스키마 매핑은 0511~0513 에서 만든 MySQL 컬럼이 받는다. 그러나 한계가 명확하다 — 페이지가 JavaScript 로 만들어지는 경우, urllib 가 가져오는 HTML 에는 실제 데이터가 아직 없다. 그것이 0515 의 동적 크롤링(Selenium) 이 답하는 의문이다.

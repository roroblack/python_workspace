초록

이 프로젝트는 부트캠프에서 진행한 파이썬 학습 기록이다. 날짜 파일은 학습 흐름을, 프로젝트 폴더는 실행 가능한 코드 근거를, 노트 폴더는 사용자가 실제로 품었던 의문을 담고 있다. 본문은 이 세 자료를 연결해 “파이썬 리스트는 왜 C 배열과 다르게 느껴지는가”, “텍스트 파일 모드와 바이너리 모드는 실제로 무엇이 다른가”, “객체지향에서 추상화와 특수 메서드는 코드 실행에서 어떤 효과를 내는가”를 확인한다.

공식 문서 기준으로 보면 리스트와 딕셔너리는 가변 객체이고 튜플은 불변 시퀀스이며, 파일 객체는 텍스트 I/O와 바이너리 I/O로 분리된다. 또한 클래스 인스턴스는 이름 공간과 어트리뷰트 조회 규칙을 통해 상태와 행동을 결합하고, `__add__`, `__len__`, `__getitem__` 같은 특수 메서드로 연산자와 내장 함수의 동작을 흉내 낼 수 있다.

연구 질문

음수 인덱싱은 리스트 길이를 어떻게 아는가?

분석 결과

1. 리스트는 “동적이고 가변적인 시퀀스 객체”다

`list_sample.py`와 `list_sample2.py`는 리스트가 생성 후에도 요소를 교체하고, `append`, `insert`, `remove`, `pop`, `extend`, `reverse`, `sort`, `count` 같은 메서드로 내부 상태를 바꿀 수 있음을 보여준다. 공식 문서의 데이터 모델은 리스트를 가변 시퀀스로 설명하며, 시퀀스는 `len()`으로 길이를 갖고 인덱싱과 슬라이싱을 지원한다.

노트의 질문

“C 배열과 파이썬 리스트는 인덱싱, 슬라이싱, 동적 배열, 다차원 배열에서 정확히 뭐가 다른가?”

공식 문서 근거

파이썬 데이터 모델은 리스트를 “임의의 파이썬 객체”를 담는 가변 시퀀스로 설명한다. 또한 내장 시퀀스는 음수 첨자를 길이에 더해 해석할 수 있다고 설명한다.

lst = [1, 2, 3, "python", 3.45, [11, 22, 33], True, False]
print(lst[-2:])      # [True, False]
print(lst[1:7])      # [2, 3, 'python', 3.45, [11, 22, 33], True]
lst[0] = 77
lst[2] = [1.2, 2.3, 3.4]

따라서 C 배열과 달리 파이썬 리스트는 “메모리 크기와 타입을 개발자가 직접 관리하는 배열”이 아니라, 객체 참조들을 담고 길이를 관리하는 런타임 객체로 보는 편이 정확하다.

2. 튜플은 리스트와 닮았지만 불변성으로 목적이 달라진다

`tuple_sample.py`는 튜플도 인덱싱과 슬라이싱을 지원하지만 요소 대입은 불가능하다는 점을 확인한다. 특히 `(1)`은 괄호로 묶은 정수이고, `(1,)` 또는 `1,`이 1개짜리 튜플이라는 점은 노트로 남길 만한 함정이다.

print(type((1)).__name__)   # int
print(type((1,)).__name__)  # tuple
a, b = (4, 7)
print(a, b)                 # 4 7

공식 문서는 튜플을 불변 시퀀스로 분류한다. 다만 튜플 안에 리스트 같은 가변 객체를 담을 수는 있으므로 “튜플 안의 참조 묶음은 고정되지만, 참조된 객체 자체가 항상 불변인 것은 아니다”라고 이해하는 것이 안전하다.

3. 딕셔너리는 인덱스가 아니라 키로 접근하는 매핑이다

`dict_sample.py`와 직원 관리 실습은 딕셔너리를 사번과 직원 정보의 관계로 사용한다. 키는 조회와 삭제의 기준이 되고, 값은 리스트처럼 복합 자료형도 가능하다. 공식 문서는 딕셔너리를 가변 매핑으로 설명하며, 존재하지 않는 키를 `d[key]`로 조회하면 `KeyError`가 발생하고 `get()`은 기본값을 줄 수 있다고 설명한다.

4. `w`와 `wb`의 차이는 문자열과 바이트의 차이다

`fileio_module.py`는 텍스트 파일을 `open("testa.txt", "wt", encoding="utf-8")`로 열어 `str`을 기록한다. 반면 `fileio_module2.py`와 직원 저장 실습은 `pickle.dump()`를 `wb` 파일 객체에 사용한다. 공식 문서의 `open()` 설명은 텍스트 모드가 `str`을 반환하고 인코딩을 적용하는 반면, 바이너리 모드는 디코딩 없이 `bytes` 객체를 다룬다고 구분한다.

노트의 질문

“`wb`랑 `w` 차이. 정확히 `b`로 열어서 `w`로 연 것처럼 작업도 되지 않아?”

정리

`w`는 텍스트 쓰기다. 한글 같은 문자는 지정한 인코딩으로 바이트화된다. `wb`는 바이너리 쓰기다. 이미 바이트인 데이터나 피클 스트림을 직접 기록한다.

with open("sample.txt", "w", encoding="utf-8") as f:
    f.write("안녕\n")

with open("employees.dat", "wb") as f:
    pickle.dump({"200": ["홍길순", "개발부", 3800000]}, f)

즉 `wb`에서 텍스트를 저장하고 싶다면 `"안녕".encode("utf-8")`처럼 직접 바이트로 바꿔야 한다. 이 경우 개행 변환과 인코딩 처리를 개발자가 책임지는 셈이다.

5. `with`는 자원 정리를 “언젠가”가 아니라 “지금”으로 만든다

OOP 노트에는 “메모리 관리도 못하는데 소멸자 만들고 호출할 필요가 있나, 그냥 `with`를 쓰면 되는 것 아닌가”라는 질문이 있었다. 공식 문서의 데이터 모델과 예외 처리 튜토리얼은 열린 파일 같은 외부 자원은 명시적으로 닫는 것이 권장되며, `try/finally`와 `with`가 이를 편하게 만든다고 설명한다.

with open("sample.txt", "r", encoding="utf-8") as f:
    content = f.read()
# 이 블록을 벗어나면 예외 여부와 관계없이 파일이 닫힌다.

`__del__`은 객체가 수거될 때 호출될 수 있지만, 그 시점에 기대는 설계는 파일/네트워크 연결 같은 자원 관리에 불리하다. 실습 코드의 `f.close()`는 학습 단계에서는 명시적이지만, 실제 코드에서는 `with open(...) as f:`가 더 견고한 기본형이다.

6. 클래스는 이름 공간과 어트리뷰트 조회를 조직화한다

`class_sample.py`는 빈 클래스도 인스턴스를 만들 수 있고, 파이썬에서는 실행 중 인스턴스에 새 어트리뷰트를 붙일 수 있음을 보여준다. 공식 클래스 튜토리얼은 인스턴스 데이터 어트리뷰트가 선언 없이 처음 대입될 때 생긴다고 설명한다.

class SClass:
    pass

ref1 = SClass()
ref2 = SClass()
ref1.score = 100
# ref2.score는 아직 존재하지 않는다.

`class_oop.py`의 `__num`은 외부 접근을 강제적으로 완전히 막는 접근 제한자가 아니라 이름 맹글링을 통한 충돌 회피 장치에 가깝다. 파이썬의 비공개성은 언어 차원의 절대 격리보다 관례와 API 설계에 더 기대는 편이다.

7. 추상화와 다형성은 “같은 호출, 다른 구현”으로 관찰된다

OOP 노트의 추상화 질문은 `class_oop3.py`에서 바로 확인된다. `Dog`, `Cat`, `Animal`은 모두 `speak()`를 갖지만 객체 타입에 따라 다른 메시지를 낸다. 호출자는 `animal.speak()`만 알고, 내부 구현은 각 클래스가 책임진다.

for animal in [Dog(), Cat(), Animal()]:
    animal.speak()

이 구조가 “무엇을 하는지”와 “어떻게 하는지”를 분리한다. 사용자는 공통 인터페이스를 호출하고, 클래스는 자신에게 맞는 행동을 제공한다.

8. 특수 메서드는 파이썬 문법을 내 클래스에 연결한다

`class_oop2.py`는 `__add__`, `__sub__`, `__mul__`, `__truediv__`, `__len__`, `__contains__`, `__getitem__`, `__str__`를 통해 객체가 연산자와 내장 함수에 반응하도록 만든다. 공식 데이터 모델은 이런 이름들을 “특수 메서드”라고 부르고, 클래스가 산술 연산, 서브스크립팅, 컨테이너 동작을 흉내 낼 수 있다고 설명한다.

class OOP:
    def __init__(self, num):
        self.__num = num
    def __add__(self, value):
        return self.__num + value

ref = OOP(100)
print(ref + 30)  # 130

따라서 연산자 오버로딩은 “문법 장난”이 아니라, 사용자 정의 객체를 파이썬의 기존 프로토콜에 참여시키는 방식이다.

9. 예외 처리는 입력 오류를 프로그램 흐름으로 복구한다

`except_sample.py`는 `ValueError`, `IndexError`, `KeyError` 같은 런타임 예외를 다룬다. 공식 예외 튜토리얼은 문법 오류와 실행 중 예외를 구분하고, `try`, `except`, `else`, `finally`, 사용자 정의 예외를 통해 예측 가능한 실패를 처리한다고 설명한다.

try:
    num = int(input("정수를 입력하세요 : "))
except ValueError:
    print("정수 숫자만 입력해야 합니다.")
finally:
    print("예외 처리 구문 테스트 종료")

부트캠프 과제 맥락에서는 점수 범위 오류, 삭제 인덱스 오류, 직원 파일 미존재 같은 문제를 예외로 모델링할 수 있다. 이렇게 하면 입력 검증, 에러 메시지, 정상 루프 복귀를 한 구조 안에서 다룰 수 있다.

컬렉션: 리스트와 튜플

## collection: list and tuple

주의할 점
